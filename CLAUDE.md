## 0. What we are building

A **self-hosted web service** that builds and maintains a self-organizing wiki from raw
documents, following the **Karpathy LLM Wiki pattern** (original: nashsu/llm_wiki
https://github.com/nashsu/llm_wiki). Key re-imagination goals:
1. Fix the 4 performance bottlenecks of llm_wiki (re-scan, graph freeze, per-token parse, WYSIWYG).
2. Make the **analysis/ingest AI fully user-selectable at runtime** (F17 — see §4).
3. Add a knowledge graph with weighted relevance, server-side layout, and a fluid web UI.
4. Documentation-as-code: every diagram lives in `docs/` as Mermaid; never goes stale.

> ⚠️ The multi-agent team (this file + `.claude/agents/`) is the **build-time** orchestration
> mechanism, used during development via Claude Code / claude-agent-sdk. It is NOT part of the
> Synapse runtime architecture. At runtime, Synapse is a FastAPI service; the AI provider is
> chosen by the end-user per vault/operation (Local/API/CLI).

## 3. Non-negotiable invariants

> Any change that would violate an invariant MUST be escalated to `solution-architect` before
> proceeding. Never work around these silently.

| # | Invariant | Rationale |
|---|-----------|-----------|
| **I1** | **INCREMENTAL INDEX ONLY.** Never full-rescan. A file change updates only affected records in Postgres (metadata/links/sources[]/coords) and Qdrant (bge-m3 embeddings). | Fix bottleneck #1 of llm_wiki |
| **I2** | **GRAPH LAYOUT IS SERVER-SIDE AND CACHED.** FA2 (igraph/graph-tool) runs offline; coordinates stored in Postgres; recompute debounced on `dataVersion` bump. Client renders precomputed coords in WebGL (sigma.js). NEVER run a force layout on the UI main thread. | Fix bottleneck #2 |
| **I3** | **NO PER-TOKEN HEAVY WORK IN CHAT.** Markdown and LaTeX parsed at stream END. Zustand uses selectors + shallow equality. No heavy re-renders on each token. | Fix bottleneck #3 |
| **I4** | **EDITOR IS CODEMIRROR 6.** No WYSIWYG/ProseMirror. All long lists (tree, message history) are virtualized (TanStack Virtual). | Fix bottleneck #4 |
| **I5** | **OBSIDIAN COMPATIBILITY.** `wiki/` must remain a valid Obsidian vault: YAML frontmatter, `[[wikilinks]]`, auto-generated `.obsidian/`. Verified each sprint. | Core K7 — non-negotiable |
| **I6** | **PLUGGABLE INFERENCE (F17).** The analysis/classification/ingest AI MUST sit behind `InferenceProvider` with 3 backends: Local (Ollama), API (Anthropic / OpenAI-compatible), CLI (claude-agent-sdk). NEVER hardcode a provider anywhere. Routing is capability-aware: `supports_agentic_loop=True` (CLI) → delegate the whole ingest; otherwise → run the ORCHESTRATED INGEST LOOP (`analyze → generate → validate → if invalid augment & retry, max N`). | Core F17 — the defining feature |
| **I7** | **LOOPS ARE BOUNDED.** Every loop (ingest refinement, deep-research, lint-fix, consistency, fallback) has a `max_iter` cap AND a `token_budget`. Log `total_cost_usd` per run. No runaway loops. | Cost control + safety |
| **I8** | **DOCS-AS-DoD.** Diagram-as-code (Mermaid), screenshots auto-captured by Playwright into `docs/screens/`, API ref auto from FastAPI OpenAPI. The ER diagram is generated from the SQLAlchemy models (`make er`) and MUST match the live schema. No sprint is done without its D-artifacts updated and consistent. | Doc quality gate |
| **I9** | **DO NOT REINVENT.** Reuse Ollama, SearXNG, Qdrant, bge-m3 (all already running). Use native Claude Code features (subagents, MCP, hooks). SearXNG IS the web-search backend (not Tavily). Mermaid/Playwright/OpenAPI for docs. | Respect existing infra |

## 5. F17 — Inference Provider detail

```
InferenceProvider (ABC)
 ├─ analyze(source_text, vault_context)   → Analysis
 ├─ generate(analysis, retrieval_context) → list[WikiPage]
 ├─ chat(messages, retrieval_context)     → stream
 └─ capabilities()                        → {mode, supports_tools, supports_agentic_loop, max_context, name}
```

| Backend | Implementation | Ingest strategy | When to use |
|---------|---------------|-----------------|-------------|
| **Local — `OllamaProvider`** | Ollama `/api/chat` with `format=json` / grammar for structured output; tool-calling only if model supports it | **Orchestrated loop** (analyze→generate→validate→retry, max N) | Offline, privacy-sensitive vaults, zero cost |
| **API — `ApiProvider`** | Anthropic Messages API (tool-use + JSON Schema) **or** OpenAI-compatible endpoint (configurable base_url) | **Orchestrated loop** with native tool-calling when available | Quality + cost control; default recommendation |
| **CLI — `CliAgentProvider`** | `claude-agent-sdk` (bundled CLI). Filesystem tools scoped to vault; in-process MCP tools (search, write_page). `permission_mode='acceptEdits'` | **Delegated** — provider runs its own agent loop | Maximum quality; Karpathy lineage |

**Capability-aware routing** (orchestrator decides, not user):
- `supports_agentic_loop == True` (CLI) → delegate full ingest; provider links pages in its own loop
- otherwise → run orchestrated ingest loop

**Config model:** `provider_config` table (scope: global | vault; override per-operation) + dropdown UI. Same abstraction for ingest AND chat (user can ingest via Local, chat via API).

## 7. Canonical loops (all bounded)

| Loop | Where | Bound |
|------|-------|-------|
| **Ingest refinement** (API/Local) | Ingest orchestrator | `max_iter` + `token_budget`; log cost |
| **Deep Research** | ops/deep_research.py | max iterations; assess sufficiency before each query |
| **Lint-fix** | ops/lint.py | max rounds; human gate before applying fixes |
| **Consistency/contradiction** | post-ingest | scan related pages via graph; flag to review (F9), bounded |
| **Provider fallback** | Provider layer | primary fail/timeout → fallback once |
| **Dev self-correction** (CI only) | QA agent | test → fail → fix → retest until green |

## 11. Workflow & commit conventions
