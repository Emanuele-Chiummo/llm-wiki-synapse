## 1. Per-workstream verdicts

### WS-A — Real-time freshness [F16/F4/F18] — **PASS**

Files: `store/statusStore.ts`, `components/activity/ActivityBar.tsx`,
`components/home/HomeDashboard.tsx`, `components/GraphViewer.tsx`.

- **Mechanism is correct and minimal.** `dataVersion` is added as a scalar to `statusStore`
  (`statusStore.ts:55`, selector `selectStatusDataVersion` at `:112`). It is populated by the
  **existing** ActivityBar `/status` poll (`ActivityBar.tsx:416` → `setDataVersion(res.data_version ?? null)`),
  which runs on the pre-existing `STATUS_POLL_MS = 30_000` timer (`ActivityBar.tsx:64`). No new
  poller, no WebSocket, no new endpoint. Confirmed by diff grep: zero new `setInterval`/`setTimeout`
  in HomeDashboard/GraphViewer. **AC-WS-A-4 satisfied.**
- **I2 (server-side layout) HELD.** GraphViewer's freshness effect (`GraphViewer.tsx:1791`) calls
  ONLY `fetchGraph(vaultId, …)` and stores the returned precomputed coords via `setGraph(...)`.
  No FA2, no d3-force, no force/circlepack/noverlap layout added — confirmed by diff grep against
  the GraphViewer patch (NONE). The frontend detects the version bump and re-pulls
  server-computed coordinates; it never computes a layout.
- **I3 (no per-tick heavy work) HELD.** Both subscribers gate on a scalar version compare before
  doing anything:
  - GraphViewer: `if (statusDataVersion === lastFetchedGraphVersionRef.current) return;`
    (`GraphViewer.tsx:1794`); effect dep is the scalar `[statusDataVersion]`.
  - HomeDashboard: `if (statusDataVersion === lastFetchedVersionRef.current) return;`
    (`HomeDashboard.tsx:1155`); `loadStats` is a stable `useCallback([])` reading the live version
    via `statusDataVersionRef` (`HomeDashboard.tsx:1128`), so a same-version tick produces zero
    fetches and zero re-render. This is the correct Zustand-selector + ref pattern.
- **Null/sentinel handling is sound.** Pre-WS-A backends (absent `data_version`) → `null`, and the
  guard `if (statusDataVersion === null) return;` prevents spurious fetches. HomeDashboard uses a
  `-1` sentinel when recording a null-version fetch so it never loops.

Verdict: **PASS.** Textbook implementation of the version-gated refresh pattern. I2 and I3 upheld.

### WS-B — Review queue status-filter [F9] — **PASS**

Files: `components/review/ReviewQueueView.tsx` (only — `store/reviewStore.ts` correctly UNCHANGED).

- Phase 0 (P0-1) reframed this from a "filter" bug to a **presentation/logic** bug: resolved/dismissed
  items rendered with the pending card UI (Crea / Salta / Ignora / Ricerca Profonda), which is a
  genuine logic hazard (re-creating an already-created page). The fix is correctly keyed on
  `item.status`.

  deep_researched | auto_resolved (`api/types.ts:333`). `STATUS_BADGE_STYLE` covers all five terminal
  values (`ReviewQueueView.tsx:173`) and falls back gracefully for any unmapped status. Both EN and IT
  carry the `review.statusBadge.*`, `resolvedAt`, `viewCreatedPage` keys.
- `data-status={item.status}` is now emitted on both card variants — good for the Playwright/vitest
  disjoint-set assertion (AC-WS-B-3).

(`ReviewQueueView.tsx:1078`, `void pageId`) and does not yet call `graphStore.selectPage(pageId)`; the

## 3. ADR decision (AQ-M136-3)

**No new ADR is required. A code comment suffices — and already exists.**

Rationale:
1. The dataVersion-polling refresh strategy introduces **no new architectural seam**: no new table,
   no new column, no new endpoint, no new contract, no new dependency. It reuses (a) the existing
   `/status` response field `data_version` (`status.py:41`, already covered by the F16dv work and
   AC-F16dv-*), (b) the existing ActivityBar `STATUS_POLL_MS` poll, and (c) the existing server-side
   graph cache decision already recorded in **ADR-0045 (ForceAtlas2 server-side layout)**.
2. The decision it embodies — "detect a version bump and re-pull, never push, never compute layout
   client-side" — is a direct application of invariants I2/I3 that are already ADR-backed; writing a
   new ADR would restate settled decisions.
3. The sprint scope (§8 anti-scope-creep) explicitly lists "Any new ADR" as out-of-scope unless
   architect-initiated as a WS-F pre-study. This strategy is not that.
4. The pattern is already self-documenting in-code with invariant-cited comments at the three touch
   points: `statusStore.ts:13`/`:46`, `GraphViewer.tsx:1664`, `HomeDashboard.tsx:1082`. That is the
   appropriate weight for a bounded, invariant-derived implementation detail.

If a future sprint introduces true push (WebSocket/SSE) or a dedicated freshness endpoint, THAT change
would warrant an ADR. The current polling approach does not.

## 4. Nits (non-blocking — do not gate merge)

1. **WS-B deep-link incomplete.** `handleOpenCreatedPage` (`ReviewQueueView.tsx:1078`) switches to the
   pages section but does not yet `selectPage(pageId)`; the created-page link lands on the section, not
   the page. Acknowledged in-code as future wiring. Recommend a follow-up ticket. No invariant impact.
2. **Frontend `act(...)` warnings** in NoteView specs are pre-existing harness noise (tests pass);
   worth a cleanup pass but out of scope here.
3. **WS-C single-file "progress" averaging** assumes per-task `progress` is a 0..1 fraction
   (`Math.round(avg*100)`). If any producer emits 0..100, the bar would read 100% early. Confirm the
   `activityStore` task `progress` unit is a fraction (ADR-0046) — the batch path is unaffected. Low
   risk; recommend a one-line assertion or clamp `Math.min(100, …)` in a follow-up.
