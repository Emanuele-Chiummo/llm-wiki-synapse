## 4. F9 Fire-and-Forget Hook Safety (AC-F9-2)

Test: `TestFireAndForgetHook::test_hook_exception_does_not_raise` (T-RV-006)

The test patches `enqueue_review` to raise `RuntimeError("Simulated DB error in F9 hook")`, then calls the orchestrator's hook pattern directly:

```python
try:
    await review_mod.enqueue_review(...)
except Exception:
    pass  # AC-F9-2: never propagate
```

Asserts: no exception propagates out of the hook invocation.
Result: PASS

Additionally, `run_ingest_pipeline` in orchestrator.py lines 377-387 wraps the entire `_enqueue_review_items()` call in `try/except Exception` with a `logger.warning` — the page write has already completed before the hook runs. The hook catching is at both the generator level (review.py) and the orchestrator level (orchestrator.py), providing defense-in-depth.

Lines: 452, 470, 504, 537, 584, 612 in test_review.py.

## 7. F9 Deep-Research Action — No Orphan Row (C1 Pattern)

Test: `test_deep_research_returns_202_with_run_id` + `test_deep_research_stores_run_id_on_item`

The implementation in `ops/review.py::deep_research()`:
1. Pre-INSERTs the `deep_research_runs` row with `status="running"` before `asyncio.create_task` fires
2. Updates `review_items.deep_research_run_id` to the pre-inserted `run_id`
3. Returns `DeepResearchResult(review_item_id, run_id)`

This is the same fire-and-poll pattern used by the research_start endpoint (ADR-0024). The C1 orphan-row risk is avoided: the run row exists in the DB before the background task starts.

Tests verify:
- 202 response contains both `run_id` and `review_item_id` as valid UUIDs
- DB row `deep_research_run_id` matches the returned `run_id`
- `status` on the review item is `deep_researched`

503 when `SEARXNG_URL` is unset: verified in T-RV-012 (the 503 guard fires before the run is pre-inserted, ensuring no orphan run row on the 503 path).

Result: PASS
