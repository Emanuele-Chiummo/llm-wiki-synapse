## 2. Acceptance Criteria Per Feature

The following are crisp, testable acceptance criteria handed to functional-analyst
for elaboration into user stories. Each AC maps 1-to-1 to its feature ID.

### F1 — 3-panel shell

- AC-F1-1: The frontend renders exactly 3 resizable panels: left (file tree), center (chat/main), right (preview/graph). Panel resize uses CSS or a drag handle; no layout calculated in JS on every frame.
- AC-F1-2: All long lists in the tree panel (more than 50 items) are virtualized using TanStack Virtual. No non-virtualized list rendering more than 50 DOM nodes at once (I4).
- AC-F1-3: The editor/preview panel uses CodeMirror 6 exclusively. No ProseMirror, no contentEditable WYSIWYG element in the DOM (I4).
- AC-F1-4: The graph sigma viewer from v0.3 is embedded in the 3-panel shell (right panel or as a tab), not removed. Its no-client-layout bundle assertion (T-NCL-001..022) must still pass.
- AC-F1-5: An activity panel (status bar or overlay) shows: current vault, active provider name, last ingest timestamp, data_version.
- AC-F1-6: At least 2 scenario templates are selectable from the UI (e.g. "Summarize document", "Find related pages"); each pre-fills the chat input with a template prompt.
- AC-F1-7: Playwright E2E test: 3 panels are visible in the DOM; resizing left panel to 200px does not cause a layout reflow longer than 16ms (captured via CDP PerformanceMetrics).

### F1-NAV — Left Navigation Rail

Visual direction: nashsu/llm_wiki-inspired persistent side navigation. The left region of
the shell becomes a navigator bar with labeled sections rather than a plain file tree.

- AC-F1-NAV-1: The shell's left region contains a persistent vertical navigation rail with at minimum 4 section entries: Pages (file tree), Graph (sigma viewer), Ingest (ingest activity view), Settings (settings panel). A Chat section item is present in the rail but renders as "coming in Phase 3" or is disabled, clearly signalling its reserved status — it must NOT open a chat panel in Phase 2.
- AC-F1-NAV-2: Clicking a rail section switches the main content area to that section's view without a full page reload. The active section item is visually indicated (highlighted/selected state).
- AC-F1-NAV-3: The rail is persistent across view switches — it does not collapse or disappear when the active section changes.
- AC-F1-NAV-4: The Pages section renders the virtualized file tree from F1 (AC-F1-2 still applies: TanStack Virtual for lists >50 items). The Graph section renders the sigma viewer (AC-F1-4 still applies). The Settings section renders the settings panel (F17-UI, F14, F16-rest).
- AC-F1-NAV-5: Rail section labels are translation keys (appear in locales/en.json and locales/it.json per F16-i18n) — no hardcoded display strings in the component.
- AC-F1-NAV-6: The activity panel (AC-F1-5: vault name, active provider, last ingest timestamp, data_version) is visible regardless of which rail section is active — it must not be hidden when switching views.
- AC-F1-NAV-7: Playwright E2E test: assert all 4 section nav items are in the DOM; click "Graph" → sigma canvas visible; click "Pages" → file tree visible; click "Settings" → settings panel visible; click "Ingest" → ingest list visible. No full page reload between switches (no navigation event fired).
- AC-F1-NAV-8: vitest test: active section state in Zustand store updates correctly on nav item click; no selector re-computation fires for unrelated store slices when nav state changes (I3 Zustand selector discipline).

### F1-INGEST-VIEW — Ingest Activity View

Boundary note: This view is READ-ONLY history + a trigger button. It is EXPLICITLY NOT the
F9 async HITL review queue. F9 (Create / Deep-Research / Skip actions, pre-generated queries,
async queue management) is M5 and must not be started here. Any engineer proposing F9 features
inside this view during M4 will be blocked.

- AC-F1-IV-1: The Ingest section (accessible via the nav rail) renders a list of recent ingest runs fetched from GET /ingest/runs. Each list row displays: status (color-coded badge: running / completed / failed / converged_false), provider type, pages created count, total cost in USD (formatted to 4 decimal places to honour I7 cost visibility), started_at timestamp (human-readable relative format, e.g. "2 minutes ago"), and error_message (truncated to 80 chars with an expand control if present).
- AC-F1-IV-2: The list is paginated or virtualized: at most 20 rows rendered per page; a "Load more" control or scroll-triggered pagination fetches the next batch. If TanStack Virtual is used, no more than 40 DOM rows are mounted at once.
- AC-F1-IV-3: A "Run Ingest" button triggers POST /ingest/trigger with the current vault_id. On success (HTTP 202), a transient success toast is shown and the list auto-refreshes (re-fetches GET /ingest/runs). On failure, an error toast shows the error message from the response body.
- AC-F1-IV-4: While an ingest run has status "running", its row displays a loading indicator (spinner or pulsing badge). The list polls or uses a short refresh interval (configurable, default 5s) while any run is in "running" status; polling stops when no "running" rows are visible.
- AC-F1-IV-5: The view is read-only. There are no "approve", "reject", "skip", or "deep-research" actions on any row. These are F9 features and must not appear here.
- AC-F1-IV-6: The backend endpoint GET /ingest/runs exists and returns the fields specified in BE-INGEST-RUNS (§1b). It is documented in docs/api/openapi.json (D4 updated via `make openapi`). Pydantic response model validated by pytest schema test.
- AC-F1-IV-7: vitest test: mock GET /ingest/runs response with 3 runs (completed, failed, running) → assert all 3 rows render with correct status badge and cost display.
- AC-F1-IV-8: Playwright E2E test: navigate to Ingest section; assert list renders; assert "Run Ingest" button is present; assert clicking it fires a POST /ingest/trigger request (network assertion via CDP or Playwright route intercept).

### BE-INGEST-RUNS — GET /ingest/runs backend endpoint

- AC-BE-IR-1: GET /ingest/runs returns HTTP 200 with a JSON body: `{items: [...], total: int, limit: int, offset: int}`. Each item contains all fields from the `ingest_runs` table as specified in §1b: id, vault_id, status, provider_type, pages_created, iterations_used, total_cost_usd, started_at, completed_at (nullable), error_message (nullable).
- AC-BE-IR-2: Query parameters: `limit` (int, default 20, max 100), `offset` (int, default 0), `vault_id` (uuid, optional filter). Invalid limit/offset values return HTTP 422 with a validation error body.
- AC-BE-IR-3: Results are ordered by started_at DESC (most recent first).
- AC-BE-IR-4: The endpoint is added to the FastAPI router and appears in the regenerated openapi.json with a correct Pydantic response schema (D4 zero-drift gate applies on next `make openapi` run).
- AC-BE-IR-5: pytest integration test: seed 3 ingest_runs rows with different statuses; call GET /ingest/runs; assert all 3 returned in started_at DESC order; assert total = 3.

### F6 — Multi-conversation persistent chat

- AC-F6-1: Conversations are persisted in Postgres (new `conversations` and `messages` tables). A page refresh restores the last active conversation.
- AC-F6-2: Multiple named conversations can be created, switched between, and deleted. Switching is <100ms (measured by vitest timer mock).
- AC-F6-3: Each assistant message displays `[n]` citation markers that link to the source page (page title shown on hover). Citation data is stored with the message.
- AC-F6-4: A "regenerate" button re-issues the last user prompt to the provider and replaces the last assistant message. The prior message is removed from the UI (not kept in history).
- AC-F6-5: A "save to wiki" button on an assistant message triggers POST /ingest with the message content as source text; the ingest result (page title + link) is shown inline.
- AC-F6-6: Message history list is virtualized (TanStack Virtual) — no more than 30 DOM nodes rendered at once regardless of conversation length (I3, I4).

### F7 — Reasoning `<think>` display

- AC-F7-1: When an assistant message contains `<think>…</think>` tags, the reasoning block is extracted and rendered in a separate collapsible section ("Reasoning"), collapsed by default.
- AC-F7-2: The `<think>` block is NOT stripped from the stored message content; it is stored in full and re-parsed on render (no irreversible mutation of stored data).
- AC-F7-3: Streaming: the `<think>` block accumulates during stream and collapses when `</think>` is received. The main content begins rendering after `</think>`. No heavy parse during stream (I3).
- AC-F7-4: vitest test: a fixture message string with `<think>…</think>` renders the reasoning section collapsed and the main content visible.

### F8 — LaTeX to Unicode

- AC-F8-1: LaTeX expressions in assistant messages are converted to Unicode at stream END (on the `done` or final chunk event). No conversion is triggered per-token (I3).
- AC-F8-2: The conversion covers at minimum: Greek letters (α β γ … Ω), common operators (∑ ∏ ∫ ∂ ∇), arrows (→ ← ↔ ⇒ ⇔), and fractions/superscripts/subscripts for common cases.
- AC-F8-3: Unconvertible LaTeX (complex display math) is left as a fenced code block, not silently dropped.
- AC-F8-4: vitest test: fixture string with 5 LaTeX expressions converts correctly after stream END; no conversion fires during incremental stream appends.

### F14 — Configurable context window

- AC-F14-1: A settings UI (settings panel or modal) exposes a context window size selector: 4K, 8K, 16K, 32K, 64K, 128K, 256K, 512K, 1M (or numeric input validated against these bounds).
- AC-F14-2: The selected size is persisted in the `provider_config` table (new column `context_window_tokens`). Default is 32K if not set.
- AC-F14-3: The retrieval assembler (F5 stub or the chat message builder) respects the budget split: 60% conversation history, 20% retrieved context, 5% system prompt, 15% generation headroom. The split is computed from the configured window, not hardcoded to absolute token counts.
- AC-F14-4: If the conversation history exceeds its 60% allocation, the oldest messages are truncated (not the most recent). The truncation boundary is computed server-side.
- AC-F14-5: Unit test: mock a 10-message conversation against a 4K window; assert messages[0..N] are truncated to keep total ≤ 60% × 4096, and the split percentages are correct.

### F17-UI — Provider Selector

- AC-F17-UI-1: A Provider Selector dropdown is visible in the settings panel or activity panel. It lists all configured providers from GET /provider/config.
- AC-F17-UI-2: Selecting a provider posts to POST /provider/config with the selected provider_type and scope=vault (current vault) or scope=global per a sub-toggle. The selection persists across page refresh.
- AC-F17-UI-3: The active provider name is shown in the activity panel (feeds AC-F1-5).
- AC-F17-UI-4: The dropdown correctly shows 3 provider types: Local (Ollama), API (Anthropic / OpenAI-compatible), CLI (claude-agent-sdk). Each shows its model_id from provider_config.
- AC-F17-UI-5: Changing the provider in the UI does not require a page reload; the next ingest or chat uses the newly selected provider. vitest test: simulate provider change → assert store updates + next mock request uses new provider_type.
- AC-F17-UI-6: No model ID or provider type is hardcoded in any frontend component; all values come from the GET /provider/config response (mirrors backend AC-F17-4 / I6).

### F16-rest — Remaining F16 sub-features

- AC-F16-i18n-1: The UI renders in Italian when the browser locale is `it` or `it-IT` and in English otherwise. Language detection uses `navigator.language`. At minimum the following strings are translated: panel titles, button labels (Ingest, Chat, Save, Settings), error messages.
- AC-F16-i18n-2: Language strings are NOT hardcoded inline; they live in a `locales/en.json` and `locales/it.json` file. vitest test: every key in `en.json` has a corresponding key in `it.json`.
- AC-F16-settings-1: User settings (active provider, preferred language, panel widths, context window size) are persisted in localStorage and restored on page load.
- AC-F16-settings-2: A "Reset settings" action clears localStorage and resets UI to defaults without requiring a hard reload.
- AC-F16-obsidian-1: The backend generates vault/wiki/.obsidian/app.json on startup if absent, and regenerates it if the vault_id changes. Generated config remains a valid Obsidian app.json (JSON-parseable, contains "legacyEditor": false). Extends AC-K7-1 (v0.1).

- AC-F16-timeout-1: Each provider call (ingest and chat) respects a configurable timeout (default 60s for chat, 120s for ingest). Timeout is stored in provider_config as `timeout_seconds`. On timeout, the UI shows a user-visible error ("Provider did not respond in Xs") and the backend logs the timeout as a provider failure (feeds the I7 bounded-fallback mechanism).

## 6. Open Handoff: TRACEABILITY Update Required (ADR-0016 §6)

by the functional-analyst before Phase 0 can be signed off.

The correction: ADR-0012 §5 documents a worked fixture with a type-only pair P3–P5
(entity pages sharing no links and no shared source). Under the old inclusion rule this
pair produced weight == 1.0 and was stored as an edge. Under ADR-0016 §1's structural gate

Required TRACEABILITY change: in the Sprint 3 coverage section for AC-F4-1 (T-GENG-001..012),
the expected value for the P3–P5 pair must change from "weight == 1.0, present" to "absent
(no structural tie)". The lower-bound assertions for P1–P2 (≥11), P1–P4 (≥8), and P2–P4 (≥5)

weights are identical).

the test fixture to match ADR-0016 before Phase 0 sign-off. This is a one-line correction,
not a re-design. Failing to fix it would leave a GREEN test that asserts the wrong behavior
(an invariant violation: the type-clique hairball would be silently re-introduced on the

### M4-EXT: Feature U — F1-UPLOAD — Document upload from the UI

**Scope summary:** A drag-and-drop / file-picker in the Ingest section that uploads a
document from the user's browser into vault/raw/sources/ and triggers ingest. The upload
path reuses the existing ingest pipeline (watcher or POST /ingest/trigger). This sprint
accepts TEXT (.txt) and MARKDOWN (.md) files only.

**F12 boundary — explicit and non-negotiable:**
Feature U accepts `.txt` and `.md` files ONLY in v0.4. Multi-format parsing (PDF, DOCX,
PPTX, XLSX, images, AV) is F12 and remains M5. The file-type gate must be enforced both
client-side (accept attribute) AND server-side (Content-Type + extension validation); any
file that is not text/plain or text/markdown is rejected with HTTP 415 and a clear error
message. This boundary is stated here so engineers do not conflate F12 with Feature U.
The upload implementation MUST NOT add any dependency on pypdf, unstructured, python-docx,
python-pptx, openpyxl, or any AV library — those belong to F12/M5.

**Backend endpoint:**

`POST /ingest/upload` — multipart/form-data
- Request fields: `file` (binary, required), `vault_id` (uuid string, optional — defaults
  to current active vault).
- The backend sanitizes the filename: strips path components, normalizes unicode, replaces
  characters outside `[a-zA-Z0-9._-]` with `_`, truncates to 200 chars. Path traversal is
  explicitly blocked (reject any filename containing `/`, `\`, `..`). The sanitized file is
  written to `vault/raw/sources/<sanitized-name>`.
- After writing the file, the backend triggers ingest via the existing ingest path
  (POST /ingest/trigger with the new file path, or by directly invoking the orchestrator).
- Response on success: HTTP 202 with `{file_path: "vault/raw/sources/<sanitized-name>",
  ingest_run_id: <uuid>}`.
- On unsupported file type: HTTP 415 with `{detail: "Unsupported file type. Only .txt and
  .md files are accepted in this version. PDF/DOCX/PPTX support is F12 (M5)."}`.
- On path-traversal attempt: HTTP 400 with `{detail: "Invalid filename."}`.
- On file too large (configurable limit, default 10 MB): HTTP 413.
- The endpoint appears in openapi.json (D4 zero-drift gate, same `make openapi` run).

**Acceptance criteria — F1-UPLOAD:**

- AC-U-1: The Ingest section contains a drag-and-drop zone and/or a file-picker button
  labeled (i18n key) "Upload document". Only `.txt` and `.md` files are selectable/droppable;
  other file types show a client-side validation error before any network request is made.
- AC-U-2: On valid file selection the frontend calls POST /ingest/upload (multipart). On
  HTTP 202 response: a success toast shows the sanitized filename and ingest_run_id; the
  ingest runs list (AC-F1-IV-1) refreshes to show the new run.
- AC-U-3: POST /ingest/upload writes the file to vault/raw/sources/ with the sanitized
  filename and triggers ingest. The ingest run appears in GET /ingest/runs within 10 seconds.
- AC-U-4 (path-traversal safety): POST /ingest/upload with a filename containing `/`, `\`,
  or `..` returns HTTP 400 "Invalid filename". No file is written. Unit test covers: `../../etc/passwd`,
  `../secrets`, `foo/bar.md`, `foo\bar.md`.
- AC-U-5 (file type gate): POST /ingest/upload with a `.pdf`, `.docx`, `.xlsx`, `.png`,
  or any non-text/non-markdown MIME type returns HTTP 415 with the exact error message
  referencing F12/M5. The client-side check (accept attribute) is a UX guard only; the
  server-side check is authoritative. Unit test: submit `.pdf` → assert HTTP 415 body.
- AC-U-6 (size limit): POST /ingest/upload with a file larger than the configured limit
  (default 10 MB, env var `UPLOAD_MAX_BYTES`) returns HTTP 413. Unit test covers this.
- AC-U-7 (I1 incremental): uploading the same file twice (identical content + name) does

- AC-U-9: POST /ingest/upload is documented in openapi.json (D4 updated via `make openapi`).

- AC-U-10: vitest test: mock POST /ingest/upload → HTTP 202 → assert success toast rendered

### M4-EXT: Feature S — F1-SCHED — Scheduled folder import (in Settings)

**Scope summary:** A "Automatic import" sub-section in the Settings panel where the user
configures a source folder and a frequency. The backend periodically scans that folder and
imports new documents into vault/raw/sources/ (then the watcher indexes them, satisfying I1).
Configuration is persisted in Postgres. A "Run now" action triggers an immediate scan.

**Critical constraint — container-mounted paths:**
The backend runs inside a Docker container. It cannot see arbitrary host filesystem paths.
The source folder configured by the user MUST be a path that is mounted as a Docker volume.
The acceptance criteria, UI hint text, and documentation must state this explicitly. The
Settings UI must show a helper note: "This folder must be a path mounted into the Synapse
container (e.g. /data/import). Arbitrary host paths are not accessible. See DEPLOY.md for
volume configuration." The backend must validate that source_dir is an absolute path and that
the directory exists and is readable from inside the container; if not, the save is rejected
with HTTP 422 and a message explaining the container constraint.

**Bounded scheduler — I7 compliance:**
The scheduler is subject to I7 (bounded loops). Each scan run has a `max_files_per_run` cap
(configurable, default 50) and a `scan_timeout_seconds` cap (configurable, default 300 s).
`total_cost_usd` is NOT applicable to the scan itself (no inference in folder scanning);
however, each ingest triggered by the scan DOES log cost through the existing ingest pipeline.
The scheduler MUST NOT trigger overlapping scan runs (if a scan is already running, skip and
log a warning).

**Backend endpoints:**

`GET /import-schedule` — returns the current scheduled import configuration for the active
vault. Response fields: `enabled` (bool), `source_dir` (string or null), `frequency`
(enum: hourly / daily / weekly / manual), `interval_minutes` (int — derived from frequency
or custom), `last_run_at` (timestamptz or null), `last_status` (enum: ok / error / never_run
or null), `last_error_message` (text or null), `max_files_per_run` (int), `scan_timeout_seconds`
(int). HTTP 200 or HTTP 404 if no schedule configured for the vault (returns a default-empty
response object).

`PUT /import-schedule` — creates or replaces the import schedule for the active vault. Request
body: `{enabled: bool, source_dir: string, frequency: "hourly"|"daily"|"weekly"|"manual",
interval_minutes: int|null, max_files_per_run: int|null, scan_timeout_seconds: int|null}`.
Validation: `source_dir` must be absolute, must exist and be readable inside the container
(checked at save time); if not, HTTP 422 with `{detail: "source_dir is not accessible from
the container. Mount the folder as a Docker volume and use its container-side path."}`.
Returns HTTP 200 with the persisted schedule object.

`POST /import-schedule/run-now` — triggers an immediate scan run (outside the scheduler
cadence). Returns HTTP 202 with `{scan_run_id: <uuid>}`. If a scan is already in progress,
returns HTTP 409 with `{detail: "A scan is already running."}`.

All 3 endpoints appear in openapi.json (D4 zero-drift gate applies on next `make openapi`).

**Persistence — new `import_schedules` table (Alembic migration):**
Columns: `id` (uuid pk), `vault_id` (uuid FK → vault, unique per vault), `enabled` (bool,
default false), `source_dir` (text), `frequency` (text enum), `interval_minutes` (int),
`last_run_at` (timestamptz nullable), `last_status` (text nullable), `last_error_message`
(text nullable), `max_files_per_run` (int, default 50), `scan_timeout_seconds` (int, default
300), `created_at` (timestamptz), `updated_at` (timestamptz).

**Acceptance criteria — F1-SCHED:**

- AC-S-1: The Settings panel contains an "Automatic import" sub-section. It exposes: an
  enabled toggle, a source folder text input, a frequency selector (Hourly / Daily / Weekly /
  Manual), and a "Save" button. A "Run now" button triggers an immediate scan regardless of
  the schedule. A "Last run" status line shows `last_run_at` (relative timestamp) and
  `last_status` (ok / error / never run).
- AC-S-2 (container path constraint — UI): The source folder input shows a persistent helper
  text: "Must be a container-mounted path (e.g. /data/import). See DEPLOY.md." This text is
  an i18n key and appears in both locales/en.json and locales/it.json.
- AC-S-3 (container path constraint — backend): PUT /import-schedule validates that
  `source_dir` is an absolute path that exists and is readable from within the container.
  If the path is inaccessible, HTTP 422 is returned with a message that explicitly references
  the Docker volume mount requirement. No schedule is saved for an inaccessible path.
  Unit test: submit a path that does not exist on the filesystem → assert HTTP 422 with
  container constraint message.
- AC-S-4: GET /import-schedule returns the persisted configuration for the active vault.
  vitest test: mock response with all fields → assert Settings sub-section renders correctly
  with enabled state, folder path, frequency, last-run info.
- AC-S-5 (I7 bounded scan): each scheduled or manual scan run respects `max_files_per_run`
  (default 50) and `scan_timeout_seconds` (default 300). If the scan runs longer than the
  timeout, it is aborted, `last_status` is set to "error", and the error is logged. The
  scheduler does NOT start a new scan if one is already running (skip + warn). Unit test
  with mock filesystem: assert scan stops at `max_files_per_run` files even when more are
  present.
- AC-S-6 (I1 incremental): the scan imports only NEW files (files not already in the
  `ingest_runs` or `pages` table for the vault). Files already indexed are skipped. Unit
  test: pre-seed 3 already-indexed files + 2 new; run scan; assert only 2 new files are
  ingested.
- AC-S-7 (I5 vault integrity): the scan copies files into vault/raw/sources/ and triggers
  ingest via the existing ingest path. Files from source_dir are NEVER written directly to
