## 2. Acceptance Criteria Per Feature

The following are crisp, testable acceptance criteria handed to functional-analyst
for elaboration into user stories. Each AC maps 1-to-1 to its feature ID.

### F1 — 3-panel shell

- AC-F1-1: The frontend renders exactly 3 resizable panels: left (file tree), center (chat/main), right (preview/graph). Panel resize uses CSS or a drag handle; no layout calculated in JS on every frame.
- AC-F1-2: All long lists in the tree panel (more than 50 items) are virtualized using TanStack Virtual. No non-virtualized list rendering more than 50 DOM nodes at once (I4).
- AC-F1-3: The editor/preview panel uses CodeMirror 6 exclusively. No ProseMirror, no contentEditable WYSIWYG element in the DOM (I4).
- AC-F1-4: The graph sigma viewer from v0.3 is embedded in the 3-panel shell (right panel or as a tab), not removed. Its no-client-layout bundle assertion (T-NCL-001..022) must still pass.
- AC-F1-5: An activity panel (status bar or overlay) shows: current vault, active provider name, last ingest timestamp, data_version.
- AC-F1-6: At least 2 scenario templates are selectable from the UI (e.g. "Summarize document", "Find related pages"); each pre-fills the chat input with a template prompt.
- AC-F1-7: Playwright E2E test: 3 panels are visible in the DOM; resizing left panel to 200px does not cause a layout reflow longer than 16ms (captured via CDP PerformanceMetrics).

### F6 — Multi-conversation persistent chat

- AC-F6-1: Conversations are persisted in Postgres (new `conversations` and `messages` tables). A page refresh restores the last active conversation.
- AC-F6-2: Multiple named conversations can be created, switched between, and deleted. Switching is <100ms (measured by vitest timer mock).
- AC-F6-3: Each assistant message displays `[n]` citation markers that link to the source page (page title shown on hover). Citation data is stored with the message.
- AC-F6-4: A "regenerate" button re-issues the last user prompt to the provider and replaces the last assistant message. The prior message is removed from the UI (not kept in history).
- AC-F6-5: A "save to wiki" button on an assistant message triggers POST /ingest with the message content as source text; the ingest result (page title + link) is shown inline.
- AC-F6-6: Message history list is virtualized (TanStack Virtual) — no more than 30 DOM nodes rendered at once regardless of conversation length (I3, I4).

### F7 — Reasoning `<think>` display

- AC-F7-1: When an assistant message contains `<think>…</think>` tags, the reasoning block is extracted and rendered in a separate collapsible section ("Reasoning"), collapsed by default.
- AC-F7-2: The `<think>` block is NOT stripped from the stored message content; it is stored in full and re-parsed on render (no irreversible mutation of stored data).
- AC-F7-3: Streaming: the `<think>` block accumulates during stream and collapses when `</think>` is received. The main content begins rendering after `</think>`. No heavy parse during stream (I3).
- AC-F7-4: vitest test: a fixture message string with `<think>…</think>` renders the reasoning section collapsed and the main content visible.

### F8 — LaTeX to Unicode

- AC-F8-1: LaTeX expressions in assistant messages are converted to Unicode at stream END (on the `done` or final chunk event). No conversion is triggered per-token (I3).
- AC-F8-2: The conversion covers at minimum: Greek letters (α β γ … Ω), common operators (∑ ∏ ∫ ∂ ∇), arrows (→ ← ↔ ⇒ ⇔), and fractions/superscripts/subscripts for common cases.
- AC-F8-3: Unconvertible LaTeX (complex display math) is left as a fenced code block, not silently dropped.
- AC-F8-4: vitest test: fixture string with 5 LaTeX expressions converts correctly after stream END; no conversion fires during incremental stream appends.

### F14 — Configurable context window

- AC-F14-1: A settings UI (settings panel or modal) exposes a context window size selector: 4K, 8K, 16K, 32K, 64K, 128K, 256K, 512K, 1M (or numeric input validated against these bounds).
- AC-F14-2: The selected size is persisted in the `provider_config` table (new column `context_window_tokens`). Default is 32K if not set.
- AC-F14-3: The retrieval assembler (F5 stub or the chat message builder) respects the budget split: 60% conversation history, 20% retrieved context, 5% system prompt, 15% generation headroom. The split is computed from the configured window, not hardcoded to absolute token counts.
- AC-F14-4: If the conversation history exceeds its 60% allocation, the oldest messages are truncated (not the most recent). The truncation boundary is computed server-side.
- AC-F14-5: Unit test: mock a 10-message conversation against a 4K window; assert messages[0..N] are truncated to keep total ≤ 60% × 4096, and the split percentages are correct.

### F17-UI — Provider Selector

- AC-F17-UI-1: A Provider Selector dropdown is visible in the settings panel or activity panel. It lists all configured providers from GET /provider/config.
- AC-F17-UI-2: Selecting a provider posts to POST /provider/config with the selected provider_type and scope=vault (current vault) or scope=global per a sub-toggle. The selection persists across page refresh.
- AC-F17-UI-3: The active provider name is shown in the activity panel (feeds AC-F1-5).
- AC-F17-UI-4: The dropdown correctly shows 3 provider types: Local (Ollama), API (Anthropic / OpenAI-compatible), CLI (claude-agent-sdk). Each shows its model_id from provider_config.
- AC-F17-UI-5: Changing the provider in the UI does not require a page reload; the next ingest or chat uses the newly selected provider. vitest test: simulate provider change → assert store updates + next mock request uses new provider_type.
- AC-F17-UI-6: No model ID or provider type is hardcoded in any frontend component; all values come from the GET /provider/config response (mirrors backend AC-F17-4 / I6).

### F16-rest — Remaining F16 sub-features

- AC-F16-i18n-1: The UI renders in Italian when the browser locale is `it` or `it-IT` and in English otherwise. Language detection uses `navigator.language`. At minimum the following strings are translated: panel titles, button labels (Ingest, Chat, Save, Settings), error messages.
- AC-F16-i18n-2: Language strings are NOT hardcoded inline; they live in a `locales/en.json` and `locales/it.json` file. vitest test: every key in `en.json` has a corresponding key in `it.json`.
- AC-F16-settings-1: User settings (active provider, preferred language, panel widths, context window size) are persisted in localStorage and restored on page load.
- AC-F16-settings-2: A "Reset settings" action clears localStorage and resets UI to defaults without requiring a hard reload.
- AC-F16-obsidian-1: The backend generates vault/wiki/.obsidian/app.json on startup if absent, and regenerates it if the vault_id changes. Generated config remains a valid Obsidian app.json (JSON-parseable, contains "legacyEditor": false). Extends AC-K7-1 (v0.1).
- AC-F16-gfm-1: Markdown in assistant messages and wiki preview is rendered as GitHub Flavored Markdown: tables, strikethrough, task lists, and autolinks are supported. No custom parser; use a library (e.g. marked or remark with GFM plugin).
- AC-F16-timeout-1: Each provider call (ingest and chat) respects a configurable timeout (default 60s for chat, 120s for ingest). Timeout is stored in provider_config as `timeout_seconds`. On timeout, the UI shows a user-visible error ("Provider did not respond in Xs") and the backend logs the timeout as a provider failure (feeds the I7 bounded-fallback mechanism).

### G3 — Streaming performance gate

- AC-G3-1: Playwright test (new, mandatory): open a chat, send a message, observe streaming. Assert no JS long task >50ms on the main thread during the stream (PerformanceObserver + CDP). This is the first mandatory G3 test; must not be left deferred-to-live at M4 sign-off unless a live environment waiver is explicitly granted by the orchestrator.
- AC-G3-2: Markdown and LaTeX parsing fires EXACTLY ONCE on the `done` event (stream end). Verified by a vitest spy: the parse function call count equals 1 regardless of how many token chunks were emitted.
- AC-G3-3: Zustand store update during streaming appends to a raw string buffer; no selector re-computation fires per token. Verified by vitest: 100 simulated token appends → selector call count = 0 during stream; = 1 after stream ends.
- AC-G3-4: The message history list does not re-render all existing messages when a new token arrives. Verified by vitest render-count spy: existing messages' render count does not increase during streaming of a new message.

### D5-update — Screenshots refreshed

If v0.3 screenshots were never committed (deferred-to-live), this is their first committed version.

## 4. Phasing Recommendation

M4 is the largest sprint in the roadmap: it encompasses a full web UI shell (F1),
a new chat feature set (F6/F7/F8), provider configuration UI (F17-UI), settings (F16-rest),
a performance gate (G3), documentation deliverables (D5/D6), devops carry-forward (NB-6),
and the GraphUX formalization (M4-GUX). It spans an estimated 2 weeks of evening development.

To deliver visibly, keep the human in the loop, and preserve the ability to verify each
layer before building on it, the following 4-phase delivery order is recommended.

### Phase 0 — GraphUX formalization (already done; needs gating only)

Duration: 1–2 sessions (gating + test fixes, no net new code)

Work:
- Functional-analyst updates TRACEABILITY.md: the ADR-0012 §5 fixture expectation for
  type-only pairs P3–P5 changes from "weight == 1.0, present" to "absent" (AC-GUX-1).
  This is the open handoff flagged in ADR-0016 §6.
- QA engineer runs the existing T-GENG suite and confirms it passes with the new engine.
  If any test is RED due to the structural-gate change, it is fixed here before P1 begins.
- Architect and tech-writer review and sign off ADR-0016 (already written; sign-off is the gate).
- D2 regenerated (make er) and D4 regenerated (make openapi) to include the edges.kind column
  and PATCH /pages/{id}/position endpoint.
- D5: seed_demo_vault.py runs on the live stack; captures updated graph screenshots that show
  node-size variation and structural-only edges. These satisfy the outstanding v0.3
  deferred-to-live D5 condition AND provide the baseline for M4 D5-update.

Exit condition for Phase 0: T-GENG suite green with ADR-0016 changes; architect + tech-writer
sign off ADR-0016; D2 + D4 zero drift; at least 1 committed graph PNG showing structural-only
rendering; EC-M3-17 human checkpoint satisfied by Emanuele.

### Phase 1 — 3-panel shell + graph wired in (F1)

Duration: 3–4 sessions

Work:
- Build the 3-panel shell (F1): resizable panels, activity panel, scenario templates.
- Wire the existing sigma graph viewer (from v0.3 + M4-GUX) into the right panel.
- Implement TanStack Virtual for the file tree list (I4).
- Verify T-NCL-001..022 still green (no-client-layout), T-GSTORE, T-GTRANS still green.
- Playwright: capture docs/screens/shell-3panel.png (first D5-update screenshot).

Exit condition for Phase 1: 3-panel shell renders in browser; graph viewer embedded and
functional; no I2/I3/I4 violations; AC-F1-1..7 green.

### Phase 2 — Provider Selector + Settings (F17-UI, F16-rest settings + .obsidian + timeout + i18n)

Duration: 2–3 sessions

Rationale: provider config must be wired before chat is built in P3, because P3's
chat.() call must use the provider that the user selected. Building F17-UI and F16-rest
settings first means P3 can test against real provider switching.

Work:
- Provider Selector dropdown (F17-UI): reads GET /provider/config; posts to POST /provider/config.
- Settings panel: context window (F14), provider (F17-UI), language (F16-i18n), panel widths.
- Settings persistence via localStorage (F16-settings).
- .obsidian auto-gen hardening (F16-obsidian): extend AC-K7-1 to cover vault_id change.
- Multi-provider chat timeout (F16-timeout): provider_config.timeout_seconds column + Alembic migration.
- i18n IT/EN (F16-i18n): locales/en.json + locales/it.json.
- NB-6 (mmdc CI render check): devops-engineer wires mmdc in CI — target this phase since
  it is infrastructure, not UI, and should not block P3.
- Playwright: capture docs/screens/shell-provider-selector.png.

Exit condition for Phase 2: Provider selector visible and functional; settings persist
across reload; .obsidian generated correctly; i18n switches language; mmdc CI green;
AC-F17-UI-1..6, AC-F14-1..5, AC-F16-i18n-1..2, AC-F16-settings-1..2,
AC-F16-obsidian-1, AC-F16-timeout-1 green.

### Phase 3 — Chat + streaming + performance gate (F6, F7, F8, G3)

Duration: 3–4 sessions

Rationale: chat is the most user-visible feature. With F1 (shell) and F17-UI (provider
config) already working, the chat feature can be wired end-to-end immediately. G3 is
gated at the end of this phase (not deferred-to-live this sprint).

Work:
- Conversations and messages tables: Alembic migration; REST endpoints (POST /chat,
  GET /conversations, POST /conversations, DELETE /conversations/{id}).
- Chat streaming: SSE or WebSocket; Zustand buffer appends raw string per token; parse fires
  on stream END only (I3, AC-G3-2).
- `<think>` display (F7): collapsible reasoning section.
- LaTeX-to-Unicode (F8): fires on stream END.
- GFM rendering (F16-gfm): marked or remark; fires on stream END.
- Cited-refs (F6): [n] markers; citation stored with message.
- Regenerate + save-to-wiki (F6).
- Message list virtualization (F6, TanStack Virtual).
- G3 Playwright test (AC-G3-1): no long task >50ms during stream. This test is MANDATORY
  in P3; it must not be deferred-to-live. If the live environment is unavailable, the
  orchestrator must grant an explicit waiver (not automatic).
- D5-update: capture shell-chat-streaming.png.
- NB-7 and NB-8 (cosmetic diagram fixes): tech-writer can fold these into P3 documentation
  pass as optional polish.

Exit condition for Phase 3: Chat streams end-to-end; G3 Playwright test green; F7/F8 pass
vitest; message list virtualizes; D5 screenshots committed; AC-F6-1..6, AC-F7-1..4,
AC-F8-1..4, AC-G3-1..4, AC-D5-upd-1..3 green.

### Docs thread (threaded throughout all phases)

- D6 USER.md and DEPLOY.md: tech-writer drafts in parallel with P1–P3; must be complete
  and reviewed (AC-D6-1..4) before the M4 DoD gate.
- D1 component.mmd: updated by tech-writer to show 3-panel shell, chat components, and

- D7 ADRs: any architectural decision made during M4 that is not already covered by an

  transport choice, conversations table schema, GFM library selection.

## 6. Open Handoff: TRACEABILITY Update Required (ADR-0016 §6)

ADR-0016 §6 explicitly flags a mandatory TRACEABILITY correction that must be performed
by the functional-analyst before Phase 0 can be signed off.

The correction: ADR-0012 §5 documents a worked fixture with a type-only pair P3–P5
(entity pages sharing no links and no shared source). Under the old inclusion rule this
pair produced weight == 1.0 and was stored as an edge. Under ADR-0016 §1's structural gate
this pair produces NO edge (the +1 type term no longer materializes an edge without a
structural tie).

Required TRACEABILITY change: in the Sprint 3 coverage section for AC-F4-1 (T-GENG-001..012),
the expected value for the P3–P5 pair must change from "weight == 1.0, present" to "absent
(no structural tie)". The lower-bound assertions for P1–P2 (≥11), P1–P4 (≥8), and P2–P4 (≥5)
are UNAFFECTED (those pairs are structural via direct link or shared source and their stored
weights are identical).

If the T-GENG test suite itself hardcodes the old expectation, the QA engineer must update
the test fixture to match ADR-0016 before Phase 0 sign-off. This is a one-line correction,
not a re-design. Failing to fix it would leave a GREEN test that asserts the wrong behavior
(an invariant violation: the type-clique hairball would be silently re-introduced on the
next engine.py edit).
