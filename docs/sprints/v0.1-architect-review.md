## 1. Review checklist (per task)

### Task 1 — I1 (incremental index only) → **APPROVE**

- **No directory walk in executable code.** Static scan of `backend/app/` for
  `os.walk` / `os.listdir` / `.rglob` / `.glob(` / `glob.glob` / `iterdir`: **zero hits in
  executable code.** The only `os.scandir` use is `watcher.py::_emit_startup_notice`, which
  opens a scandir iterator and calls `next(it)` **exactly once** inside a `with` block to test
  non-emptiness (O(1)), then logs the AQ-3 INFO line. It does not loop, does not collect
  entries, and triggers no ingest. This is the AQ-3 "no rescan, one INFO line" behaviour from
  ADR-0006, correctly implemented — it is an existence probe, not an enumeration. **Compliant.**
- **Watcher reacts to events only.** `VaultWatcher.start()` registers a watchdog `Observer`
  with a `_MarkdownHandler` and returns; it never enumerates. `on_created`/`on_modified` →
  `ingest_file`; `on_deleted` → `delete_file`; `on_moved` correctly decomposed into
  ingest(dst) + delete(src) for atomic-rename saves. **Compliant.**
- **mtime-then-hash gate matches ADR-0001.** `ingest_file` implements the exact 3-case policy:
  (1) `existing.source_mtime_ns == current_mtime_ns` → SKIP, no read/hash/write; (2) mtime
  differs → read + sha256, and if `content_hash` unchanged → `_touch_mtime` only, SKIP (no
  embed/log/version bump); (3) hash differs or new file → full upsert. Hash is sha256 over raw
  bytes (frontmatter + body), so frontmatter-only edits are detected. The gate lives in the
  seam (shared by watcher and REST), per ADR-0001/0003. **Exact match to ADR-0001.**
- **Soft-delete path correct.** `delete_file` selects the live row (`deleted_at IS NULL`),
  sets `deleted_at = now()` + `updated_at`, leaves all other columns intact, then
  hard-deletes the Qdrant point (asymmetric soft/hard per ADR-0002/0005). No-op + warning if
  no live row. Resurrection is handled in `persist_metadata` (clears `deleted_at` on re-ingest
  of the same path). Does not bump `data_version` on delete (ADR-0005, correct for v0.1).
  **Compliant.**
- No spurious `data_version` bump on skip: `bump_version` is only reached on the
  content-changed branch. Monotonic-by-construction holds (AC-F16dv-4). **Compliant.**

**I1 verdict: APPROVE.** No re-scan anywhere; incremental-only honoured strictly.

### Task 2 — I6 (F17 pluggability preserved) → **APPROVE**

- **Seam is thin and provider-agnostic.** `ingest_file` is the single entry point; both the
  watcher and `POST /ingest/trigger` call **only** `ingest_file` (verified in `watcher.py`
  and `main.py::trigger_ingest`). Neither caller reaches Postgres/Qdrant/embedding directly.
  This satisfies the ADR-0003 review rule.
- **Extension point correctly placed.** The F17 slot sits **after** the hash gate and
  **before** `persist_metadata` (`orchestrator.py` lines ~136–140), exactly where v0.2 will
  resolve a provider from `provider_config`, call `capabilities()`, and branch
  orchestrated-loop vs delegated-CLI. The mechanical primitives (`persist_metadata`,
  `upsert_vector`, `append_log`, `bump_version`) are factored as standalone async helpers, so
  the v0.2 loop reuses them as `validate`/`write_page` without rewriting callers. The
  delegated-CLI vs orchestrated-loop separation (I6 routing) is documented in the module
  docstring as the v0.2 insertion contract. **The seam will accept the F17 loop without a
  caller refactor — design intent preserved.**
- **Zero provider/LLM/model-id leakage.** Static scan for `claude-`, `gpt-`, `anthropic`,
  `openai`, `ollama`, `inferenceprovider`, `provider_config`, `llm`: **no hits in executable
  code.** The only matches are in comments/docstrings describing the v0.2 extension point
  (`# otherwise (OllamaProvider / ApiProvider)` and `v0.1 is MECHANICAL ONLY — no LLM call`).
  No provider instantiation, no provider import, no model id, no `provider_config` table.
  `EMBEDDING_URL` is correctly scoped to vectorisation (I9), not to the F17 abstraction.
  The `graph/`, `mcp/`, `ops/`, `rag/` package dirs are **empty** — no v0.2+ code prebaked.
  **Compliant.**

**I6 verdict: APPROVE.** F17 extension point is correctly seated; v0.2 plugs in without
touching callers. No provider leaked into v0.1.

### Task 3 — I9 (no reinvention) → **APPROVE**

- **Embeddings only via `EMBEDDING_URL`.** `HttpEmbeddingClient` POSTs to
  `settings.embedding_url` with `{"model": settings.embedding_model, "prompt": text}`. No
  in-process model load, no subprocess spawn. `EmbeddingClient` ABC + `FakeEmbeddingClient`
  is a test seam (GAP-4), not a second backend. **Compliant.**
- **Qdrant only via `QDRANT_URL`.** `AsyncQdrantClient(url=settings.qdrant_url)`; thin
  wrapper exposing `ensure_collection` / `upsert_point` / `delete_point`. No new vector store.
  **Compliant.**
- **No new vector/embedding service introduced.** `docker-compose.yml` defines only
  `postgres` + `synapse-backend`; Qdrant and bge-m3 are external (reached over the network),
  per the I9 reuse mandate. No Tavily reference anywhere (`grep`: zero hits). **Compliant.**
- **`EMBEDDING_DIM` is config, not a literal.** `Settings.embedding_dim: int` is a **required**
  env var (no default → pydantic fails fast if absent). `.env.example` documents `1024`. Scan
  for `1024`/`768`/`1536` in executable code: **zero hits** (only a docstring mention in
  `config.py`). The Qdrant collection vector size flows from `settings.embedding_dim`, and
  startup validates it against the live bge-m3 via `probe_dimension()` (ADR-0004 fail-fast).
  `ensure_collection` refuses to silently recreate a dimension-mismatched collection.
  **Compliant.**

**I9 verdict: APPROVE.** Existing infra reused; no reinvention; dimension is config-driven.

### Task 4 — Data model fidelity (matches spec + migration matches models = D2 truth) → **APPROVE**

`pages` columns vs spec (architecture §2.1): all present and typed as specified —
`id` (UUID PK, default `gen_random_uuid()`), `vault_id`, `file_path`, `title`,
`type` (mapped as `page_type`, column name `"type"`), `sources` (JSONB), `content_hash`
(String(64)), `source_mtime_ns` (BigInteger), `qdrant_point_id` (UUID), `deleted_at`
(TIMESTAMPTZ), `created_at`, `updated_at`. **12/12 columns — exact match.**

- Partial unique index `uix_pages_vault_file_path_live` on `(vault_id, file_path)
  WHERE deleted_at IS NULL` — present in `models.py __table_args__` AND in the Alembic
  migration (`postgresql_where=sa.text("deleted_at IS NULL")`). Enforces one live row per
  path at the DB level (ADR-0005's earlier "application-logic only" gap is now closed in the
  schema). **Compliant.**

`vault_state` columns vs spec: `id` (UUID PK), `vault_id` (String + `UniqueConstraint
uq_vault_state_vault_id`), `data_version` (Integer, server_default 0), `updated_at`. **4/4 —
exact match.** Monotonic semantics enforced in code (`bump_version` is the only writer that
increments; seed/restart/GET/skip/delete never change it). **Compliant.**

- **Migration matches models (D2 reflects reality).** `alembic/versions/0001_initial_schema.py`
  creates both tables with identical column names, types, nullability, server defaults,
  PK/unique constraints, and the partial unique index. Column `"type"` is correctly emitted in
  SQL (not the Python attribute `page_type`). `provider_config` correctly absent (ADR-0003).
  The committed `docs/er/schema.mmd` reflects all 12 `pages` + 4 `vault_state` columns including
  `source_mtime_ns` (the ADR-0001 superset) and `deleted_at`. The model is the single source of
  truth and `make er` output matches it. **Compliant.**

> Non-blocking note (v0.2 hygiene, not a defect): `gen_random_uuid()` is native to
> Postgres 13+; the target stack is Postgres 16, so no `pgcrypto` extension dependency is
> introduced. No action required for v0.1.

**Data model verdict: APPROVE.** Models match spec; migration matches models; ER reflects reality.
