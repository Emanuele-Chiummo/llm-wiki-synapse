## Critical checks (recently-found bug classes)

### 1. chat() signature (I6) — **PASS**

The ABC signature is locked: `async def chat(self, messages, retrieval_context)` (`provider/base.py:106`) — 2 args, no `token_budget`. `review.py:_single_chat_call` calls `provider.chat(messages, "")` (`:267`) and enforces the budget at the orchestration layer via `budget_chars = token_budget * _CHARS_PER_TOKEN` (char/4, `:264`), breaking the async-gen loop once `total >= budget_chars` and calling `aclose()`. It consumes the dual shape (`maybe = provider.chat(...); agen = await maybe if inspect.isawaitable(maybe) else maybe`, `:267-268`) exactly like `chat/stream.py`. The prior `token_budget=` kwarg bug is fixed and the budget is actually enforced (truncates accumulated output, not a no-op). Timeout wraps the whole consumption (`:211`), so it covers network latency, not just first-token.

### 2. deep-research run_id threading (no C1 recurrence) — **PASS**

`review.py:deep_research` mints `run_id` (`:395`), pre-INSERTs the `DeepResearchRun` row with status='running' and frozen bounds (`:403-422`), stores `run_id_str` on the review row (`:438`), then schedules `run_deep_research(..., run_id=run_id)` (`:445-453`). `run_deep_research` reuses the caller's row when `run_id is not None` and only mints+inserts when it is None (`deep_research.py:152-160`) — the documented C1-safe path. No orphan row, no double-insert, no stuck-"running". The manual pre-insert mirrors `_create_run_row` field-for-field (all NOT-NULL columns populated), so the INSERT will not violate schema.

### 3. F9 hook safety — **PASS**

`run_ingest_pipeline` runs the hook only on the orchestrated branch, AFTER `write_wiki_page` + `_update_overview` (`orchestrator.py:367-388`). The hook is wrapped in `try/except Exception … log WARNING` (`:377-387`). Inside `_enqueue_review_items`, the excerpt read is itself guarded (`:974-996`), and `generate_review_queries` never raises (it returns `None` on any provider/config error — `review.py:178-230`). A generator failure cannot propagate into the ingest caller or affect `IngestRunResult`. Delegated (CLI) branch correctly does NOT invoke the hook (ADR §3.3.4 reserved gap, acknowledged).

### 4. F12 companion flow (I1/I5) — **PASS**

Binary upload path (`main.py:1132-1176`): (1) original binary atomically moved to `raw/sources/` and preserved; (2) `extract_text(dst)` synchronously; (3) companion `<stem>.extracted.md` written with valid Obsidian frontmatter — `type: source`, `title`, `sources: ["raw/sources/<name>.<ext>"]` (`:1151-1152`, F3 traceback, I5); (4) returns 202. The watcher then fires on `.extracted.md` (matches `.md` in the unchanged `_ALLOWED_EXTENSIONS`) and the binary is ignored (its extension is not watched) → one `ingest_file`, one hash gate, one `pages` row, one Qdrant point (I1). Extraction failure is non-fatal: binary stays saved, no companion, watcher silently skips it (`:1166-1174`). `_ALLOWED_EXTENSIONS` is unchanged.

## Conditions

1. **(Non-blocking, must-track) DB-level CHECK constraints absent.** ADR-0025 §3.1 specifies
   CHECK constraints enumerating `item_type` (`new_page|update_page|deep_research_candidate`)
   and `status` (`pending|approved|skipped|deep_researched`). The model documents them as
   "enum-by-convention (CHECK constraint)" but migration 0010 creates plain `Text` columns with
   no CHECK. Enforcement today is application-level only (handler `Literal` types + status
   helper). This does not violate an invariant and does not gate Phase 3, but it is a deviation
   from the signed ADR. Resolve by either (a) adding the two CHECK constraints in a follow-up
   migration, or (b) amending ADR-0025 to record enum-by-convention as the accepted decision.
   Track to M5 close.

2. **(Gate housekeeping) D4 regeneration.** Confirm `make openapi` has been run so
   `docs/api/openapi.json` carries the 4 `/review/queue*` routes and the widened
   `/ingest/upload` accept-list; confirm `make er` output (`schema.mmd`) is committed (it is).
   Zero-drift is I8 — verify in the docs gate before phase sign-off.

## Notes / observations (no action required)

- `list_queue` queries ALL statuses (not just pending) so the UI can show the full queue; the
  store filters actioned items out optimistically. Consistent with ADR §3.5 ("the full queue")
  and the event-log model (§3.1). Acceptable.
- Reserved gaps are explicitly coded, not silently dropped: delegated-CLI enqueue (ADR §7.1),
  consistency scan (§3.4), binaries-outside-upload auto-extract (§4.3). Each is gated behind a
  future ADR. No invariant tension introduced.
- The `_query_token_budget` default (2000) and `_query_timeout` (30s) read from `settings` at
  call time with safe fallbacks — bounded per I7, logged via `UsageAccumulator` (`review.py:233`).
- No new container/component topology — `review.py`/`extract.py` are components inside the
  existing FastAPI service. D1 (C4) unchanged, as ADR-0025 §8 handoff predicted.
