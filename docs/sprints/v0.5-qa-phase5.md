## 3. Nothing-Hardcoded Confirmation (I6 + I9 — Headline)

### Backend

The `get_mcp_info()` handler at line 1626 of `backend/app/main.py`:
- Calls `await _mcp_server.list_tools()` to source the tool list; builds `McpToolInfo` from `t.name`, `t.description`, `t.parameters` — no string literal for any tool name, description, or schema.
- Returns `server_name=_mcp_server.name` — derived from the live FastMCP object.
- Returns `transport=settings.mcp_transport` — from `app/config.py`, env var `MCP_TRANSPORT`, default `"stdio"`.
- Returns `entry_point_command=settings.mcp_entry_command` — from `app/config.py`, env var `MCP_ENTRY_COMMAND`, default `"python -m app.mcp.server"`.
- No DB query (`get_session` is not called). No Qdrant call. No MCP stdio transport opened.

Test `test_mcp_info_tools_match_live_registry` specifically verifies that the returned tool names equal exactly `{t.name for t in await mcp.list_tools()}` — a dynamic check that would fail if the handler returned a hardcoded list.

Tests `test_mcp_info_entry_point_command_from_settings` and `test_mcp_info_transport_from_settings` assert the response values equal `settings.mcp_entry_command` and `settings.mcp_transport` respectively.

### Frontend

`SectionApiMcp` in `frontend/src/components/settings/SettingsPanel.tsx`:
- Calls `fetchMcpInfo(signal)` on mount; renders from `info.tools` list exclusively — no JSX literal for any tool name or description.
- `buildClaudeDesktopSnippet(info)` tokenises `info.entry_point_command` and keys the snippet by `info.server_name` — both from the API response.
- No Zustand store created for this panel (local `useState` only, mirrors `SectionEmbeddings`).

The `mcpClient.test.ts` suite includes a dedicated describe block ("Claude Desktop config snippet — tokenisation contract") with 6 tests verifying that the snippet is derived from the payload and not from hardcoded strings (including a test with a custom `server_name: "custom-server"` that asserts the hardcoded key "synapse" does NOT appear).
