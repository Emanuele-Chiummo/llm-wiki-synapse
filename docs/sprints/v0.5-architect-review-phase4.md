## Verdict: **APPROVE-WITH-CONDITIONS**

The implementation faithfully realises ADR-0026: single-pass, inference-free, loop-free; references found from the `links` back-reference index (no vault walk on the happy path); preserve-shared partition is exact; dead-link rewrites are frontmatter-safe via `python-frontmatter` round-trip; `data_version` is bumped exactly once; no migration was added (D2 unchanged by F13); soft-delete only; raw source file removed. All 10 Do-NOT triggers are HELD in the code.

Four conditions below are correctness/robustness refinements, none of which trade an invariant. C1 (transaction atomicity) is the only one I rate above cosmetic, and it is a hardening item, not an invariant breach — the operation is still correct on the success path. None block the phase gate provided they are tracked.

## 3-method matching + preserve-shared + no-rescan cleanup — confirmation

### 3-method matching (AC-F13-1/-2)
- **(a) exact** — `_method_a_exact`: `SELECT source_page_id, target_title FROM links WHERE target_page_id == page_id OR target_title == T`, then resolves `file_path` only for **live** wiki pages (`deleted_at IS NULL`). Pure index read (`:203-243`). **Confirmed.**
- **(b) slug** — `_method_b_slug`: reads `links` rows whose `source_page_id` is not already found, compares `_slugify(target_title) == _slugify(T)` in Python. No filesystem walk (`:246-289`). **Confirmed.**
- **(c) full-text fallback** — `_method_c_fulltext`: enumerates live wiki pages from the `pages` table (NOT a vault walk), bounded by `CASCADE_FULLTEXT_MAX_FILES` (default 5000, env-overridable), reads each candidate file once to substring-match (`:292-343`). **Confirmed bounded.**
- **Method labels logged** — one `logger.info` per found reference with its method label: (a) `:455-460`, (b) `:480-485`, (c) `:507-511` / `:336-341`. `match_methods_used[fp]` records the winning method (`:461`, `:486`, `:512`). **Confirmed (AC-F13-2 log assertion satisfied).**

### No-rescan linchpin (the §3.1 happy-path gate) — **CONDITION C3**
ADR §3.1 / §8 #1 and the §9 gate require that **(c) is SKIPPED on the links-table-hit happy path**. The current code **always runs (c)** (`:502-503`), merely de-duplicating its results against (a)∪(b) by `source_page_id`/`file_path`. So on the happy path (c) still enumerates and **opens every live wiki file** for substring scanning — the exact disk-read the ADR says to avoid on that path.

This does NOT violate I1 as an *indexing strategy* (it is not re-deriving the index, not re-embedding, not rebuilding the graph), so I do not rate it a Do-NOT #1 reject. But it **does not satisfy the ADR §3.1 "skipped entirely when (a)∪(b) already covers the candidate set" gate**, and it makes the "(c) NOT invoked on the happy path" test (gate item §9.2) impossible to pass against the current control flow. This is **Condition C3** — see below.

### Preserve-shared (AC-F13-2/-3)
- Partition is exact (Do-NOT #6 evidence). When the target is a **wiki page** (not a source), `will_delete = [target]` and the partition is empty — the rule degenerates cleanly (`:563-572`; `_partition_shared_wiki_pages` only runs for `raw/sources/` targets). **Confirmed.**
- Shared-entity warnings are advisory: `_get_shared_entity_warnings` reads `kind='source'` edges, logs a WARNING, and the delete always proceeds (`:391-425`, warnings emitted `:627-628`, never gate the flow). **Confirmed non-blocking.**

### No-rescan dead-link cleanup (AC-F13-1 step 6, AC-F13-4a/b)
- Only files in `plan.wikilinks_to_rewrite` are opened for writing (`:656`). Each opened once, body-only rewrite, `persist_links` re-run per file (`:679-685`). **Confirmed targeted.**
- `index.md` regenerated exactly once via `update_index` (`:711-717`). **Confirmed.**
- `files_written` is counted and returned (`:670`, `:744`). See C2: `files_written` can legitimately be `< len(wikilinks_to_rewrite)` when a rewrite produces no body change (`:662-664` `continue`), so the strict `files_written == len(plan.wikilinks_to_rewrite)` equality from AC-F13-4a is **not** guaranteed by this code. Flagging for QA — see C2.

### Preview read-only / DELETE-only mutation
- `plan_cascade_delete` issues only `select` statements; no `update`/`delete`/file write/`bump`. **Confirmed read-only.**
- `POST .../preview` → 200, 404 on missing/soft-deleted; `DELETE /pages/{id}` → 200, 404 on double-delete (idempotent). Matches ADR §6.1 shape. **Confirmed.**

### Migration / D2
- Highest migration is `0010_review_items`; **no 0011 added**. F13 reuses `deleted_at`, `links.dangling`/`target_page_id`, `edges` endpoint indexes, `pages.sources` JSONB, `vault_state`. The +163-line `schema.mmd` delta vs `main` traces to the earlier M5 commit `254d27d` (F9/F12), not to F13. **D2 unchanged by F13 — confirmed.**

## Conditions

**C1 — Multi-statement atomicity (robustness; track, not a gate-block).**
`cascade_delete` opens a **separate `get_session()` transaction per step** (soft-delete `:633`, dangling `:693`, edge-delete `:699`, prune `:799`, bump `:831`), each auto-committing on exit (`db.py:51`). A crash between steps leaves a partially-applied delete (e.g. pages tombstoned + Qdrant points gone, but `links.dangling` not yet set, or `index.md` not regenerated). The vault is not *corrupted* (no dead links surface because the page row is gone from the live index query, and a re-run is idempotent on the already-soft-deleted target → 404), but the stores can be transiently inconsistent until a manual re-run. **Condition:** document the non-atomic, idempotent-on-retry semantics in the ADR (a Consequences note) OR wrap the DB mutations in a single session/transaction in a follow-up. Acceptable for M5 ship as-is given idempotency; must be tracked.

**C2 — `files_written == len(wikilinks_to_rewrite)` is not strictly guaranteed (test-contract alignment).**
The rewrite loop `continue`s without incrementing `files_written` when `_rewrite_body` yields no change (`:662-664`), and on a write exception (`:676-677`). AC-F13-4a asserts equality. Either (a) tighten the plan so `wikilinks_to_rewrite` only contains files that will actually change (the `_count_body_occurrences > 0` guard at plan-build time `:464`/`:489`/`:515` should already ensure this, making the `continue` dead code on the happy path), or (b) relax the AC assertion to `files_written <= len(...)`. **Condition:** confirm with QA which contract holds; align code or AC so the equality assertion is deterministic. No invariant impact.

**C3 — (c) fallback is not skipped on the happy path (ADR §3.1 / §9.2 gate).**
As detailed above, `_build_wikilink_rewrites` unconditionally calls `_method_c_fulltext` (`:502-503`). The ADR mandates (c) be skipped when (a)∪(b) already covers the candidate set, and the review gate requires a test proving (c) is not invoked on a links-hit fixture. **Condition (required before phase-gate close):** add an explicit gate — e.g. skip (c) when `match_methods_used` is non-empty AND there were no dangling/title-ambiguous links unaccounted for, or introduce a `CASCADE_FULLTEXT_FALLBACK` guard that defaults to running (c) only when (a)∪(b) returned zero references. This is the one condition that touches a named invariant rationale (I1/I7 disk-read minimisation) and the §9.2 exit criterion directly; it must be resolved or the §9.2 gate item formally waived by the orchestrator.

**C4 — Method (b)/(c) `not_in(...)` over the full `links`/`pages` set + `or_(..., True)` (minor; performance/typing).**
`_method_b_slug` reads **all** `links` rows when `already_found` is empty (`:264-268`, the `else True` branch) and slug-normalises in Python; `_method_c_fulltext` similarly. Correct, but on large vaults (b) materialises the whole `links` table. Combined with C3, the happy path currently does a full `links` scan (b) + full wiki-file read (c). **Condition (advisory):** scope (b) to rows sharing the target slug's first-letter bucket or push the slug comparison into SQL where feasible; gate (c) per C3. Not an invariant breach; bounded by `CASCADE_FULLTEXT_MAX_FILES`. Track for a perf follow-up.

## Notes for QA (test-quality, not code — flagged per request)

The backend tests use `AsyncMock` sessions rather than in-memory SQLite. **Assessment of CODE confidence under that test style:**
- Do-NOT #1/#7/#8/#10 (no glob, read-only preview, no loop/provider, soft-delete) are **statically verifiable from the code** and do not depend on a real DB — AsyncMock does not undermine them.
- Do-NOT #3 (single bump), #6 (preserve-shared partition logic), #4/#5 (frontmatter-safe rewrite), #9 (file unlink) involve **real data round-trips** (JSONB partition, SQL UPDATE counts, on-disk frontmatter). With AsyncMock the SQL is not actually executed, so a test can pass while asserting only that `.execute` was *called* — it cannot prove the `deleted_at` UPDATE matched the right rows, that the partition query returns the right pages, or that the `links.dangling` UPDATE actually flipped flags. **Recommendation to QA:** the frontmatter round-trip (#4/#5) and the preserve-shared partition (#6) deserve at least one real-filesystem + real-(SQLite/PG)-session integration test each, since those are the I5 and AC-F13-3 headline behaviours and AsyncMock cannot enforce them. `test_obsidian_check.py` 15/15 after a cascade delete (gate §9.4) must run against a real vault, not mocks.
