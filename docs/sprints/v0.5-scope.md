## 1. Sprint Objective

Deliver "feature parity core" — the set of features that transform Synapse from a usable
prototype into a system that can handle real-world research workflows. The headline theme
is **depth over breadth**: F5 retrieval closes the chat-quality loop; F10 Deep Research
adds autonomous web-sourcing; F9 HITL review surfaces AI outputs for human curation; F12
multi-format ingest removes the .txt/.md constraint; F13 cascade deletion makes the vault
self-healing.

The **headline invariant** for M5 is **I7 (bounded loops)**. F10 Deep Research is the most
loop-intensive feature in the entire roadmap — its multi-query SearXNG loop has explicit
max_iter and token_budget caps, an assess-sufficiency gate before each query, and a
hardcoded concurrency ceiling of 3. Any PR that introduces an unbounded loop anywhere in
M5 work is a P0 blocker.

The **headline reuse rule** for M5 is **I9**: SearXNG is the only permitted web-search
backend. Tavily must never be introduced. Qdrant + bge-m3 are the embedding stack. The
existing OllamaProvider / ApiProvider / CliAgentProvider must not be bypassed — all new
inference calls go through `InferenceProvider`.

## 4. Acceptance Criteria Per Feature

### F5 — 4-phase RAG retrieval

- AC-F5-1: `backend/app/rag/retrieval.py` implements all 4 phases in sequence:
  (1) tokenized keyword search over Qdrant (bge-m3 vectors + BM25 hybrid if available);
  (2) graph-expansion: for each top-k result, follow edges in the graph (links + shared-source)
  up to depth 1 or 2; (3) token-budget allocation per F14 20%-retrieved-context slot;
  (4) context assembly — passages ranked by relevance, truncated to budget, returned as
  `RetrievalContext` with page refs.
- AC-F5-2: Each retrieved passage carries a citation index `n` (1-based). The assembled
  context string includes `[n]` markers inline. The `RetrievalContext` object maps each `n`
  to a `PageRef` (id, title, url-friendly slug).
- AC-F5-3: The retrieval pipeline calls only Qdrant (bge-m3) and reads from the Postgres
  `links` and `edges` tables — no new embedding service introduced (I9). Static import check
  confirms no direct bge-m3 or Ollama embedding calls outside the existing Qdrant wrapper.
- AC-F5-4: The token budget for retrieved context (20% of configured window per F14) is
  respected. If candidate passages exceed the budget, the lowest-ranked passages are dropped.
  Unit test: 10 fixture passages totaling 2× budget → assert assembled context ≤ budget.
- AC-F5-5: Graph-expansion phase uses the existing `edges` table (Postgres) — does NOT
  trigger a graph recompute or FA2 run (I2 preserved). Verified: data_version is unchanged
  after a retrieval call.
- AC-F5-6: `GET /search?q=<query>&vault_id=<id>` endpoint returns the top retrieval results
  with citation indices. Response documented in openapi.json (D4 zero-drift gate).
- AC-F5-7: Unit tests cover: 0-result query, single-result, multi-page expansion, budget
  overflow truncation. Integration test: seed 5 pages with known wikilinks → retrieval
  returns linked pages in expansion phase.
- AC-F5-8: After F5 ships, the chat endpoint passes the assembled `RetrievalContext` to
  the active provider's `chat()` call so that `[n]` markers appear in assistant messages
  (unblocks F6 AC-F6-3 and F17 CliAgentProvider.chat()).

### F10 — Deep Research loop

- AC-F10-1: `backend/app/ops/deep_research.py` implements the full loop:
  (1) generate 2–5 SearXNG search queries from the research topic (via InferenceProvider);
  (2) execute queries against SearXNG (I9: never any other search backend); concurrency=3
  (asyncio gather with semaphore); (3) fetch and parse result pages (html → markdown);
  (4) assess sufficiency: call InferenceProvider to judge whether collected content answers
  the research topic; if yes → proceed to synthesize; if no → generate follow-up queries
  (up to max_iter); (5) synthesize: call InferenceProvider to produce a synthesis document;
  (6) auto-ingest: feed synthesis into the ingest pipeline (not directly into vault/wiki/;
  must go through the orchestrated loop to satisfy I6).
- AC-F10-2 (I7 — headline invariant): Every deep research run is strictly bounded:
  `max_iter` (configurable, default 3 refinement rounds), `token_budget` (configurable,
  default 100K tokens), `max_queries_per_iter` (default 5), `concurrency` (hardcoded 3,
  not configurable beyond 3 without architect approval). `total_cost_usd` logged per run
  in a new `deep_research_runs` table. Any PR that removes or weakens these bounds is a
  P0 blocker.
- AC-F10-3 (I9): SearXNG is the only web search backend. The SearXNG base URL is read from
  `SEARXNG_URL` env var (already in the stack). No Tavily, DuckDuckGo, or Google API may be
  introduced. Static import test: no `tavily`, `ddg`, or `google` package imports anywhere
  in the codebase.
- AC-F10-4: The deep research run is triggered via:
  - POST /research/start — body: {vault_id, topic, max_iter?, token_budget?}; returns HTTP
    202 with {run_id: uuid};
  - GET /research/runs — list of past runs with status + cost;
  - GET /research/runs/{id} — details of a run including queries used, sources fetched,
    synthesis text.
  All endpoints in openapi.json (D4 zero-drift gate).
- AC-F10-5: F9 integration: when F9's "Deep-Research" action fires, it calls POST
  /research/start with the pre_generated_query as topic. The run_id is stored on the
  review_item row for traceability.
- AC-F10-6: New Postgres tables: `deep_research_runs` (id, vault_id, topic, status, queries
  used JSONB, sources_fetched int, total_cost_usd, started_at, completed_at, error_message)
  and `deep_research_sources` (id, run_id, url, title, fetched_content_md, relevance_score).
  Alembic migration. D2 regenerated via `make er`.
- AC-F10-7: Unit tests: mock SearXNG returning 3 results → assert 3 fetch calls; mock
  InferenceProvider returning "sufficient" on first assess → assert loop stops after 1
  round; mock InferenceProvider returning "insufficient" for max_iter rounds → assert loop
  stops at max_iter, status="max_iter_reached".
- AC-F10-8: Playwright E2E: navigate to "Deep Search" nav section (previously placeholder);
  enter a topic; assert POST /research/start fires; assert run appears in runs list.

### F13 — Cascade deletion

- AC-F13-1: `backend/app/ops/cascade_delete.py` implements `cascade_delete(page_id)`:
  (1) identify the target page in Postgres; (2) find all wikilinks pointing TO this page
  (links table, target_title match); (3) remove the target page's Postgres row (soft delete:
  deleted_at = now()); (4) remove the Qdrant vector for the page; (5) remove the page's
  entry from vault/wiki/index.md; (6) scan all pages in vault/wiki/ that contain
  `[[<target_title>]]` wikilinks and replace them with `<target_title>` (dead wikilink
  becomes plain text); (7) update the dangling=True flag on affected links table rows.
- AC-F13-2: 3-method matching for finding references: (a) exact wikilink title match in
  links table; (b) slug-normalised match (lower-case, hyphenated); (c) full-text scan of
  wiki/ .md files as fallback. Method used is logged per deletion.
- AC-F13-3: Shared entities are preserved: if a page being deleted has content that is
  also referenced by other pages (via source overlap in the `edges` table), the deletion
  logs a warning listing the sharing pages and proceeds — it does NOT block. The human
  operator is responsible for reviewing the warning.
- AC-F13-4 (I1): Cascade delete only touches records related to the deleted page. It NEVER
  triggers a full vault rescan or graph recompute (data_version IS bumped so the graph
  eventually updates via the debounce, but no immediate FA2 run is forced).
- AC-F13-5: REST endpoint: `DELETE /pages/{id}` triggers cascade_delete(page_id). Response:
  HTTP 200 with {deleted_page_id, wikilinks_cleaned int, index_entry_removed bool,
  shared_entity_warnings list[str]}. Documented in openapi.json (D4 zero-drift gate).
- AC-F13-6: A "Delete page" action is available in the wiki page view (frontend). It shows
  a confirmation modal listing any shared-entity warnings before executing. On confirm, calls
  DELETE /pages/{id}; on success, navigates back to the page tree.
- AC-F13-7: Unit tests: delete a page with 3 incoming wikilinks → assert all 3 cleaned;
  delete a page with shared source overlap → assert warning logged, deletion completes;
  double-delete (page already deleted) → assert HTTP 404 gracefully.

## 6. Invariants in Force — I7 and I9 Are First-Class This Sprint

All 9 invariants apply. I7 and I9 are the **headline invariants** for M5.

### I7 — LOOPS ARE BOUNDED (headline for M5)

F10 Deep Research is the most loop-intensive feature in the roadmap. Every component of
I7 must be demonstrably satisfied in all M5 loops:

1. `deep_research.py`: `max_iter` + `token_budget` + `concurrency ≤ 3`. No exceptions.
2. `review.py`: pre_generated_query generation is bounded to 1 call per item with timeout.
3. F9 Deep-Research action: delegates to `deep_research.py` — inherits its bounds.
4. F17 CliAgentProvider.chat(): the delegated agent loop must have `max_iter` + `token_budget`.
5. F12 extraction: no loop (stateless per-file extraction). Not in scope for I7.
6. F13 cascade_delete: single pass (not a loop). data_version bump triggers the existing
   debounced recompute — already bounded by GraphCache (from v0.3).
7. Every new run (deep_research_runs, any future loop log) must log `total_cost_usd`.
   Anomaly threshold per ADR-0009: flag if > $1.00 per run.

Violation of I7 in any M5 feature is a P0 blocker. QA must include a specific I7 test
for F10: mock InferenceProvider always returns "insufficient" → assert loop terminates at
max_iter, status = "max_iter_reached", no further calls made.

### I9 — DO NOT REINVENT (headline for M5)

SearXNG is already running on the developer's TrueNAS stack. F10 must reuse it.

1. F10: SearXNG base URL from `SEARXNG_URL` env var. Never Tavily, DuckDuckGo, or Google.
2. F5: Qdrant + bge-m3 already running. No new embedding service. No new vector DB.
3. F12: pypdf / python-docx / python-pptx / openpyxl / unstructured are extractors, not
   inference providers. They must not make LLM calls directly — all inference goes through
   InferenceProvider (I6 also applies).
4. F13: Uses existing Postgres `links` and `pages` tables. No new data structure.

### I1 — INCREMENTAL INDEX ONLY

F13 (cascade delete) is the highest I1 risk this sprint: it modifies vault/wiki/ files.
The implementation MUST:
- Only touch files that reference the deleted page (not all files).
- Not trigger a full watcher rescan.
- Use targeted file updates (read → modify → write specific files only).

### I6 — PLUGGABLE INFERENCE

F10 Deep Research uses InferenceProvider for (a) query generation, (b) sufficiency
assessment, (c) synthesis. All three calls go through the active provider's methods.
No direct Ollama/Anthropic/claude-agent-sdk calls in `deep_research.py` — these must
route through the existing `InferenceProvider` abstraction. Static guard T-CQ-009 remains.

### I5 — OBSIDIAN COMPATIBILITY

- F13: dead wikilinks cleaned in wiki/ files must not break YAML frontmatter.

### I2 — GRAPH LAYOUT IS SERVER-SIDE AND CACHED

F13 data_version bump triggers the debounced recompute — already server-side (no change).

### I3 — NO PER-TOKEN HEAVY WORK IN CHAT

## 10. Out-of-Scope Boundary Reminders

Critical boundaries that engineers must not cross during M5:

**F9 is NOT the Ingest Activity View.** F1-INGEST-VIEW (done in M4) is read-only history
+ a trigger button in the Sources nav section. F9 is a separate Review nav section with
queue management and AI-driven action routing. These are different features on different
nav sections. Any engineer adding Approve/Skip/Deep-Research actions to the Sources section
(F1-INGEST-VIEW) during M5 is out of scope and will be blocked.

**F10 SearXNG only.** If during F10 implementation an engineer discovers that SearXNG is
not accessible in the test environment, the correct response is to fix the SearXNG
connection (check SEARXNG_URL env var, check container networking). It is NOT to substitute
Tavily or any other search backend — that violates I9 and requires architect escalation.

**F12 extraction only, not new inference.** The multi-format extractors (pypdf, python-docx,
etc.) extract text. They do NOT call any LLM for OCR, transcription, or summarization.
Extracted text is passed as-is to the existing `analyze()`/`generate()` pipeline.

**F13 targeted edits only.** Cascade delete must use targeted file modifications (not
write all wiki files). A PR that rewrites vault/wiki/index.md or vault/wiki/*.md files
en masse violates I1 and will be blocked.

## 11. Velocity Note

M4 was the largest sprint delivered ON SCOPE with all 25+ original items delivered within
the 2-week envelope (plus M4-HARD hardening). M5 is the largest sprint by feature count:
5 major features + 2 doc deliverables + 3 carry-forward ACs. The phased delivery plan
(4 phases, each with independent gate chain) is designed to keep this manageable for a
solo evening developer. Each phase is scoped to 3–4 evenings.

Risk assessment:
- F10 (Deep Research) is the highest technical risk: most loop-complexity, I7 compliance
  scrutiny, live SearXNG dependency. Placed in Phase 2 with its own gate before F9 depends
  on it.
- F13 (Cascade delete) is the highest vault-integrity risk: directly modifies wiki/ files.
  Placed last (Phase 4) when the vault is populated and stable.
- F5 (Retrieval) is the highest architectural risk: first full RAG pipeline. Placed first
  (Phase 1) to unblock carry-forwards quickly and validate the architecture.
- F12 (Multi-format) is low risk: extractors are well-established Python libraries. Can
  move to Phase 3 without concern.

If during execution any phase proves larger than estimated, the FIRST escalation action is
to the orchestrator — not to silently defer ACs or reduce scope without PM approval.
