## 1. Sprint objective

Make the knowledge graph live with no main-thread layout freeze. This sprint introduces the
first frontend code in the project — a thin, focused sigma.js viewer page, not the full
3-panel shell (F1 is v0.4). The headline invariant is I2: FA2 layout runs server-side via
igraph, coordinates are stored in Postgres, and the client renders precomputed coords in
WebGL only. The browser NEVER runs a force-directed layout algorithm on the main thread.

This is also the first sprint where performance gates G1–G4 are mandatory acceptance criteria
per the QA charter.

## 4. Invariants in force — I2 and I4 are first-class this sprint

All 9 invariants apply. I2 and I4 are the DEFINING acceptance gates for M3.

### I2 — GRAPH LAYOUT IS SERVER-SIDE AND CACHED (headline invariant)

This is the raison d'etre of M3. Every component of I2 must be demonstrably satisfied:

1. FA2 runs ONLY in `backend/app/graph/engine.py` via igraph (Python). NEVER in the browser.
2. Coordinates (x, y per node) are written to Postgres after every successful FA2 run.
3. Recompute is triggered ONLY by a dataVersion bump, with a debounce window. The second
   `GET /graph` call (with same dataVersion) serves cached coordinates — no re-layout.
4. The `GET /graph` response is a static payload of precomputed coordinates. The client
   calls sigma.js `graph.addNode(id, {x, y, ...})` with these coords; it does NOT call any
   sigma or graphology layout function (ForceAtlas2, random, circular, or any other).
5. Violation of I2 is a P0 blocker. Any PR that introduces client-side layout code — even
   temporarily — MUST be escalated to solution-architect before merge.

### I4 — EDITOR IS CODEMIRROR 6; LONG LISTS VIRTUALIZED

The sigma.js viewer itself renders in WebGL (a canvas element), not as DOM nodes per graph
node. This satisfies the "no unbounded DOM growth" principle of I4 for the graph.

In v0.3 the only list in the viewer is potentially a search/filter result list if one is
added. If any list of nodes/pages is rendered in the DOM (e.g., a sidebar), it MUST be
virtualized with TanStack Virtual if it can exceed 50 items. For the thin v0.3 viewer, if no
such list is present, I4 is satisfied by default.

No CodeMirror editor is introduced this sprint (no editor surface exists yet).

### I1 — INCREMENTAL INDEX ONLY

The graph engine reads the links table and pages table; it does NOT rescan vault/. Coord
updates are column-level upserts on the pages table (x, y columns). This must be verified
by a pytest integration test: ingest one file → coords appear for affected nodes only.

### I3 — NO PER-TOKEN HEAVY WORK IN CHAT

No chat UI in v0.3. I3 is passively satisfied. Any Zustand store introduced for the
sigma viewer must use selectors + shallow equality — no unconditional global re-renders
on graph state changes.

### I5 — OBSIDIAN COMPATIBILITY

The graph coord columns (x, y on pages) and edges table are Postgres-only — no new fields
written to vault/ .md files. vault/wiki/ must remain a valid Obsidian vault. Verified by
the existing test_obsidian_check.py suite (15 tests); confirm it stays green.

### I6 — PLUGGABLE INFERENCE

No new inference calls are added by the graph engine. Graph computation is deterministic
(igraph FA2) — not LLM-driven. The InferenceProvider ABC is not touched this sprint.
The I6 static guard (T-CQ-009 scoped to import lines — NB-2 fix, see §7) must remain green.

### I7 — LOOPS ARE BOUNDED

The graph recompute is a SINGLE PASS (not a loop), but the debounce and the background
job queue must be bounded: max one queued recompute job at a time (subsequent dataVersion
bumps during an in-progress recompute collapse into one follow-up run). Log recompute
duration and node/edge count per run.

### I8 — DOCS-AS-DoD

D3 (graph recompute sequence), D5 (Playwright screenshots), D1 update, D2 update, D4 update
are all mandatory before M3 sign-off. The sprint cannot be declared done without them.

### I9 — DO NOT REINVENT

igraph is the specified graph-tool backend (CLAUDE.md §6 stack, R9). sigma.js is the
specified WebGL renderer (R10). TanStack Virtual for virtualized lists (R11). Do not
introduce other graph libraries (vis.js, d3-force, cytoscape). Playwright for D5 (R16).

## 5. Performance gates G1–G4

Per the QA charter, performance gates are mandatory from v0.3 onward. Each gate maps to a
concrete, Playwright-testable assertion.

### G1 — No full-rescan (I1 proof)

Definition: a file change updates only the affected Postgres records (pages, links, coords)
and Qdrant vectors. No full vault/ directory walk triggered by ingest.

Status in v0.3: G1 was proven green in v0.1/v0.2. The graph engine must not introduce a
new rescan path. Verify: after adding a single file, only that file's page row and related
link rows are updated; unrelated pages are not touched.

Test: `test_incremental_graph_update` — ingest file A (N nodes written), ingest file B,
assert only B's node coord is new/updated; A's coords are unchanged.

### G2 — Graph layout cached; no main-thread long task on second open (I2 proof)

Definition: opening the sigma viewer for the first time triggers one FA2 layout computation
(server-side). Opening it a second time (same dataVersion) serves precomputed coords with
no layout recompute.

Playwright test (mandatory for M3 sign-off):
1. Load the sigma viewer → wait for graph to render.
2. Capture Playwright performance trace: assert no JavaScript long task >50ms on the main
   thread during graph render (all heavy work is precomputed; sigma.js renders from coords).
3. Reload the viewer → assert `GET /graph` response includes a cache-hit header or the same
   coordinates are returned without a new FA2 run (verify via backend log or response header
   `X-Graph-Cache: hit`).
4. Specifically assert: no call to `graphology-layout-forceatlas2`, `sigma.ForceAtlas2`,
   or any `requestAnimationFrame`-based layout loop is present in the client-side JavaScript
   bundle (static bundle analysis or Playwright network check).

### G3 — Streaming; no per-token heavy parse (I3 proof)

Status: chat streaming is NOT in scope for v0.3 (F6 is v0.4). G3 is formally N/A this
sprint. It becomes mandatory from v0.4 onward when chat streaming is introduced.

Placeholder: QA must confirm at v0.4 that the Zustand store used in v0.3 for graph state
does not inadvertently introduce patterns that would violate I3 when chat state is added.
This is a pre-emptive design review task for v0.4, not a v0.3 gate.

### G4 — WebGL rendering at ≥60fps for large graphs; bounded DOM (I4 proof)

Definition: the sigma.js viewer renders a graph of at least 200 nodes and 500 edges at
≥60fps (16ms frame time) without adding unbounded DOM nodes.

Playwright test (mandatory for M3 sign-off):
1. Seed a fixture graph of 200 nodes / 500 edges in Postgres (test helper generates coords
   directly — no FA2 required for this perf fixture).
2. Load the sigma viewer. Assert the canvas element is present (sigma renders in a single
   WebGL canvas — NOT one DOM element per node).
3. Use Playwright's `page.evaluate` to read sigma's renderer FPS counter or measure
   `requestAnimationFrame` timing over 60 frames: assert mean frame time ≤16ms.
4. Assert DOM node count in the graph container does not exceed a fixed ceiling (e.g., <20
   DOM elements regardless of graph size — all nodes are WebGL drawn).

If the project's test fixture graph from v0.2 is <200 nodes, a synthetic fixture must be
generated. QA engineer owns this fixture.
