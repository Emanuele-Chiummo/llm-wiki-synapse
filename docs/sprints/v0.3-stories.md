## 1. F4 — GraphEngine: 4-signal edge weighting

### US-F4-ENGINE — 4-signal weighted graph construction

As the GraphEngine (backend/app/graph/engine.py), I want to compute edge weights between
wiki pages using four independent, additive signals — direct-link (×3), source-overlap (×4),
Adamic-Adar (×1.5), and type-affinity (×1) — so that the knowledge graph reflects the
strength of relationships between pages in a principled, auditable way.

**Invariants touched:** I1 (graph is built from Postgres tables, no vault/ filesystem rescan),
I9 (igraph is the specified library — R9).

**D-artifacts:** D2 (edges table added to schema), D3 (graph-recompute sequence touches
engine).

**Fixture specification for weight testing (QA must implement):**

Use a 5-node fixture graph loaded directly into the pages and links tables. Nodes:

| page_id | title | type | sources[] |
|---------|-------|------|-----------|
| P1 | "Alpha" | entity | ["doc_a.md"] |
| P2 | "Beta" | entity | ["doc_a.md"] |
| P3 | "Gamma" | concept | ["doc_b.md"] |
| P4 | "Delta" | entity | ["doc_a.md", "doc_b.md"] |
| P5 | "Epsilon" | concept | ["doc_c.md"] |

Links (wikilinks — rows in the links table):

| source_page_id | target_title | dangling |
|----------------|-------------|---------|
| P1 | "Beta" | False |
| P2 | "Alpha" | False |
| P3 | "Delta" | False |
| P4 | "Alpha" | False |

Expected edge weights (additive — all applicable signals sum per edge):

- Edge P1-P2 (bidirectional links + shared source "doc_a.md" + same type "entity"):
  - direct-link: P1→P2 exists (×3) + P2→P1 exists (×3) = 6
  - source-overlap: both have "doc_a.md" (×4) = 4
  - Adamic-Adar: common neighbours via links table — P4 links to P1, P3 links to P4 which
    links to... (computed by igraph AA on the directed/undirected adjacency; exact value
    depends on igraph implementation — see AQ-v0.3-2 below)
  - type-affinity: both "entity" (×1) = 1
  - Minimum expected weight (without AA contribution, AA adds >= 0): 6 + 4 + 1 = 11 + AA

The QA test must verify each signal's contribution INDEPENDENTLY by constructing edge pairs
that isolate each signal. Specifically:

- Edge P3-P5 (no link, no shared source, different type): no contribution from any signal.
  Expected weight = 0 (this edge should NOT appear in the edges table).
- Edge P1-P4 (P4→P1 link only, shared source "doc_a.md", same type "entity"):
  - direct-link: P4→P1 exists (×3) = 3
  - source-overlap: both have "doc_a.md" (×4) = 4
  - type-affinity: both "entity" (×1) = 1
  - Expected minimum weight: 8 + AA (AA from any shared neighbours)
- Edge P2-P4 (no direct link, shared source "doc_a.md", same type "entity"):
  - direct-link: 0
  - source-overlap: both have "doc_a.md" (×4) = 4
  - type-affinity: both "entity" (×1) = 1
  - Expected minimum weight: 5 (AA may add further)

DESIGN QUESTION for architect (AQ-v0.3-1): Are the 4 signals always SUMMED per edge
(additive model), or is any signal multiplicative / capped? The AC text above assumes
additive. If the architect changes this to a different combining formula, the QA fixture
expected values must be recomputed. Functional analyst flags this as AQ-v0.3-1.

**AC-F4-1**
Given a fixture graph with 5 nodes and the links/sources/types described in the fixture
specification above has been loaded into the test database,
when GraphEngine.build_weights() is called,
then:
(a) The edges table contains a row for P1-P2 with weight >= 11 (direct + source + type
    minimum contribution, before AA).
(b) The edges table contains a row for P1-P4 with weight >= 8.
(c) The edges table contains a row for P2-P4 with weight >= 5.
(d) The edges table does NOT contain a row for P3-P5 (or if it does, weight == 0).
(e) Each signal's contribution to at least one edge is independently verifiable via a
    pytest parametrize fixture that zeroes out all other signals and asserts non-zero
    contribution of the target signal.
Test target: `test_graph_engine.py` (new) — unit test with SQLite in-memory + fixture inserts.
EC reference: EC-M3-1.

## 5. F4 — Incremental graph update (G1 proof)

### US-F4-G1 — Incremental coord update preserves unaffected pages

As the GraphEngine, I want to update only the page coordinate rows affected by a new ingest
event, so that ingesting one file does not rewrite the entire pages table and the I1
incremental-index invariant is preserved for graph coordinates.

**Invariants touched:** I1 (incremental only — no full table rewrite on a single ingest).

**D-artifacts:** none directly, but the graph-recompute.mmd (D3) should annotate that
recompute writes coords to Postgres for all nodes in the igraph graph (which itself is
built from DB rows). The "incremental" guarantee at the coord level means coord rows are
not unnecessarily zeroed or rewritten for nodes unrelated to the triggering ingest.

DESIGN CLARIFICATION (for architect — AQ-v0.3-4): The graph recompute is a full FA2 run
over all current pages (the igraph graph includes all non-deleted pages). Coords for ALL
nodes are therefore rewritten on every recompute. The "incremental" guarantee of I1 at the
graph level means: (a) the recompute is DEBOUNCED (not triggered per-file-change), and
(b) new page rows for newly-ingested files are added incrementally without deleting and
recreating unrelated rows. The coord UPDATE (via FA2) touches all rows because FA2 is a
global layout, but it does so only once per debounce window — not per ingest event.
Functional analyst recommends the test assert that: after ingesting one new file, only one
new pages row is added (the new file's page); no other pages rows are deleted or recreated.
The coord columns (x, y) of existing pages may change after the debounced FA2 recompute —
this is acceptable because FA2 is a global layout algorithm. Please confirm this
interpretation via AQ-v0.3-4 before QA writes the test.

**AC-F4-9**
Given a vault with N pre-existing pages all having non-NULL x/y coordinates in Postgres,
when one new file is ingested via the watcher (adding one new page row),
then:
(a) Exactly one new row is added to the pages table (the new page); no existing page rows
    are deleted or have their id, title, type, sources changed.
(b) The new page row exists in the pages table with the correct file_path and title.
(c) The Postgres pages table row count increases by exactly 1.
Test target: `test_incremental_graph_update.py` (new) — seed N pages with x/y; trigger one
ingest; assert table count N+1; assert all N pre-existing page ids still present with
unchanged id/title/type.
EC reference: EC-M3-8, G1.
