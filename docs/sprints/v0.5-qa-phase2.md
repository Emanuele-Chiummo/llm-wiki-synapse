## 3. I7 — Bounded Loops Gate (HEADLINE RISK — each bound individually verified)

### 3.1 Structural loop cap: `for range`, not `while True`

**Finding:** `backend/app/ops/deep_research.py` line 187:
```python
for iteration in range(1, max_iter_local + 1):  # ← HARD CAP (ADR-0024 §3.2)
```
No `while True`, no `while condition`, no dynamic upper bound. The cap is structural.

**Test:** T-DR-002 — `assess_calls[0] == 3` exactly when `max_iter=3`. PASS.

### 3.2 Budget gate fires at TOP of round BEFORE spending

**Finding:** The budget check `if accumulator.total_tokens >= token_budget_local: status = "budget_exhausted"; break` is the first statement inside the `for` body, before any search/fetch/assess call.

**Test:** T-DR-005 — pre-fills `accumulator.input_tokens = 100` exceeding `token_budget=10`; asserts `status == "budget_exhausted"` and `assess_calls[0] == 0`. PASS.

### 3.3 Concurrency never exceeds 3

**Finding:** A single `asyncio.Semaphore(CONCURRENCY=3)` is defined at module level in `ops/searxng.py`. It is imported by `deep_research.py` (not re-instantiated). Both `searxng_search_many` and `_fetch_and_extract` acquire this same semaphore. There is no second semaphore, no unbounded `gather` without acquisition.

**Test:** T-DR-007 — `assert CONCURRENCY == 3` and `assert _semaphore._value == CONCURRENCY`. PASS.

Note: T-DR-007 is a structural/constant assertion, which is the appropriate unit-test form for a semaphore bound. Live concurrent execution would require a multi-threaded integration test; the structural test is sufficient to prove Do-NOT #4 compliance.

### 3.4 Status never stuck "running" — terminal write in `finally`

**Finding:** The `_finalize_run_row` call is inside a `finally` block. `status` defaults to `"max_iter_reached"` (pessimistic) and is overwritten only on real exits. The `except Exception` block sets `status = "error"`.

**Test:** T-DR-010 — `_exploding_generate` raises; asserts `result.status == "error"` and `finalize_calls[0]["status"] == "error"` and `finalize_calls[0]["status"] != "running"`. PASS.
T-DR-003 — after a converged run, `result.status in {"converged", ...}` and `!= "running"`. PASS.

### 3.5 Sufficiency assessed BEFORE refine

**Finding:** `_assess_sufficiency` is called inside the loop body before the `if iteration < max_iter_local` refine branch. This is structurally enforced — there is no code path that calls `_generate_queries` (refine) in the same iteration without first calling `_assess_sufficiency`.

**Test:** T-DR-008 — order tracking confirms every `generate-N` (N > 0) is preceded by `assess-N`. PASS.

### 3.6 Bounds frozen at run start, never re-read mid-loop

**Finding:** Lines 175-176 in `deep_research.py` read `max_iter_local` and `token_budget_local` from `run_obj` (the DB row snapshot), not from settings or environment. The loop body uses only `max_iter_local` and `token_budget_local`. The module accessors `_default_max_iter()` / `_default_token_budget()` are called only before the run row INSERT, not inside the loop.

**Test:** Covered by T-DR-002/005 structure — the run mock returns fixed values that are never re-read from settings mid-loop.

### 3.7 total_cost_usd accumulated + logged + $1 anomaly WARNING

**Finding:** `UsageAccumulator` is bound to the provider. On exit (finally), `total_cost_usd = round(accumulator.total_cost_usd, 4)` is computed and written to the run row. A `logger.warning(...)` fires if `total_cost_usd > 1.00`.

**Test:** Cost accumulation path is exercised structurally in T-DR-010 (provider bound, accumulator created). The $1 threshold constant `COST_ANOMALY_THRESHOLD_USD = 1.00` is module-level.

**Gap noted:** There is no dedicated test asserting the `logger.warning` fires when `total_cost_usd > 1.00`. This is a minor gap (the cost anomaly is a logging concern, not a safety bound). The constant and the code path are present. Adding a dedicated test for this would be belt-and-suspenders. Logged as a non-blocking note.

## 6. Frontend I3/I4 — Polling and virtualization

### I3 — Polling is setTimeout chain, not setInterval, stops on terminal status

**Finding in `researchStore.ts`:**
- `startPollingDetail` uses `setTimeout(() => void tick(), POLL_INTERVAL_MS)` inside an async `tick()` function. The next `setTimeout` is only scheduled at the end of `tick()` if `!isTerminal(updated.status)`. On terminal status, `setTimeout` is never re-scheduled.
- Cleanup: returns a closure that calls `ctrl.abort()`. The component calls this cleanup on unmount and on terminal transition (`stopPollRef.current()` in `useEffect`).
- No `setInterval`, no `requestAnimationFrame`, no tight loop.

**Frontend tests:**
- `researchStore.test.ts` — "stops polling after cleanup is called" (T-FE-STORE-poll-1): `stop()` immediately → timer advances 6s → fetch not called. PASS.
- `researchStore.test.ts` — "stops polling when status transitions to terminal": first tick returns `"converged"` → second 5s interval passes → `callCount` still 1. PASS.

### I4 — Runs list virtualized with TanStack Virtual

**Finding in `DeepSearchView.tsx`:** `ResearchRunList` uses `useVirtualizer` from `@tanstack/react-virtual` for all run cards.

**Frontend tests:** `DeepSearchView.test.tsx` — "renders the run list container when runs are present" verifies the virtualizer container exists. Note: jsdom has no layout engine so virtual rows themselves are not mounted in unit tests. The E2E Playwright test (AC-F10-8) is the complete end-to-end gate.

**I3/I4 verdict: PASS** (structural + unit test coverage; E2E deferred to live Playwright run).

## 10. Non-blocking notes

1. **$1 cost anomaly WARNING test gap:** No dedicated unit test asserts that `logger.warning(...)` fires when `total_cost_usd > 1.00`. The constant and code path are present and reviewed. Non-blocking because the anomaly is a logging/observability feature, not a safety bound — it cannot cause runaway execution.

2. **T-DR-007 concurrency test is structural:** Asserts `_semaphore._value == 3` (reads the Python `asyncio.Semaphore` internal `_value` attribute). This proves the constant is correctly applied but does not prove no concurrent execution exceeds 3 in a live async test. A live integration test would require N goroutines and a counter. Acceptable for this phase; the structural proof plus Do-NOT #4 code review is sufficient.

3. **AC-F10-8 Playwright E2E:** `frontend/e2e/deep-search.spec.ts` is noted in the AC as the E2E gate. This requires a live backend+frontend. It is deferred to live-demo (same pattern as all other Playwright E2E tests in this project). The unit/component tests above cover the UI logic.
