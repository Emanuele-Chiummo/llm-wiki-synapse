## How to read this document

Each user story block contains:
- Story ID (e.g. S-F5-1): referenced in TRACEABILITY.md.
- Story header: "As a [role], I want [capability], so that [outcome]."
- Numbered acceptance criteria (AC-F5-1, AC-F5-2, …): each is a concrete binary assertion.
- Invariants touched: priming engineers on which non-negotiables are exercised.
- D-artifacts impacted.
- Planned test target(s): forward reference for qa-test-engineer.

Binary pass/fail interpretation: every AC is either fully met or not met. No partial credit.

## Phase 1 — Retrieval foundation + carry-forward resolution

### S-F5-1 — 4-phase RAG retrieval pipeline

As a chat user, I want my questions answered using context retrieved from the wiki via a
4-phase pipeline (tokenized keyword search → graph-expansion → token-budget allocation →
context assembly), so that assistant answers are grounded in the actual vault content rather
than the model's parametric knowledge alone.

**Invariants touched:** I1 (no vault rescan; reads from Qdrant + Postgres only), I2 (graph-expansion
reads from the existing `edges` table — does NOT trigger FA2 recompute), I3 (retrieval
assembled server-side; no per-token heavy work), I9 (reuse Qdrant + bge-m3; no new
embedding service).

**D-artifacts:** D4 (`GET /search` endpoint in openapi.json).

**AC-F5-1**
Given `backend/app/rag/retrieval.py` is implemented,
when a retrieval call is made,
then the pipeline executes all four phases in order: (1) tokenized keyword search over
Qdrant using bge-m3 vectors; (2) graph-expansion following edges from top-k results in
the Postgres `links` and `edges` tables, up to depth 2; (3) token-budget allocation using
20% of the configured context window (per F14); (4) context assembly producing a
`RetrievalContext` Pydantic object with page refs.
Test target: `backend/tests/test_retrieval.py` — unit test with mocked Qdrant client and
Postgres edges; assert all 4 phases execute and `RetrievalContext` is returned.

**AC-F5-2**
Given the pipeline has assembled a `RetrievalContext`,
when the context is inspected,
then every retrieved passage carries a 1-based citation index `n`; the assembled context
string contains inline `[n]` markers; and `RetrievalContext` maps each `n` to a `PageRef`
with fields `id` (uuid), `title` (str), and `slug` (url-friendly str, not null).
Test target: `backend/tests/test_retrieval.py` — assert `len(context.citations) == n_passages`,
each citation has `id`, `title`, `slug`; assert `[1]` appears in `context.text`.

**AC-F5-3**
Given the retrieval pipeline is executing,
when it performs vector search and graph-expansion,
then it calls only the existing Qdrant wrapper (bge-m3) and reads from Postgres `links`/`edges`
tables — no import of a new embedding service, no new Qdrant collection, no direct Ollama
embedding call.
Test target: `backend/tests/test_code_quality.py` — static import check: no `bge`, `sentence_transformers`,
or new Qdrant collection creation outside the existing Qdrant wrapper module.

**AC-F5-4**
Given 10 fixture passages totalling 2× the configured token budget,
when the context assembly phase runs,
then the assembled `RetrievalContext.text` length (in tokens) is less than or equal to the
budget; the lowest-ranked passages are dropped first; no passage is silently truncated
mid-sentence without removal.
Test target: `backend/tests/test_retrieval.py` — seed 10 fixture passages; configure budget
to half their total; assert `len(context.citations) < 10` and total token length ≤ budget.

**AC-F5-5**
Given a retrieval call has completed,
when the Postgres `vault_state.data_version` is read before and after the call,
then the value is identical (graph-expansion reads the `edges` table; it does NOT bump
`data_version` or trigger FA2).
Test target: `backend/tests/test_retrieval.py` — assert `data_version_before == data_version_after`.

**AC-F5-6**
Given the FastAPI application is running,
when `GET /search?q=<query>&vault_id=<uuid>` is called,
then the endpoint returns HTTP 200 with a JSON body containing at least `results` (array of
citation objects) and `context` (assembled string with `[n]` markers); the endpoint schema
is present in `docs/api/openapi.json`.
Test target: `backend/tests/test_api.py` — ASGI test client; assert HTTP 200 and required
fields. `backend/tests/test_docs.py` — assert openapi.json contains `"/search"` path.

**AC-F5-7**
Given the retrieval unit test suite,
when all unit tests in `test_retrieval.py` run,
then the following scenarios all pass: (a) 0-result query returns empty `RetrievalContext`
with no citations; (b) single-result query returns `RetrievalContext` with exactly 1
citation; (c) multi-page expansion returns linked pages in the expansion phase (seed 5
pages with known wikilinks; assert linked pages appear in results); (d) budget-overflow
truncation drops lowest-ranked passages until budget is satisfied.
Test target: `backend/tests/test_retrieval.py` — 4 parametrized test cases.

**AC-F5-8**
Given F5 retrieval is implemented,
when the chat endpoint processes a user message,
then the assembled `RetrievalContext` from F5 is passed to the active provider's
`chat(messages, retrieval_context)` call for all three backends (OllamaProvider,
ApiProvider, CliAgentProvider), so that `[n]` markers appear in assistant messages.
Test target: `backend/tests/test_chat_endpoint.py` — mock all three providers; assert each
`chat()` call receives a `RetrievalContext` argument (not None).

## Phase 2 — Deep Research loop

### S-F10-1 — Deep Research loop: bounded multi-query SearXNG loop

As a researcher, I want Synapse to autonomously gather web sources on a topic by running
multiple SearXNG queries, assessing whether collected content is sufficient, and synthesizing
a new wiki page — all within explicit iteration and cost bounds, so that I can augment the
vault with current information without manual web searching.

**Invariants touched:** I7 (max_iter + token_budget + concurrency ≤ 3 are mandatory; any
unbounded path is a P0 blocker), I9 (SearXNG is the only permitted web search backend;
never Tavily, DuckDuckGo, or Google), I6 (all inference calls go through InferenceProvider
— query generation, sufficiency assessment, synthesis), I1 (synthesis fed through the
ingest pipeline, not written directly to vault/wiki/).

**D-artifacts:** D2 (`deep_research_runs` + `deep_research_sources` tables in schema.mmd),
D3 (`docs/sequences/deep-research.mmd`), D4 (research endpoints in openapi.json).

**AC-F10-1**
Given `backend/app/ops/deep_research.py` is implemented,
when `run_deep_research(vault_id, topic, max_iter, token_budget)` is called,
then the function executes all 6 steps in order: (1) generate 2–5 SearXNG queries via
InferenceProvider; (2) execute queries against SearXNG with concurrency=3 (asyncio semaphore);
(3) fetch and parse result pages to markdown; (4) call InferenceProvider to assess sufficiency;
if sufficient → step 5; if not and iterations < max_iter → generate follow-up queries and
repeat from step 2; (5) synthesize via InferenceProvider; (6) feed synthesis into the ingest
pipeline (not directly into vault/wiki/).
Test target: `backend/tests/test_deep_research.py` — mock SearXNG and InferenceProvider;
assert all 6 steps execute; assert synthesis routed to ingest pipeline.

**AC-F10-2**
(I7 — headline invariant for M5)
Given `deep_research.py` is running with a mock InferenceProvider that always returns "insufficient",
when the loop executes,
then: (a) the loop terminates after exactly `max_iter` refinement rounds; (b) the run status
is set to `"max_iter_reached"` (not still `"running"`); (c) no further InferenceProvider
calls are made after `max_iter` is reached; (d) `max_queries_per_iter` is not exceeded per
round (default 5); (e) `concurrency` is at most 3 (asyncio semaphore prevents more than 3
concurrent SearXNG fetches).
Test target: `backend/tests/test_deep_research.py` — mock "always insufficient" provider;
`assert call_count == max_iter`; `assert run.status == "max_iter_reached"`.
THIS TEST IS MANDATORY — EC-M5-5 explicitly requires it.

**AC-F10-3**
(I9)
Given the deep research implementation and all its dependencies,
when `python -m pytest backend/tests/test_code_quality.py -k "no_tavily"` runs,
then: (a) no file in the codebase imports `tavily`, `duckduckgo_search`, `googlesearch`,
or `google-search-results`; (b) `SEARXNG_URL` env var is the only web-search configuration
entry point.
Test target: `backend/tests/test_code_quality.py` — static import scan; assert zero matches
for `tavily|ddg|duckduckgo|googlesearch` in all `.py` files.

**AC-F10-4**
