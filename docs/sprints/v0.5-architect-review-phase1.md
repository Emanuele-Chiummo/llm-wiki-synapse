## Accepted-limitation honesty check (ADR-0022 §4)

`[n]` resolves to the **source-document `pages` row**, not the synthesized `wiki/` page. Preserved
honestly and **not silently worsened**:
- `_load_page_meta` reads `pages.{id,title,file_path}` (L544–558); `file_path` is the
  `raw/sources/...` path, and `_load_passage` reads that source body. The citation therefore points
  at the indexed source — exactly the §4 boundary, no new degradation.
- `title` falls back to the filename stem so it is never empty (L555–557, §2.6).
- The frontend renders the title on hover and navigates by slug (`decorateCitations.ts` L100,
  `MarkdownView.tsx` L89–103) — it surfaces whatever the backend cites; it does not paper over the
  source-vs-wiki distinction. The limitation remains visible and deferred post-M5, as designed.

## Invariant-tension scan (beyond the Do-NOT list)

No new tension introduced. Two points worth recording for the gate:

1. **Retrieval failure is non-fatal to chat (stream.py L184–197).** `retrieve()` errors are caught
   and chat proceeds with empty retrieval context (no citations). This is a sound resilience choice
   and does not violate any invariant — but it means a silent retrieval regression would degrade
   grounding without failing the turn. The WARNING log (L193) is the only signal. Acceptable for
   Phase 1; flag for QA to assert the log path and for a future health metric. **Not a condition.**

2. **`_expand_frontier` IN-list interpolation (retrieval.py L314–328, L348–351, L539–547).** Bind
   placeholder NAMES (`:f0,:f1,…`) are f-string-interpolated, but every id value travels as a bound
   parameter — never as SQL text. `# noqa: S608` is correctly scoped and the justification is
   documented inline. Not an injection vector. **HELD, no action.**

Both are consistent with the ADR's "simple and correct over clever and fragile" stance.

## Conditions (documentation-only — do NOT block the phase gate)

**C1 — Amend ADR-0022 for `query_points` (REQUIRED, doc drift / I8).**
ADR-0022 §2.2 phase 1 and §3 (AQ-v0.5-1) literally reference
`qdrant.search(synapse_pages, vector, limit=k, with_payload=True)`. The implementation uses
`client.query_points(collection_name=…, query=vector, limit=k, with_payload=True)` and reads
`response.points` (`retrieval.py` L232–246). **This is semantically equivalent and the correct
call**: qdrant-client removed `search()` (deprecated, dropped by 1.18); `query_points` is the
current dense top-k API. Against the existing single-dense `synapse_pages` collection it performs
the **identical cosine top-k** — same collection, same vector, same `ScoredPoint(id, score,
payload)` results, same `id == pages.id` mapping (ADR-0002). No collection-shape change, no sparse
vector. **Acceptable and approved.** ADR-0022's code reference should be updated from `.search()`
to `.query_points()` (with a one-line note that `.search()` was removed in client 1.18) so the
frozen contract matches the shipped code. Hand to tech-writer.

**C2 — Pin the qdrant-client floor (RECOMMENDED, reproducibility).**
`backend/pyproject.toml` declares `qdrant-client>=1.9.0`, but `query_points` returning a
`QueryResponse.points` list is the modern shape; `1.9` is the lower bound that introduced it and
older patch behaviour differs. Since the code now depends on `query_points`/`.points`, devops
should confirm the lockfile resolves to a `query_points`-bearing release (≥1.10 is safe) and,
ideally, bump the floor so a fresh environment cannot resolve a `search()`-only client. Non-blocking
for this gate (CI installs a current client), but record it. Hand to devops-engineer.

No code conditions. Nothing in the implementation needs to change to pass the gate.
