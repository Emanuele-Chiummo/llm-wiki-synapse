## Two defects found in live testing and fixed

1. **Local (Ollama) provider could not be configured** (commit `3e6bcaf`). The
   Settings → LLM Models "Add provider" form hid the `base_url` field for
   `provider_type=local`, so the Ollama server URL was stored as NULL and
   `OllamaProvider` raised at runtime unless `OLLAMA_URL` env was set. The form now
   shows `base_url` for both `local` and `api`, with type-specific placeholders
   (`qwen2.5:3b`, `http://localhost:11434`). This is what blocked the developer from
   pointing at his installed `qwen2.5:3b` model.

2. **Embeddings section was a dead "coming in M5" stub** (commit `3e6bcaf`). Replaced
   with a live read-only display of `EMBEDDING_URL` / `EMBEDDING_MODEL` / `EMBEDDING_DIM`
   via a new `GET /config/embedding` endpoint, so the developer can confirm `bge-m3` is
   wired without opening `.env`.

Both slipped the sandbox gate because neither is verifiable without a human driving the
provider form against a live Ollama — exactly the class of defect the human checkpoint
exists to catch.

## Dev-environment improvement (not an M4 defect)

To make the human checkpoint repeatable on macOS, `docker-compose.dev.yml` was added
(commits `196cf99`, `e4f3d1a`): a local-dev override that adds Qdrant + the Vite dev
server as containers so the whole stack starts with one command. Ollama stays on the host.
Production `docker-compose.yml` is unchanged (I9 respected).
