# Synapse v1.3.9 — Cloudflare Access Gate & Guided Settings

_Released: 2026-07-07_

Places the entire service behind Cloudflare Access authentication (ADR-0062), reorganizes Settings into five intent-based groups for clarity, and fixes a 502 error when the CLI inference provider attempted to create pages from the Review queue or via Lint generation. Backend rebuild required; no DB migration.

## Added

- **Settings reorganized into 5 intent groups** (Essentials · Content & Sources · AI Behaviour · Access & Security · System), each with a descriptive tagline; advanced items are badged; default landing page is "AI & provider" [F16].
- **CLI provider setup unified**: the subscription token is now configured on the same Settings page as the provider selector with an inline guide, replacing a separate tab [F17].

## Fixed

- **502 on "Create" in the Review queue and Lint generation with the CLI provider** — the on-demand generation path forced the orchestrated loop (`analyze()`), which is invalid for agentic providers. Routing is now capability-aware: `supports_agentic_loop=True` → delegated path; otherwise → orchestrated loop (I6) [F9][F17].

## Security

- **Cloudflare Access gate** (ADR-0062) — closes public vault exposure. Browser clients authenticate via the interactive login (cookie/OTP); non-browser clients (desktop, iOS, Chrome clipper, MCP) pass the gate with a CF Service Token (`CF-Access-Client-Id` / `CF-Access-Client-Secret`) configurable in Settings [F15][F16].
- **Security response headers** (nginx) — HSTS, `X-Content-Type-Options`, `X-Frame-Options: DENY`, CSP `frame-ancestors 'none'`, `Referrer-Policy`, `Permissions-Policy`.
- **PWA manifest behind Access** — `crossorigin="use-credentials"` on the manifest link prevents it from being diverted to the Cloudflare login page (CORS error).

## Upgrade notes

- Backend rebuild required (the 502 fix is backend-side): re-pull `ghcr.io/emanuele-chiummo/llm-wiki-synapse-backend:v1.3.9` and the frontend image. No DB migration.
- One-time Cloudflare setup: create a Service Token (Zero Trust → Access → Service Auth) with a "Service Auth" policy and paste it into the respective client settings; enable "Always Use HTTPS". For the MCP endpoint (Bearer-token only), configure an Access Bypass for `/mcp/server` — see `docs/DEPLOY.md §5.6b`.
