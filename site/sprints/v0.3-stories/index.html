    Sprint 3 — v0.3 — User Stories + Acceptance Criteria¶

    Milestone: M3 — "Graph live, no main-thread freeze" Authored by: functional-analyst Date: 2026-06-28 Source of truth for AC IDs: BACKLOG.md §Sprint 3 Exit criteria reference: docs/sprints/v0.3-scope.md §6 (EC-M3-1..17) Invariants with heightened priority: I2 (server-side cached layout — HEADLINE), I4 (WebGL + virtualization) All 9 invariants apply.

    How to read this document¶

    Each user story block contains: - Story header: "As a [role], I want [capability], so that [outcome]." - Given/When/Then acceptance criteria, each mapped to a BACKLOG.md AC ID. - Invariants touched — priming engineers on which non-negotiables are exercised. - D-artifacts impacted. - Planned test target(s) — forward reference for qa-test-engineer.

    Binary pass/fail interpretation: every AC is either fully met or not met. No partial credit. Criteria phrasing such as "correct" or "valid" is resolved by the exact assertions listed in the Given/When/Then body.

    1. F4 — GraphEngine: 4-signal edge weighting¶

    US-F4-ENGINE — 4-signal weighted graph construction¶

    As the GraphEngine (backend/app/graph/engine.py), I want to compute edge weights between wiki pages using four independent, additive signals — direct-link (×3), source-overlap (×4), Adamic-Adar (×1.5), and type-affinity (×1) — so that the knowledge graph reflects the strength of relationships between pages in a principled, auditable way.

    Invariants touched: I1 (graph is built from Postgres tables, no vault/ filesystem rescan), I9 (igraph is the specified library — R9).

    D-artifacts: D2 (edges table added to schema), D3 (graph-recompute sequence touches engine).

    Fixture specification for weight testing (QA must implement):

    Use a 5-node fixture graph loaded directly into the pages and links tables. Nodes:

    page_id title type sources[]
    P1 "Alpha" entity ["doc_a.md"]
    P2 "Beta" entity ["doc_a.md"]
    P3 "Gamma" concept ["doc_b.md"]
    P4 "Delta" entity ["doc_a.md", "doc_b.md"]
    P5 "Epsilon" concept ["doc_c.md"]

    Links (wikilinks — rows in the links table):

    source_page_id target_title dangling
    P1 "Beta" False
    P2 "Alpha" False
    P3 "Delta" False
    P4 "Alpha" False

    Expected edge weights (additive — all applicable signals sum per edge):

    • Edge P1-P2 (bidirectional links + shared source "doc_a.md" + same type "entity"):
    • direct-link: P1→P2 exists (×3) + P2→P1 exists (×3) = 6
    • source-overlap: both have "doc_a.md" (×4) = 4
    • Adamic-Adar: common neighbours via links table — P4 links to P1, P3 links to P4 which links to... (computed by igraph AA on the directed/undirected adjacency; exact value depends on igraph implementation — see AQ-v0.3-2 below)
    • type-affinity: both "entity" (×1) = 1
    • Minimum expected weight (without AA contribution, AA adds >= 0): 6 + 4 + 1 = 11 + AA

    The QA test must verify each signal's contribution INDEPENDENTLY by constructing edge pairs that isolate each signal. Specifically:

    • Edge P3-P5 (no link, no shared source, different type): no contribution from any signal. Expected weight = 0 (this edge should NOT appear in the edges table).
    • Edge P1-P4 (P4→P1 link only, shared source "doc_a.md", same type "entity"):
    • direct-link: P4→P1 exists (×3) = 3
    • source-overlap: both have "doc_a.md" (×4) = 4
    • type-affinity: both "entity" (×1) = 1
    • Expected minimum weight: 8 + AA (AA from any shared neighbours)
    • Edge P2-P4 (no direct link, shared source "doc_a.md", same type "entity"):
    • direct-link: 0
    • source-overlap: both have "doc_a.md" (×4) = 4
    • type-affinity: both "entity" (×1) = 1
    • Expected minimum weight: 5 (AA may add further)

    DESIGN QUESTION for architect (AQ-v0.3-1): Are the 4 signals always SUMMED per edge (additive model), or is any signal multiplicative / capped? The AC text above assumes additive. If the architect changes this to a different combining formula, the QA fixture expected values must be recomputed. Functional analyst flags this as AQ-v0.3-1.

    AC-F4-1 Given a fixture graph with 5 nodes and the links/sources/types described in the fixture specification above has been loaded into the test database, when GraphEngine.build_weights() is called, then: (a) The edges table contains a row for P1-P2 with weight >= 11 (direct + source + type minimum contribution, before AA). (b) The edges table contains a row for P1-P4 with weight >= 8. (c) The edges table contains a row for P2-P4 with weight >= 5. (d) The edges table does NOT contain a row for P3-P5 (or if it does, weight == 0). (e) Each signal's contribution to at least one edge is independently verifiable via a pytest parametrize fixture that zeroes out all other signals and asserts non-zero contribution of the target signal. Test target: test_graph_engine.py (new) — unit test with SQLite in-memory + fixture inserts. EC reference: EC-M3-1.

    2. F4 — GraphEngine: FA2 server-side layout¶

    US-F4-LAYOUT — Server-side FA2 layout via igraph¶

    As the GraphEngine, I want to run the ForceAtlas2 layout algorithm using python-igraph on the server, writing x and y float coordinates for every page to Postgres, so that the client can render the graph from precomputed positions without ever executing a layout algorithm in the browser (satisfying invariant I2 completely).

    Invariants touched: I2 (FA2 runs server-side, coords in Postgres, NEVER in browser), I1 (coord update is incremental — only affected page rows updated), I9 (igraph R9 is the specified tool, not graph-tool, not d3-force).

    D-artifacts: D2 (pages table gains x, y float columns; edges table is new — must appear in schema.mmd after make er), D3 (graph-recompute.mmd shows FA2 call).

    AC-F4-2 Given GraphEngine.recompute() has been called on a seeded test graph, when the Postgres pages table is queried, then every page that was in the igraph Graph object has a non-NULL x (float) and y (float) value in the pages table; no page has both x IS NULL and y IS NULL after a successful recompute. Test target: test_graph_engine.py — call engine.recompute() with 5-node fixture; assert all 5 pages have non-NULL x/y via SQLAlchemy query. EC reference: EC-M3-2.

    AC-F4-2b (I2 no-client-layout static check) Given the frontend bundle has been built via make build, when the bundle output files (dist/) are scanned for forbidden strings, then NO file in the bundle contains any of the following substrings: graphology-layout-forceatlas2, ForceAtlas2, d3-force, forceSimulation, requestAnimationFrame when followed within 100 characters by layout or force. Test target: test_graph_engine.py or frontend/tests/bundle-check.test.ts — grep/scan the Vite build output directory. EC reference: EC-M3-5 (G2 static bundle check).

    AC-F4-2c (igraph version pin and no alternative library) Given the backend requirements are installed, when the module backend/app/graph/engine.py is imported, then the import succeeds using igraph (python-igraph); the module does NOT import graph_tool, networkx, vis, cytoscape, or d3. Test target: test_graph_engine.py — assert import igraph succeeds; assert that engine.py source text does not contain the string "import graph_tool" or "import networkx" (static check). EC reference: EC-M3-2, I9.

    DESIGN QUESTION for architect (AQ-v0.3-2): FA2 in igraph requires a seed for deterministic coordinate output. Should the engine always seed FA2 with a fixed integer (e.g., seed=42) so that two calls with identical graph topology produce identical coordinates (enabling byte-stable regression tests)? Or is non-determinism acceptable (test only checks non-NULL, not specific values)? Functional analyst flags this as AQ-v0.3-2. Impact: if seed is fixed, add seed=42 parameter to igraph FA2 call and add a determinism assertion to the test. If non-deterministic, remove the determinism test and accept only non-NULL + range checks.

    3. F4 — Graph Cache: dataVersion-debounced recompute¶

    US-F4-CACHE — Debounced FA2 recompute on dataVersion change¶

    As the GraphCache (backend/app/graph/cache.py), I want to monitor vault_state.data_version and trigger exactly one FA2 recompute after a configurable debounce window when the version changes, collapsing bursts of rapid ingests into a single recompute run, so that the graph coordinates remain accurate without wasting compute on intermediate states and without any unbounded job queue (satisfying I2's "debounced on dataVersion" clause and I7's bounded-job requirement).

    Invariants touched: I2 (cached + debounced), I7 (bounded queue — max 1 queued job), I1 (recompute reads DB tables, not filesystem).

    D-artifacts: D3 (graph-recompute.mmd shows debounce collapse and cache-hit/miss paths).

    AC-F16db-1 Given vault_state.data_version is at version N, when data_version is updated to N+1 (simulated by direct DB write in test), then GraphCache detects the change (via polling or event) and, after the debounce window elapses, exactly one call to GraphEngine.recompute() is recorded. Test target: test_graph_cache.py (new) — mock GraphEngine.recompute(); mock clock; bump data_version once; advance clock past debounce window; assert recompute called exactly 1 time. EC reference: EC-M3-3.

    AC-F16db-2 Given GraphCache is monitoring data_version, when data_version is bumped 5 times in rapid succession within a single debounce window, then exactly 1 call to GraphEngine.recompute() is made (not 5), verified by a mock call counter. Test target: test_graph_cache.py — mock clock; bump version 5 times within debounce window; advance clock past window end; assert recompute called == 1. EC reference: EC-M3-3, I2 (debounce).

    AC-F16db-3 Given GraphEngine.recompute() is currently in-flight (simulated by a blocking mock), when data_version is bumped again during the in-flight recompute, then the in-flight recompute is NOT interrupted; after it completes, exactly 1 additional recompute is triggered (the deferred bump). Total calls: 2 (first + one deferred), not N. Test target: test_graph_cache.py — use asyncio or threading mock; assert the in-flight recompute completes before the second starts; assert exactly 2 total calls on the mock. EC reference: I7 (bounded queue), I2.

    AC-F16db-4 Given GraphCache is running, when GET /status is called, then the response JSON includes data_version equal to the current vault_state.data_version value (no regression from v0.1 AC-F16dv-3). Test target: test_api.py — extend existing T-API-004 to assert data_version still present after v0.3 changes. EC reference: v0.1 regression gate.

    4. F4 — GET /graph REST endpoint¶

    US-F4-API — GET /graph returns precomputed graph payload¶

    As the vault owner (Emanuele) querying the Synapse REST API, I want GET /graph to return a typed JSON payload containing precomputed node coordinates, weighted edges, the current data_version, and a cache-hit indicator, so that the frontend can render the graph immediately from server-computed data without triggering any layout computation.

    Invariants touched: I2 (endpoint serves precomputed coords, emits cache header), I8 (D4 OpenAPI must document this endpoint and its response schema).

    D-artifacts: D4 (openapi.json updated with GET /graph response schema).

    Response schema (to be locked by architect — AQ-v0.3-3): The GET /graph response must conform to: 

    {
  "nodes": [
    {"id": "<uuid>", "title": "<string>", "type": "<string>", "x": <float>, "y": <float>}
  ],
  "edges": [
    {"source": "<uuid>", "target": "<uuid>", "weight": <float>}
  ],
  "data_version": <int>,
  "cached": <bool>
}
    The X-Graph-Cache response header is set to "hit" when the response is served from cached coordinates (no FA2 recompute triggered) and "miss" when FA2 was triggered.

    DESIGN QUESTION for architect (AQ-v0.3-3): Should GET /graph block until FA2 is complete (synchronous), or return HTTP 202 with a job_id when FA2 is not yet done and coords are stale? Impact on test strategy is significant: if synchronous, test can assert full payload; if async, test must poll. Functional analyst recommends synchronous for v0.3 simplicity (graph is fast on igraph for the expected vault sizes), but flags this as AQ-v0.3-3 for architect decision before engineering begins.

    AC-F4-3 Given a seeded graph with at least 3 nodes and 2 edges in Postgres with x/y coords set, when GET /graph is called, then: (a) HTTP 200 is returned. (b) Response body is valid JSON with keys: nodes, edges, data_version, cached. (c) nodes is a non-empty array; each node object has keys: id, title, type, x, y with correct types (string, string, string, float, float). (d) edges is a non-empty array; each edge object has keys: source, target, weight with correct types (string, string, float). (e) data_version is an integer >= 1. (f) cached is a boolean. Test target: test_api.py (extend) + test_graph_api.py (new) — ASGI integration test with pre-seeded pages/edges rows. EC reference: EC-M3-4.

    AC-F4-4 Given GET /graph has been called once (cache-miss, FA2 run logged), when GET /graph is called a second time without any data_version change, then: (a) Response cached field is true. (b) X-Graph-Cache response header equals "hit". (c) Backend logs do NOT contain a second FA2 invocation (verified by asserting the mock GraphEngine.recompute() call count remains 1 after the second GET /graph call). Test target: test_graph_api.py — use mock GraphEngine; call GET /graph twice; assert recompute called exactly once; assert second response has cached=True and header X-Graph-Cache=hit. EC reference: EC-M3-3 (cache-hit verification).

    AC-D4v3-1 Given GET /graph has been added to FastAPI and make openapi has been run, when docs/api/openapi.json is inspected, then the file contains a path entry for /graph with a GET operation that documents: - 200 response schema matching the structure above (nodes array, edges array, data_version int, cached bool). - All node fields (id, title, type, x, y) typed in the schema. - All edge fields (source, target, weight) typed in the schema. Test target: test_docs.py (extend) — assert openapi.json contains /graph path with correct response schema; run via pytest or CI make openapi + parse check. EC reference: EC-M3-4, EC-M3-12, I8.

    5. F4 — Incremental graph update (G1 proof)¶

    US-F4-G1 — Incremental coord update preserves unaffected pages¶

    As the GraphEngine, I want to update only the page coordinate rows affected by a new ingest event, so that ingesting one file does not rewrite the entire pages table and the I1 incremental-index invariant is preserved for graph coordinates.

    Invariants touched: I1 (incremental only — no full table rewrite on a single ingest).

    D-artifacts: none directly, but the graph-recompute.mmd (D3) should annotate that recompute writes coords to Postgres for all nodes in the igraph graph (which itself is built from DB rows). The "incremental" guarantee at the coord level means coord rows are not unnecessarily zeroed or rewritten for nodes unrelated to the triggering ingest.

    DESIGN CLARIFICATION (for architect — AQ-v0.3-4): The graph recompute is a full FA2 run over all current pages (the igraph graph includes all non-deleted pages). Coords for ALL nodes are therefore rewritten on every recompute. The "incremental" guarantee of I1 at the graph level means: (a) the recompute is DEBOUNCED (not triggered per-file-change), and (b) new page rows for newly-ingested files are added incrementally without deleting and recreating unrelated rows. The coord UPDATE (via FA2) touches all rows because FA2 is a global layout, but it does so only once per debounce window — not per ingest event. Functional analyst recommends the test assert that: after ingesting one new file, only one new pages row is added (the new file's page); no other pages rows are deleted or recreated. The coord columns (x, y) of existing pages may change after the debounced FA2 recompute — this is acceptable because FA2 is a global layout algorithm. Please confirm this interpretation via AQ-v0.3-4 before QA writes the test.

    AC-F4-9 Given a vault with N pre-existing pages all having non-NULL x/y coordinates in Postgres, when one new file is ingested via the watcher (adding one new page row), then: (a) Exactly one new row is added to the pages table (the new page); no existing page rows are deleted or have their id, title, type, sources changed. (b) The new page row exists in the pages table with the correct file_path and title. (c) The Postgres pages table row count increases by exactly 1. Test target: test_incremental_graph_update.py (new) — seed N pages with x/y; trigger one ingest; assert table count N+1; assert all N pre-existing page ids still present with unchanged id/title/type. EC reference: EC-M3-8, G1.

    AQ ID Blocks ACs Question Recommended resolution Urgency
    AQ-v0.3-1 AC-F4-1 Are the 4 edge-weight signals ADDITIVE (summed per edge) or do they use a different combining formula (e.g., max, product, weighted product)? The AC fixture expected values assume additive. The BACKLOG.md spec says "×3", "×4", "×1.5", "×1" — these are multipliers on what base signal? Is it: base_signal=1 for presence, then multiplied? Or is weight = sum of (multiplier × signal_value) for each contributing signal? Functional analyst recommends: for presence-based signals (direct-link, source-overlap, type-affinity), base = 1 if condition is met, 0 otherwise; weight = sum over all contributing signals of (multiplier × base). For Adamic-Adar, multiply the igraph-computed AA coefficient by 1.5. Total edge weight = sum of all applicable signal contributions. Architect to confirm or revise formula and publish in ADR before test fixture values are finalized. P0 — blocks AC-F4-1 unit test values
    AQ-v0.3-2 AC-F4-2 Should igraph FA2 be seeded with a fixed integer (e.g., seed=42) for deterministic output, or is non-deterministic output acceptable? Recommend: fix seed=42 for reproducibility in tests and regression detection. Non-deterministic coords make regression testing harder. If architect wants non-deterministic (e.g., for better layout variety on vault changes), the unit test must only check non-NULL + coordinate range rather than exact values. P1 — blocks AC-F4-2 determinism assertion
    AQ-v0.3-3 AC-F4-3, AC-F4-4 Should GET /graph block (synchronous, return 200 with full payload only when FA2 is complete) or return HTTP 202 + job_id when FA2 is running and coords are stale? Functional analyst recommends: synchronous for v0.3. Graph recompute on igraph for expected vault sizes (<500 nodes) should complete in <2s. Return 200 when coords are ready. If no coords exist yet (first ever call), trigger FA2 synchronously and return 200. Add a "stale" indicator in the response if desired but do not return 202. P0 — blocks AC-F4-3 test design
    AQ-v0.3-4 AC-F4-9 "Incremental" at the graph coord level: FA2 is a global layout algorithm that repositions ALL nodes on every recompute. Does AC-F4-9 assert (a) only 1 new pages row is added per ingest (correct), or (b) only the new page's coords change while others remain frozen (incorrect — FA2 moves all nodes)? Confirm interpretation: AC-F4-9 tests row-level incrementality (exactly 1 new row added per file ingest), NOT coord-value stability across recomputes. Coord values for existing pages WILL change after FA2 reruns — this is expected and correct. Document this explicitly in test_incremental_graph_update.py to avoid confusion. P1 — clarifies AC-F4-9 assertion scope
    AQ-v0.3-5 AC-F4-1, AC-D2v3-1 Is the edges table stored persistently in Postgres (written after every FA2 run and read directly for GET /graph) or computed in-memory on every graph request? Recommend: store in Postgres edges table (as specified in BACKLOG.md §F4 scope). This enables GET /graph to be purely a read (no recompute) when cache=hit. In-memory computation would require FA2 on every GET /graph, violating I2 cache intent. P1 — BACKLOG already specifies columns on pages; confirm no change
    AQ-v0.3-7 AC-F16db-1 How does GraphCache detect dataVersion changes: (a) polling vault_state table on a configurable interval (e.g., every 2 seconds), (b) Postgres LISTEN/NOTIFY, or (c) in-process event emitted by the watcher after each ingest?
