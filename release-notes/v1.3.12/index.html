    Synapse v1.3.12 — Review pass: correttezza, performance & hardening¶

    Giro di review multi-asse (correttezza · sicurezza · performance · frontend · provider) guidato dalla metodologia agent-skills, con test come gate di verifica. Chiude una serie di difetti latenti (non coperti dai test esistenti) e aggiunge tre hardening di sicurezza a rischio zero. Nessuna migrazione DB. Ribuilda backend e frontend.

    🔒 Sicurezza¶

    • Warning di postura auth all'avvio [H1] — se SYNAPSE_AUTH_TOKEN è vuoto l'API è di fatto aperta (il middleware fa pass-through): ora un WARNING in startup lo rende visibile nei log. Non blocca il boot — impostare il token resta scelta dell'operatore.
    • Rate-limit consapevole del proxy [H3] — il limiter ora usa lo stesso resolver trusted-proxy della classificazione sorgente (estratto in app/client_ip.py): dietro Cloudflare/Tailscale (con MCP_TRUSTED_PROXIES) conta per client reale invece di collassare in un unico bucket globale. XFF fidato solo da un proxy fidato.
    • Header di risposta di hardening [H4] — X-Content-Type-Options: nosniff, X-Frame-Options: SAMEORIGIN, Referrer-Policy su ogni risposta; HSTS solo su HTTPS (via x-forwarded-proto, non rompe l'http locale). CSP rimandata a un intervento dedicato.

    ⚡ Performance¶

    • Graph: persistenza in batch [F4][I2] — _persist_results non fa più un round-trip per nodo/arco: executemany unico (SQL invariato → semantica Postgres identica). Su vault grandi la fase di scrittura passa da migliaia di round-trip sequenziali a due chiamate.
    • RAG: lettura passaggi non-bloccante [F5][I3] — la lettura del file sorgente in fase di assemblaggio esce dall'event loop (asyncio.to_thread): non blocca più le chat in corso.
    • Embeddings: client HTTP persistente [F5] — pool di connessioni riusato tra le chiamate invece di un handshake TCP/TLS nuovo a ogni embedding (chiuso allo shutdown).
    • Stats: niente doppia scansione [F18] — /stats/sections non riscansiona più la tabella pages una seconda volta per i titoli.

    🐛 Fix¶

    • Deep-research: turni provider limitati nel tempo [F10][I7] — le 3 chiamate del loop (query, sufficienza, sintesi) ora hanno un timeout (asyncio.wait_for): un provider bloccato non può più lasciare un run appeso a running per sempre.
    • Ops: scoping per vault [F13][K2] — il rilevamento orfani e il match slug della cascade-delete ora filtrano per vault_id (correttezza in scenari multi-vault).
    • Provider: gate costo ripristinato [F17][I7] — se PROVIDER_PRICE_MAP non è impostata sul backend a pagamento, ora parte un WARNING una-tantum invece di registrare silenziosamente total_cost_usd=0 (che disattivava il controllo anomalie di costo).
    • Chat: meno re-render in streaming [F6][I3] — MessageRow non riceve più l'intero array messaggi come prop (leggeva dallo store al click), così il memo non salta a ogni token; la telemetria di costo per-turno passa a console.info solo in dev.

    ⬆️ Aggiornamento¶

    • Backend + frontend: rebuild — ripulla ghcr.io/emanuele-chiummo/llm-wiki-synapse-backend:v1.3.12 e -frontend:v1.3.12. Nessuna migrazione DB, nessuna azione manuale.
    • Nuova env (opzionale): DEEP_RESEARCH_PROVIDER_TIMEOUT_SECONDS (default 120) — timeout per turno delle chiamate provider del deep-research.
