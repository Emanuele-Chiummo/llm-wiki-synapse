    Architect Review — v0.5 / M5 Phase 5 (F1-MCP-UI)¶

    • Branch: sprint/v0.5
    • Feature: F1-MCP-UI (PM Amendment A1) — read-only MCP server introspection endpoint + Settings panel
    • Contract: ADR-0027 (docs/adr/0027-mcp-info-ui.md)
    • Reviewer: solution-architect
    • Date: 2026-06-29
    • Scope: read-only implementation review against ADR-0027 + invariants. Test suite NOT run (QA owns, in parallel).
    • Headline invariants: I9 (don't reinvent MCP — display only) · I6 (nothing about the MCP server hardcoded)

    Verdict: APPROVE¶

    The implementation is a faithful, minimal realization of ADR-0027. Every Do-NOT holds. The endpoint is pure in-process introspection of the already-built mcp object; the panel is read-only with no mutation path; all MCP metadata is derived from the live object + settings, none hardcoded. No DB, no migration, no second server, no tool invocation, no transport session.

    Do-NOT verification (ADR-0027 §4)¶

    # Do-NOT (ADR-0027 §4) Status Evidence
    1 No hardcoded tool list / name / description / schema (backend or frontend) HELD Backend derives all from await _mcp_server.list_tools() — backend/app/main.py:1635-1644 (t.name, t.description, t.parameters). Frontend renders from info.tools.map(...) — SettingsPanel.tsx:751-790; no JSX literal tool data. i18n values carry no tool/server literals (verified en.json/it.json).
    2 No tool invocation from UI; endpoint opens NO MCP transport/stdio session (no asyncio.run, no thread bridge, no DB, no Qdrant) HELD Handler body is only await _mcp_server.list_tools() + field reads — main.py:1634-1652. No DB session, no Qdrant client, no mcp.run(...). Module import is side-effect-free: mcp.run(transport="stdio") is guarded by if __name__ == "__main__" — mcp/server.py:346-348. Frontend has no code path to any tool: fetchMcpInfo is the only call — providerClient.ts:134-141.
    3 No config-write / CRUD / mutation — read-only end to end HELD Endpoint is @app.get("/mcp/info") only — main.py:1613. No POST/PUT/PATCH/DELETE added. Frontend has no write action; handleCopy writes to clipboard only — SettingsPanel.tsx:670-677.
    4 No new MCP capability / second server / duplication of server.py logic (I9) HELD Reuses the existing instance via from app.mcp.server import mcp as _mcp_server — main.py:77. No new FastMCP(...), no @mcp.tool, no reimplemented tool logic.

    (Also confirmed: Do-NOT #6 no DB table/migration — no mcp migration in backend/alembic/versions; Do-NOT #7 no Zustand store — SectionApiMcp uses local useState + AbortController on mount, mirroring SectionEmbeddings, SettingsPanel.tsx:636-648.)

    "Nothing hardcoded" confirmation (I6)¶

    • server_name ← _mcp_server.name (the live FastMCP object) — main.py:1647.
    • transport ← settings.mcp_transport (env MCP_TRANSPORT, default "stdio") — main.py:1648, config.py:183-188.
    • entry_point_command ← settings.mcp_entry_command (env MCP_ENTRY_COMMAND, default "python -m app.mcp.server") — main.py:1649, config.py:190-195.
    • tool_count ← len(tools); tool name/description/schema ← list_tools() items — main.py:1637-1651.
    • Claude Desktop snippet is generated from the payload: entry_point_command tokenised to command + args, server keyed dynamically by server_name — SettingsPanel.tsx:655-668. The displayed JSON is the single source copied to clipboard (buildClaudeDesktopSnippet(info) rendered at :721, copied at :672). Not a JSX literal.

    No tool name, description, schema, server name, transport, or entry command appears as a literal in the route function or in JSX. Defaults live in config.py and are env-overridable — consistent with how embedding_* are read. I6 holds end-to-end.

    Read-only confirmation¶

    Backend: single GET route, body performs introspection + field reads only — no write to DB, file, Qdrant, or MCP transport. Frontend: fetch-on-mount + render; the only side effect is navigator.clipboard.writeText. There is no code path from the UI to a tool invocation or a config write. Read-only end to end.

    Additional checks (ADR-0027 §2.2 / §2.5)¶

    • Clean async introspection: await _mcp_server.list_tools() inside async def get_mcp_info — direct await, no asyncio.run, no sync-bridge hack (main.py:1626,1635). Matches the ADR §2.2 contract. .parameters (not .inputSchema) is read, with a {} fallback when None (main.py:1641) — defensive and correct per the §2.2 verified-facts note.
    • D4 (OpenAPI): GET /mcp/info present in docs/api/openapi.json (3 occurrences). Zero-drift gate satisfied for the route. (Confirm regeneration timestamp is current at the docs gate.)
    • D2 (ER): unchanged — no DB table, no Alembic migration touching MCP. Correct.
    • i18n parity: settings.apiMcp.* key-sets identical in en.json and it.json (12 keys each incl. paramCount/paramCount_plural). apiMcp.comingSoon fully retired (no remaining reference). API-sourced values are never i18n keys — only labels are translated. Parity gate satisfied.

    Invariant tension¶

    None blocking. Two notes for the record (both already documented as ADR-0027 §3 trade-offs, accepted):

    1. Import-time registry, not a live handshake. /mcp/info reflects the in-process registry, not a live stdio session. Intentional and correct for a static, code-defined tool set; if tools ever become runtime-dynamic, list_tools() is already called per request so it would still reflect them. No action.
    2. Snippet assumes command + args launch. Bespoke wrappers (e.g. docker exec) are out of scope for M5; the derived default is editable by the user after paste. No action.

    Minor observations (non-blocking, no conditions attached)¶

    • buildClaudeDesktopSnippet was placed inline in SectionApiMcp (SettingsPanel.tsx:655) rather than in providerClient.ts as the task framing implied. This does not violate the ADR (which only requires the snippet be generated from the payload, not a literal — satisfied). Acceptable as-is; if a future panel needs the same builder, lift it to the client then. Not a condition.
    • Tool description truncation splits on [.!?] then caps at 80 chars (SettingsPanel.tsx:753-756) — matches ADR §2.4 "first sentence, truncated to 80". Good.

    Conditions¶

    None. Approval is unconditional for the architecture/contract gate.

    Out-of-scope reminders for the phase gate (not my gate to close): - QA: confirm pytest (200, tool_count >= 4, server_name == "synapse", entry_point_command non-empty) and vitest (4 tool rows rendered, copy button present) are green. - tech-writer/devops: confirm make openapi output committed and docs/screens/settings-api-mcp.png (D5) captured at the docs gate.

    Handoff: verdict → orchestrator. ADR-0027 unchanged (implementation matched the design; no amendment required).
