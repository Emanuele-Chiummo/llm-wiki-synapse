    Sprint 5 — v0.5 — User Stories + Acceptance Criteria¶

    Milestone: M5 — "Feature parity core" Authored by: functional-analyst (claude-sonnet-4-6) Date: 2026-06-29 Source of truth for ACs: docs/sprints/v0.5-scope.md §4 Exit criteria reference: docs/sprints/v0.5-scope.md §8 (EC-M5-1..21 + EC-M5-HCP) Invariants with heightened priority: I7 (bounded loops — HEADLINE), I9 (SearXNG reuse — HEADLINE) All 9 invariants apply.

    How to read this document¶

    Each user story block contains: - Story ID (e.g. S-F5-1): referenced in TRACEABILITY.md. - Story header: "As a [role], I want [capability], so that [outcome]." - Numbered acceptance criteria (AC-F5-1, AC-F5-2, …): each is a concrete binary assertion. - Invariants touched: priming engineers on which non-negotiables are exercised. - D-artifacts impacted. - Planned test target(s): forward reference for qa-test-engineer.

    Binary pass/fail interpretation: every AC is either fully met or not met. No partial credit.

    Roles used in stories¶

    Role Who
    vault owner Emanuele — uses Synapse to manage and query a personal knowledge base

    Phase 1 — Retrieval foundation + carry-forward resolution¶

    S-F5-1 — 4-phase RAG retrieval pipeline¶

    As a chat user, I want my questions answered using context retrieved from the wiki via a 4-phase pipeline (tokenized keyword search → graph-expansion → token-budget allocation → context assembly), so that assistant answers are grounded in the actual vault content rather than the model's parametric knowledge alone.

    Invariants touched: I1 (no vault rescan; reads from Qdrant + Postgres only), I2 (graph-expansion reads from the existing edges table — does NOT trigger FA2 recompute), I3 (retrieval assembled server-side; no per-token heavy work), I9 (reuse Qdrant + bge-m3; no new embedding service).

    D-artifacts: D4 (GET /search endpoint in openapi.json).

    AC-F5-1 Given backend/app/rag/retrieval.py is implemented, when a retrieval call is made, then the pipeline executes all four phases in order: (1) tokenized keyword search over Qdrant using bge-m3 vectors; (2) graph-expansion following edges from top-k results in the Postgres links and edges tables, up to depth 2; (3) token-budget allocation using 20% of the configured context window (per F14); (4) context assembly producing a RetrievalContext Pydantic object with page refs. Test target: backend/tests/test_retrieval.py — unit test with mocked Qdrant client and Postgres edges; assert all 4 phases execute and RetrievalContext is returned.

    AC-F5-2 Given the pipeline has assembled a RetrievalContext, when the context is inspected, then every retrieved passage carries a 1-based citation index n; the assembled context string contains inline [n] markers; and RetrievalContext maps each n to a PageRef with fields id (uuid), title (str), and slug (url-friendly str, not null). Test target: backend/tests/test_retrieval.py — assert len(context.citations) == n_passages, each citation has id, title, slug; assert [1] appears in context.text.

    AC-F5-3 Given the retrieval pipeline is executing, when it performs vector search and graph-expansion, then it calls only the existing Qdrant wrapper (bge-m3) and reads from Postgres links/edges tables — no import of a new embedding service, no new Qdrant collection, no direct Ollama embedding call. Test target: backend/tests/test_code_quality.py — static import check: no bge, sentence_transformers, or new Qdrant collection creation outside the existing Qdrant wrapper module.

    AC-F5-4 Given 10 fixture passages totalling 2× the configured token budget, when the context assembly phase runs, then the assembled RetrievalContext.text length (in tokens) is less than or equal to the budget; the lowest-ranked passages are dropped first; no passage is silently truncated mid-sentence without removal. Test target: backend/tests/test_retrieval.py — seed 10 fixture passages; configure budget to half their total; assert len(context.citations) < 10 and total token length ≤ budget.

    AC-F5-5 Given a retrieval call has completed, when the Postgres vault_state.data_version is read before and after the call, then the value is identical (graph-expansion reads the edges table; it does NOT bump data_version or trigger FA2). Test target: backend/tests/test_retrieval.py — assert data_version_before == data_version_after.

    AC-F5-6 Given the FastAPI application is running, when GET /search?q=<query>&vault_id=<uuid> is called, then the endpoint returns HTTP 200 with a JSON body containing at least results (array of citation objects) and context (assembled string with [n] markers); the endpoint schema is present in docs/api/openapi.json. Test target: backend/tests/test_api.py — ASGI test client; assert HTTP 200 and required fields. backend/tests/test_docs.py — assert openapi.json contains "/search" path.

    AC-F5-7 Given the retrieval unit test suite, when all unit tests in test_retrieval.py run, then the following scenarios all pass: (a) 0-result query returns empty RetrievalContext with no citations; (b) single-result query returns RetrievalContext with exactly 1 citation; (c) multi-page expansion returns linked pages in the expansion phase (seed 5 pages with known wikilinks; assert linked pages appear in results); (d) budget-overflow truncation drops lowest-ranked passages until budget is satisfied. Test target: backend/tests/test_retrieval.py — 4 parametrized test cases.

    AC-F5-8 Given F5 retrieval is implemented, when the chat endpoint processes a user message, then the assembled RetrievalContext from F5 is passed to the active provider's chat(messages, retrieval_context) call for all three backends (OllamaProvider, ApiProvider, CliAgentProvider), so that [n] markers appear in assistant messages. Test target: backend/tests/test_chat_endpoint.py — mock all three providers; assert each chat() call receives a RetrievalContext argument (not None).

    Phase 2 — Deep Research loop¶

    S-F10-1 — Deep Research loop: bounded multi-query SearXNG loop¶

    As a researcher, I want Synapse to autonomously gather web sources on a topic by running multiple SearXNG queries, assessing whether collected content is sufficient, and synthesizing a new wiki page — all within explicit iteration and cost bounds, so that I can augment the vault with current information without manual web searching.

    Invariants touched: I7 (max_iter + token_budget + concurrency ≤ 3 are mandatory; any unbounded path is a P0 blocker), I9 (SearXNG is the only permitted web search backend; never Tavily, DuckDuckGo, or Google), I6 (all inference calls go through InferenceProvider — query generation, sufficiency assessment, synthesis), I1 (synthesis fed through the ingest pipeline, not written directly to vault/wiki/).

    D-artifacts: D2 (deep_research_runs + deep_research_sources tables in schema.mmd), D3 (docs/sequences/deep-research.mmd), D4 (research endpoints in openapi.json).

    AC-F10-1 Given backend/app/ops/deep_research.py is implemented, when run_deep_research(vault_id, topic, max_iter, token_budget) is called, then the function executes all 6 steps in order: (1) generate 2–5 SearXNG queries via InferenceProvider; (2) execute queries against SearXNG with concurrency=3 (asyncio semaphore); (3) fetch and parse result pages to markdown; (4) call InferenceProvider to assess sufficiency; if sufficient → step 5; if not and iterations < max_iter → generate follow-up queries and repeat from step 2; (5) synthesize via InferenceProvider; (6) feed synthesis into the ingest pipeline (not directly into vault/wiki/). Test target: backend/tests/test_deep_research.py — mock SearXNG and InferenceProvider; assert all 6 steps execute; assert synthesis routed to ingest pipeline.

    AC-F10-2 (I7 — headline invariant for M5) Given deep_research.py is running with a mock InferenceProvider that always returns "insufficient", when the loop executes, then: (a) the loop terminates after exactly max_iter refinement rounds; (b) the run status is set to "max_iter_reached" (not still "running"); (c) no further InferenceProvider calls are made after max_iter is reached; (d) max_queries_per_iter is not exceeded per round (default 5); (e) concurrency is at most 3 (asyncio semaphore prevents more than 3 concurrent SearXNG fetches). Test target: backend/tests/test_deep_research.py — mock "always insufficient" provider; assert call_count == max_iter; assert run.status == "max_iter_reached". THIS TEST IS MANDATORY — EC-M5-5 explicitly requires it.

    AC-F10-3 (I9) Given the deep research implementation and all its dependencies, when python -m pytest backend/tests/test_code_quality.py -k "no_tavily" runs, then: (a) no file in the codebase imports tavily, duckduckgo_search, googlesearch, or google-search-results; (b) SEARXNG_URL env var is the only web-search configuration entry point. Test target: backend/tests/test_code_quality.py — static import scan; assert zero matches for tavily|ddg|duckduckgo|googlesearch in all .py files.

    AC-F10-4 Given the FastAPI application is running with deep research implemented, when the following REST calls are made: - POST /research/start with body {vault_id, topic, max_iter?, token_budget?}, - GET /research/runs, - GET /research/runs/{id}, then: (a) POST returns HTTP 202 with {run_id: uuid}; (b) GET /research/runs returns a list of past runs (each with status and total_cost_usd); (c) GET /research/runs/{id} returns run detail including queries_used (JSONB array), sources_fetched (int), and synthesis_text (str or null if not yet complete); (d) all three endpoints are documented in docs/api/openapi.json. Test target: backend/tests/test_research_api.py — ASGI test client; assert each response shape. backend/tests/test_docs.py — assert openapi.json contains "/research/start".

    AC-F10-5 Given the F9 review queue is operational and a Deep-Research action fires on a review item, when POST /review/queue/{id}/deep-research is called, then: (a) POST /research/start is called with the review item's pre_generated_query as the topic (or the page topic if query is null); (b) the returned run_id is stored on the review_items row as a foreign key or JSONB reference; (c) the review item status is set to "deep_researched". Test target: backend/tests/test_review_integration.py — seed a review_item with pre_generated_query; call deep-research action; assert run_id stored and status updated.

    AC-F10-6 Given a deep research run completes (success or max_iter_reached), when the Postgres tables are inspected, then: (a) a row exists in deep_research_runs with columns: id, vault_id, topic, status, queries_used (JSONB), sources_fetched (int), total_cost_usd, started_at, completed_at, error_message (nullable); (b) one or more rows exist in deep_research_sources with columns: id, run_id, url, title, fetched_content_md, relevance_score; (c) Alembic migration files for both tables exist in backend/alembic/versions/; (d) make er exits 0 and docs/er/schema.mmd reflects both tables. Test target: backend/tests/test_models_schema.py — static introspection of SQLAlchemy models; assert all required columns present. CI: make er exits 0.

    AC-F10-7 Given the deep research unit test suite, when all tests in test_deep_research.py run, then the following scenarios all pass: (a) mock SearXNG returning 3 results → assert exactly 3 fetch calls made; (b) mock InferenceProvider returning "sufficient" on the first assessment → assert loop completes after exactly 1 round and proceeds to synthesis; (c) mock InferenceProvider returning "insufficient" for max_iter rounds → assert loop terminates at max_iter with status "max_iter_reached" and no further provider calls; (d) synthesis output is passed to the ingest pipeline, not written directly to vault/wiki/. Test target: backend/tests/test_deep_research.py — 4 parametrized test cases.

    S-F12-1 — Multi-format document extraction and ingest¶

    As a vault owner, I want to upload PDF, DOCX, PPTX, XLSX, and image files to the Sources section and have them automatically ingested into the wiki, so that research materials in common office formats can be added to the knowledge base without manual conversion.

    Invariants touched: I1 (uploading the same file twice does not create a duplicate — existing content-hash check applies regardless of file type), I5 (extracted text goes to vault/raw/sources/ as a companion .extracted.md; the original binary is preserved; vault/wiki/ is only written by the ingest pipeline — K1 3-layer rule intact), I6 (extractors do not call any LLM; all inference goes through InferenceProvider in the existing ingest pipeline).

    D-artifacts: D4 (updated MIME type list in openapi.json).

    AC-F12-1 Given backend/app/ingest/extract.py is implemented with a extract_text(file_path) -> str dispatch function, when files of each supported type are passed to extract_text, then: (a) PDF (.pdf): text from all pages extracted via pypdf; (b) DOCX (.docx): paragraph text extracted via python-docx; (c) PPTX (.pptx): slide text extracted via python-pptx; (d) XLSX (.xlsx): cell data from all sheets extracted as markdown tables via openpyxl; (e) Images (.png, .jpg, .jpeg, .gif, .webp): OCR via unstructured if installed, otherwise returns the string "image file: no text extracted" — no exception raised in either case; (f) AV files (.mp3, .mp4, .wav, .m4a): returns "AV file: transcript not available in this release". The extracted text is passed unchanged to the existing analyze()/generate() ingest loop. Test target: backend/tests/test_extract.py — fixture files for each type; assert extracted text contains expected content; assert no LLM call made during extraction.
