    Architect PR Review + Manual Diagram Sign-off — Sprint 1 / v0.1 / M1¶

    Reviewer: solution-architect Date: 2026-06-28 Branch: sprint/v0.1 Scope: pre-milestone invariant gate (EC-13) + manual C4/ER sign-off (AC-D1-3, AC-D2-3) Basis: CLAUDE.md §3 invariants · docs/sprints/v0.1-architecture.md · ADR 0001–0006 Inputs reviewed: models.py, watcher.py, ingest/orchestrator.py, main.py, vault.py, embeddings.py, qdrant_client.py, config.py, alembic/versions/0001_initial_schema.py, docs/architecture/context.mmd, docs/architecture/container.mmd, docs/er/schema.mmd, QA report, DOCS_STATUS.

    This is the invariant gate. I reviewed the implementation against my own design note and the 9 invariants. No code was modified. Verdict per item below, then overall.

    1. Review checklist (per task)¶

    Task 1 — I1 (incremental index only) → APPROVE¶

    • No directory walk in executable code. Static scan of backend/app/ for os.walk / os.listdir / .rglob / .glob( / glob.glob / iterdir: zero hits in executable code. The only os.scandir use is watcher.py::_emit_startup_notice, which opens a scandir iterator and calls next(it) exactly once inside a with block to test non-emptiness (O(1)), then logs the AQ-3 INFO line. It does not loop, does not collect entries, and triggers no ingest. This is the AQ-3 "no rescan, one INFO line" behaviour from ADR-0006, correctly implemented — it is an existence probe, not an enumeration. Compliant.
    • Watcher reacts to events only. VaultWatcher.start() registers a watchdog Observer with a _MarkdownHandler and returns; it never enumerates. on_created/on_modified → ingest_file; on_deleted → delete_file; on_moved correctly decomposed into ingest(dst) + delete(src) for atomic-rename saves. Compliant.
    • mtime-then-hash gate matches ADR-0001. ingest_file implements the exact 3-case policy: (1) existing.source_mtime_ns == current_mtime_ns → SKIP, no read/hash/write; (2) mtime differs → read + sha256, and if content_hash unchanged → _touch_mtime only, SKIP (no embed/log/version bump); (3) hash differs or new file → full upsert. Hash is sha256 over raw bytes (frontmatter + body), so frontmatter-only edits are detected. The gate lives in the seam (shared by watcher and REST), per ADR-0001/0003. Exact match to ADR-0001.
    • Soft-delete path correct. delete_file selects the live row (deleted_at IS NULL), sets deleted_at = now() + updated_at, leaves all other columns intact, then hard-deletes the Qdrant point (asymmetric soft/hard per ADR-0002/0005). No-op + warning if no live row. Resurrection is handled in persist_metadata (clears deleted_at on re-ingest of the same path). Does not bump data_version on delete (ADR-0005, correct for v0.1). Compliant.
    • No spurious data_version bump on skip: bump_version is only reached on the content-changed branch. Monotonic-by-construction holds (AC-F16dv-4). Compliant.

    I1 verdict: APPROVE. No re-scan anywhere; incremental-only honoured strictly.

    Task 2 — I6 (F17 pluggability preserved) → APPROVE¶

    • Seam is thin and provider-agnostic. ingest_file is the single entry point; both the watcher and POST /ingest/trigger call only ingest_file (verified in watcher.py and main.py::trigger_ingest). Neither caller reaches Postgres/Qdrant/embedding directly. This satisfies the ADR-0003 review rule.
    • Extension point correctly placed. The F17 slot sits after the hash gate and before persist_metadata (orchestrator.py lines ~136–140), exactly where v0.2 will resolve a provider from provider_config, call capabilities(), and branch orchestrated-loop vs delegated-CLI. The mechanical primitives (persist_metadata, upsert_vector, append_log, bump_version) are factored as standalone async helpers, so the v0.2 loop reuses them as validate/write_page without rewriting callers. The delegated-CLI vs orchestrated-loop separation (I6 routing) is documented in the module docstring as the v0.2 insertion contract. The seam will accept the F17 loop without a caller refactor — design intent preserved.
    • Zero provider/LLM/model-id leakage. Static scan for claude-, gpt-, anthropic, openai, ollama, inferenceprovider, provider_config, llm: no hits in executable code. The only matches are in comments/docstrings describing the v0.2 extension point (# otherwise (OllamaProvider / ApiProvider) and v0.1 is MECHANICAL ONLY — no LLM call). No provider instantiation, no provider import, no model id, no provider_config table. EMBEDDING_URL is correctly scoped to vectorisation (I9), not to the F17 abstraction. The graph/, mcp/, ops/, rag/ package dirs are empty — no v0.2+ code prebaked. Compliant.

    I6 verdict: APPROVE. F17 extension point is correctly seated; v0.2 plugs in without touching callers. No provider leaked into v0.1.

    Task 3 — I9 (no reinvention) → APPROVE¶

    • Embeddings only via EMBEDDING_URL. HttpEmbeddingClient POSTs to settings.embedding_url with {"model": settings.embedding_model, "prompt": text}. No in-process model load, no subprocess spawn. EmbeddingClient ABC + FakeEmbeddingClient is a test seam (GAP-4), not a second backend. Compliant.
    • Qdrant only via QDRANT_URL. AsyncQdrantClient(url=settings.qdrant_url); thin wrapper exposing ensure_collection / upsert_point / delete_point. No new vector store. Compliant.
    • No new vector/embedding service introduced. docker-compose.yml defines only postgres + synapse-backend; Qdrant and bge-m3 are external (reached over the network), per the I9 reuse mandate. No Tavily reference anywhere (grep: zero hits). Compliant.
    • EMBEDDING_DIM is config, not a literal. Settings.embedding_dim: int is a required env var (no default → pydantic fails fast if absent). .env.example documents 1024. Scan for 1024/768/1536 in executable code: zero hits (only a docstring mention in config.py). The Qdrant collection vector size flows from settings.embedding_dim, and startup validates it against the live bge-m3 via probe_dimension() (ADR-0004 fail-fast). ensure_collection refuses to silently recreate a dimension-mismatched collection. Compliant.

    I9 verdict: APPROVE. Existing infra reused; no reinvention; dimension is config-driven.

    Task 4 — Data model fidelity (matches spec + migration matches models = D2 truth) → APPROVE¶

    pages columns vs spec (architecture §2.1): all present and typed as specified — id (UUID PK, default gen_random_uuid()), vault_id, file_path, title, type (mapped as page_type, column name "type"), sources (JSONB), content_hash (String(64)), source_mtime_ns (BigInteger), qdrant_point_id (UUID), deleted_at (TIMESTAMPTZ), created_at, updated_at. 12/12 columns — exact match.

    • Partial unique index uix_pages_vault_file_path_live on (vault_id, file_path) WHERE deleted_at IS NULL — present in models.py __table_args__ AND in the Alembic migration (postgresql_where=sa.text("deleted_at IS NULL")). Enforces one live row per path at the DB level (ADR-0005's earlier "application-logic only" gap is now closed in the schema). Compliant.

    vault_state columns vs spec: id (UUID PK), vault_id (String + UniqueConstraint uq_vault_state_vault_id), data_version (Integer, server_default 0), updated_at. 4/4 — exact match. Monotonic semantics enforced in code (bump_version is the only writer that increments; seed/restart/GET/skip/delete never change it). Compliant.

    • Migration matches models (D2 reflects reality). alembic/versions/0001_initial_schema.py creates both tables with identical column names, types, nullability, server defaults, PK/unique constraints, and the partial unique index. Column "type" is correctly emitted in SQL (not the Python attribute page_type). provider_config correctly absent (ADR-0003). The committed docs/er/schema.mmd reflects all 12 pages + 4 vault_state columns including source_mtime_ns (the ADR-0001 superset) and deleted_at. The model is the single source of truth and make er output matches it. Compliant.

    Non-blocking note (v0.2 hygiene, not a defect): gen_random_uuid() is native to Postgres 13+; the target stack is Postgres 16, so no pgcrypto extension dependency is introduced. No action required for v0.1.

    Data model verdict: APPROVE. Models match spec; migration matches models; ER reflects reality.

    Task 6 — Any invariant violation → NONE FOUND¶

Compliant. - I7 (bounded loops): no while True / while 1; no loops exist in v0.1; the empty F17 branch is where v0.2's max_iter+token_budget loop lands. Compliant (not exercised). - I8 (docs-as-DoD): D1 (C4 context+container), D2 (ER from models), D4 (OpenAPI) present and consistent; ADRs 0001–0006 committed. Compliant. - I2/I3/I4: not exercised in v0.1 (no graph/chat/UI); nothing in v0.1 precludes them. data_version is correctly shaped as the future FA2 debounce signal. No regression.

    No blocking defect found across all 9 invariants.

    2. Manual diagram sign-off (AC-D1-3, AC-D2-3)¶

    AC-D1-3 — C4 context + container → SIGNED OFF (APPROVE)¶

    docs/architecture/context.mmd: - Actors/systems accurate to v0.1: Person(emanuele), System(synapse) (explicitly "NO LLM inference in v0.1"), System_Ext(qdrant), System_Ext(bgem3), System_Ext(vaultfs). - Relationships correct: Synapse reads raw/sources/ events and writes wiki/log.md + .obsidian/; requests embeddings from bge-m3; upserts/deletes vectors in Qdrant. - No v0.2+ component drawn as present — no InferenceProvider, no MCP server, no SearXNG, no UI client, no provider_config. Skeleton-correct.

    docs/architecture/container.mmd: - System_Boundary(synapse) wraps exactly the v0.1 components: FastAPI service, Watcher, Ingest seam, Postgres 16. Externals (Qdrant, bge-m3, vault filesystem) sit outside the boundary — correct, since they are reused infra (I9), not Synapse-owned containers. - The seam container description explicitly labels the v0.2 F17 extension point ("Thin provider-agnostic seam — F17 loop slots in here at v0.2 (I6)") — accurate and forward-honest without drawing v0.2 as built. - Relationships match the architecture note §2 and the actual call graph: both api and watcher call seam via in-process ingest_file; only seam touches bge-m3/Qdrant/log; api reads pg. The 202-on-trigger edge is annotated. Accurate.

    Minor presentational note (non-blocking, no change required): the C4 "Container" elements here model in-process modules (watcher/seam run inside the FastAPI process) rather than separate deployable containers. This is an accepted and common C4 modelling choice for a single-process service and is internally consistent with the deploy reality (one synapse-backend compose service). No correction demanded.

    AC-D1-3: APPROVED. C4 context + container are accurate and skeleton-correct.

    AC-D2-3 — ER diagram → SIGNED OFF (APPROVE)¶

    docs/er/schema.mmd: - PAGES lists all 12 columns with correct types and the id PK marker; type is rendered (not the Python attribute page_type), matching the SQL column. source_mtime_ns and deleted_at are present (the ADR-0001/0005 superset). - VAULT_STATE lists all 4 columns with id PK. - provider_config correctly absent (ADR-0003). - ER matches models.py and the Alembic migration exactly (zero drift, confirmed by DOCS_STATUS regeneration check). The partial unique index and vault_id unique constraint are not renderable in erDiagram syntax — this is a Mermaid format limitation, documented in the architecture note §2.1, not a drift.

    AC-D2-3: APPROVED. ER reflects the live schema; no v0.2+ table drawn.

    The v0.1 implementation faithfully realises the signed-off design. I1 (incremental-only, mtime-then-hash, soft-delete), I6 (thin F17-preserving seam with zero provider leakage), and I9 (reuse Qdrant + bge-m3, config-driven dimension) are honoured with no violations. The C4 context/container and ER diagrams are accurate and skeleton-correct — AC-D1-3 and AC-D2-3 are signed off. No invariant was traded for convenience.
