    0. Context and Mandate¶

    M4 was declared MET. The developer (Emanuele) conducted human browser testing (EC-M4-HCP) and found usability problems. The orchestrator has already shipped a first-pass fix for most of them (commits 65a6407, f7c7865, 59765f3). This hardening increment retroactively locks that scope, classifies what is done vs in-progress vs deferred, adds two new PM rulings (Point A — nav labels; Point B — M5 placeholder strategy), and defines acceptance criteria for the in-progress items so they can pass through the full gate chain.

    This is a hardening pass. It does NOT advance the milestone number. Sprint 5 (M5) remains blocked until M4-HARD exits cleanly (4 sign-offs) and Emanuele confirms EC-M4-HCP.

    2. PM Rulings on the Two New Feedback Points¶

    Ruling on Point B — M5 nav placeholders: REMOVE, do not label¶

    DECISION: Remove Search, Lint, Review, and Deep Search from the M4 nav rail entirely. Do not show them as greyed-out disabled items with or without labels.

    Rationale:

    The current M4 nav rail shows these four items in a separate group below a separator, with disabled: true styling (color #30363d — near-invisible on the dark rail) and title tooltip "Coming in M5". The human tester found them confusing: grey icons with no visible label and no click behavior are not informative — they create uncertainty about whether the UI is broken or the feature is intentionally absent.

    Two options were considered:

    Option A — Keep them, add visible "coming soon" label text. Option B — Remove them from the rail until M5 ships the logic.

    Option A fails for M4 because it requires rendering 4 additional nav items whose sole purpose is to communicate absence. A "coming soon" label next to a disabled icon does not help the user accomplish any task and adds visual noise in an already new UI. The developer has confirmed the items are non-functional; showing them in M4 is purely aspirational chrome. It also sets a precedent that every M6 feature should appear as a placeholder in M5, and so on — this is scope-creep of UI surface area.

    Option B is the correct approach for M4: the nav rail should contain only sections the user can actually use. When M5 ships Search, Lint, Review, and Deep Search logic, those items are added back to the rail with full functionality. This matches how mature products handle progressive disclosure: features appear when they work.

    The separator between active and placeholder items is also removed (there is nothing to separate). The final M4 nav rail is: Chat / Wiki / Sources / Graph / (spacer) / Settings.

    Note: the i18n translation keys for these items (nav.search, nav.lint, nav.review, nav.deepSearch, nav.comingSoon) are RETAINED in en.json and it.json so M5 can use them without introducing new keys. Only the component entries in NavRail.tsx M5_ITEMS array are removed (or the M5_ITEMS array is rendered as an empty block).

    Ruling on Point A — Nav label form: PERSISTENT LABELS BESIDE ICONS (wider rail)¶

    DECISION: Each nav item renders the icon plus a short text label below or beside the icon. The rail is widened from 48px to approximately 72px to accommodate. Labels are the existing i18n translation keys (already present in en.json and it.json). There is no tooltip-only or expandable-rail option at this stage.

    Rationale:

    Three forms were considered:

    Form 1 — Wider rail with persistent labels beside/below icons. Form 2 — Icon-only with hover tooltips. Form 3 — Expandable/collapsible rail (collapsed = icon-only, expanded = icon + label).

    Form 2 is rejected because it requires the user to hover each item to discover what it does. The human tester's exact complaint was "non parlanti" — the icons do not speak for themselves. Tooltips are a discoverability aid, not a substitute for visible labels. The icons used (message-square for Chat, files for Wiki, download for Sources, share for Graph, gear for Settings) are standard but not universal — a user unfamiliar with them cannot navigate without labels.

    Form 3 would require state management for the collapsed/expanded rail toggle, an additional Zustand slice or localStorage key, and a persistence mechanism. It introduces more complexity than the problem warrants in a hardening pass. It also creates an inconsistent initial state (collapsed by default means the user sees unlabeled icons on first load — the same problem we are trying to fix).

    Form 1 is the correct approach for M4-HARD. It is a CSS and layout change: widen the rail, add a with the label text below each icon, keep the existing i18n key bindings. The implementation is contained to NavRail.tsx and its CSS. No new Zustand state, no new API calls, no new i18n keys.

    Implementation note for frontend-engineer: the rail width change from 48px to ~72px must not break the react-resizable-panels layout. The rail is a fixed-width flex child; verify the panels still resize correctly after the width change (F1 AC-F1-7 panel resize test must still pass). The label font should be 10–11px to keep the rail compact. The active-state highlight (currently a rounded rectangle around the icon only) should extend to cover both icon and label.

    3. Acceptance Criteria¶

    F1-HARD-SETTINGS — 9-section settings panel¶

    • AC-HARD-SET-1: The Settings section renders a two-column layout: a left sub-nav listing all 9 sections (General, LLM Models, Embeddings, Source Watch, API+MCP, Output, Interface, Maintenance, About) and a right content pane showing the selected section's content.
    • AC-HARD-SET-2: Clicking a sub-nav item switches the right pane without a page reload or full component remount. Active sub-nav item is visually highlighted.
    • AC-HARD-SET-3: All 9 section titles and sub-nav labels are i18n translation keys present in both en.json and it.json (keys: settings.nav.general, settings.nav.llmModels, etc.). No hardcoded display strings.
    • AC-HARD-SET-4: Sections with placeholder content (Embeddings, Source Watch, API+MCP, Interface) render a descriptive placeholder text referencing M5 (already in en.json: settings.embeddings.comingSoon, settings.apiMcp.comingSoon). These do NOT render empty white space — they communicate clearly that the feature is coming.
    • AC-HARD-SET-5: Settings panel is accessible via the Settings nav rail item. Sub-nav keyboard navigation (arrow keys) works as expected.
    • AC-HARD-SET-6: vitest test: renders all 9 sub-nav items; clicking each switches the active pane; active item has correct aria-current or aria-selected attribute.

    F1-HARD-COLLAPSE — Panel collapse/expand¶

    • AC-HARD-COL-1: The left panel and the right panel each have a visible chevron button (or equivalent collapse trigger). Clicking the button collapses that panel to a minimal width (0px or a defined collapsed width) without removing it from the DOM.
    • AC-HARD-COL-2: Clicking the collapsed panel's expand trigger (or the chevron in its collapsed state) restores the panel to its previous width.
    • AC-HARD-COL-3: Collapse/expand state is preserved in localStorage (consistent with F16-settings AC-F16-settings-1: panel widths persist).
    • AC-HARD-COL-4: Collapsing a panel does not trigger a force layout or cause a long task on the main thread (I3). vitest: collapse action → assert no layout recompute signal.
    • AC-HARD-COL-5: vitest test: collapse left panel → assert panel width is at collapsed minimum; expand → assert panel width is restored.

    F1-HARD-PROVIDER-EDIT — Editable LLM Models section¶

    • AC-HARD-PROV-1: The LLM Models settings section renders the list of existing provider configs from GET /provider/config. Each row shows: provider type, model_id, scope (vault/global), and a Delete button.
    • AC-HARD-PROV-2: An "Add provider" form (or expandable row) allows the user to select provider type, enter a model_id, optionally enter a base_url, and select scope. Submitting calls POST /provider/config. On success (HTTP 201 or 200), the new row appears in the list without a page reload. On failure, an error toast shows the response detail.
    • AC-HARD-PROV-3: Clicking Delete on a provider row calls DELETE /provider/config/{id}. On success (HTTP 200 or 204), the row is removed from the list. A confirmation prompt is shown before the DELETE is issued (text: i18n key settings.llmModels.confirmDelete).
    • AC-HARD-PROV-4: No model_id or provider type is hardcoded in any component. The list is entirely driven by GET /provider/config. (I6 reinforced.)
    • AC-HARD-PROV-5: vitest test: mock GET /provider/config with 2 providers → assert 2 rows rendered. Mock POST → assert new row appears. Mock DELETE → assert row removed.
    • AC-HARD-PROV-6: If the user attempts to delete the last remaining provider, a warning is shown ("This is the last configured provider. Chat and ingest will fail without a provider.") but the delete is still permitted — the system does not enforce a minimum of 1 provider at the UI level.

    F1-HARD-NAV-ORDER — Nav rail order and default section¶

    • AC-HARD-ORD-1: The nav rail renders items in the following top-to-bottom order (after M4-HARD-M5-PLACEHOLDER items are removed — see below): Chat, Wiki, Sources, Graph, (spacer), Settings.
    • AC-HARD-ORD-2: The default active section on first load (no localStorage state) is Chat. Verified by vitest: fresh Zustand store → activeSection === "chat".
    • AC-HARD-ORD-3: The section previously active is restored from localStorage on page reload (F16-settings). If the restored section is one of the removed M5 items, the default (chat) is used instead — no broken state.

    F1-HARD-NAV-LABELS — Persistent text labels beside icons (Point A ruling)¶

    • AC-HARD-LBL-1: Every nav rail item renders both an icon AND a visible text label. The label is positioned below the icon (or to its right, designer's choice, but it must be persistently visible — not hidden unless hovered).
    • AC-HARD-LBL-2: The rail width is increased from 48px to accommodate the labels. Target: approximately 72px. The exact value is at the frontend-engineer's discretion provided all labels are fully readable without truncation at 72px.
    • AC-HARD-LBL-3: Label text is sourced from the existing i18n translation keys (nav.chat, nav.wiki, nav.sources [maps to ingest section], nav.graph, nav.settings). No new i18n keys are required. Labels switch language when the locale changes (i18n already wired; this is a layout change only).
    • AC-HARD-LBL-4: The active-state visual indicator (currently a rounded highlight on the icon button) is extended to encompass both icon and label. The active state must remain clearly distinguishable in both EN and IT locales (label lengths differ).
    • AC-HARD-LBL-5: Font size for labels is 10–11px (do not exceed 12px — keeps the rail visually compact and does not push items into scroll territory at typical viewport heights).
    • AC-HARD-LBL-6: The react-resizable-panels layout adjustment (rail now ~72px) does not break the F1 AC-F1-7 panel resize Playwright test. The test may need its width assertions updated to account for the new rail width — this is an expected test update, not a regression.
    • AC-HARD-LBL-7: vitest test: render NavRail → assert each item's rendered output contains both an SVG icon element and a non-empty label span with the correct i18n key value for the "en" locale.
    • AC-HARD-LBL-8: The ingest badge (running-count dot) remains functional and visible after the label is added. It does not overlap the label text.

    F1-HARD-M5-PLACEHOLDER — Remove M5 nav items from M4 rail (Point B ruling)¶

    • AC-HARD-M5P-1: The nav rail in M4 does NOT render Search, Lint, Review, or Deep Search nav items. None of these appear as enabled OR disabled items in the DOM in M4.
    • AC-HARD-M5P-2: The separator line between "active items" and "M5 placeholder items" is also removed (there is nothing to separate in M4).
    • AC-HARD-M5P-3: The M5_ITEMS array in NavRail.tsx is emptied (all 4 items removed) OR the M5 group rendering block renders nothing. The data structure may be retained as a comment or empty array for M5 to populate.
    • AC-HARD-M5P-4: The i18n keys nav.search, nav.lint, nav.review, nav.deepSearch, and nav.comingSoon are RETAINED in en.json and it.json. Only the rendered components are removed, not the translation keys. This preserves M5's ability to add items without touching the i18n files.
    • AC-HARD-M5P-5: The Section type in graphStore.ts is NOT changed — "search", "lint", "review", "deep-search" may remain as valid type members to avoid downstream TypeScript churn. The nav rail simply does not render buttons for them. If the active section is restored from localStorage as one of these values, the component falls back to "chat" (AC-HARD-ORD-3).
    • AC-HARD-M5P-6: vitest test: render NavRail → assert no button with data-section="search", data-section="lint", data-section="review", or data-section="deep-search" is in the DOM.
    • AC-HARD-M5P-7: Playwright E2E test: load the app → assert the nav rail renders exactly 5 interactive items: Chat, Wiki, Sources, Graph, Settings (4 top + 1 bottom). Assert no greyed or disabled items are present.

    Already-done stubs (MCP, Embeddings, Conv-History) — gate-only ACs¶

    These were already implemented. The QA gate must verify them; no new code is expected.

    • AC-HARD-MCP-1: Settings > API+MCP section renders with the placeholder text (settings.apiMcp.comingSoon i18n key). No interactive elements other than the sub-nav item itself. Verified by vitest render test.
    • AC-HARD-EMBD-1: Settings > Embeddings section renders with the placeholder text (settings.embeddings.comingSoon i18n key). Verified by vitest render test.
    • AC-HARD-CONV-1: Settings > Output section renders a conversation history length selector with options 2, 4, 6, 8, 10, 20. Selected value is persisted in localStorage under a stable key. Verified by vitest: change value → assert localStorage updated.
    • AC-HARD-CONV-2: The conversation history length value is read by the chat message assembler when constructing the messages array sent to the provider. If the selected value is 4, at most the last 4 messages (2 user + 2 assistant turns) are included. Verified by unit test: mock 10 messages + history=4 setting → assert sent array length is at most 4 (plus system prompt).
