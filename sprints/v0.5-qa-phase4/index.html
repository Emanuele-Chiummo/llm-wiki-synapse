    4. Gate-specific checks¶

    4.1 I1 — No full-rescan¶

    • Static proof: grep -n "glob\|walk\|listdir\|scandir" cascade_delete.py returns zero results. No vault walk in the production code.
    • Primary path (methods a/b): References are found exclusively via SELECT ... FROM links WHERE target_page_id = :id OR target_title = :T (method a) and slug-normalized comparison over links rows (method b). Both are pure index reads.
    • Method (c) fallback: Bounded by CASCADE_FULLTEXT_MAX_FILES=5000. When invoked, it queries the pages table (not a filesystem walk) to enumerate live wiki pages, then reads each file once. This is I1-safe by the ADR §3.1 definition.
    • DEFECT-F13-001: The fallback is invoked unconditionally, even when the links-table hit is complete. This is a behavioral deviation from ADR §3.1 (extra SQL query to enumerate live wiki pages on the happy path). Not a hard I1 violation (no vault walk; references are not computed from the walk) but violates the ADR architect-review gate.
    • Verdict on I1: Functionally correct (no full-rescan used to identify changes); ADR §3.1 happy-path gate not met.

    5. Defect register¶

    DEFECT-F13-001 — P1 — I1: method (c) called on happy path¶

    • Filed against: backend-engineer
    • Location: backend/app/ops/cascade_delete.py, function _build_wikilink_rewrites, lines 500-503
    • Description: _method_c_fulltext is called unconditionally regardless of whether method (a) or (b) already found all references. On the happy path (links-table hit), this issues an extra SQL query to enumerate all live wiki pages from the pages table. While no file read or full vault walk occurs in this case (because already_found filters all candidates), the extra DB query violates the ADR §3.1 gate: "(c) is skipped entirely when (a)∪(b) already covers the candidate set".
    • ADR reference: ADR-0026 §3.1 and §9 (architect-review gate bullet 2).
    • Test: T-CD-026 (test_method_c_not_called_on_links_table_hit) — xfail(strict).
    • Fix: Add an early-return guard before the _method_c_fulltext call. Simplest implementation: if all_found_ids and not b_results: return rewrites, match_methods_used (skip (c) when (a) found hits and (b) found nothing new). A more complete guard would check whether all live wiki pages enumerable by the pages table are a subset of all_found_ids, but the common case (links table is up-to-date) is captured by the simpler guard.
    • Risk: Without the guard, every cascade delete issues an extra paginated SQL query. With CASCADE_FULLTEXT_MAX_FILES=5000 this can be significant on large vaults.

    DEFECT-F13-002 — P1 — I5: frontmatter key order corrupted by PyYAML sort¶

    • Filed against: backend-engineer
    • Location: backend/app/ops/cascade_delete.py, all uses of frontmatter.dumps() in cascade_delete() (step 4, line 667) and _prune_sources() (line 808)
    • Description: python-frontmatter uses PyYAML's default Dumper which sorts mapping keys alphabetically. This means any file rewritten by F13 has its YAML frontmatter key order changed, regardless of the author's original ordering. Observed: title, type, sources, tags, created_at → created_at, sources, tags, title, type. Also: list items lose the 2-space indent (- item → - item). Both changes are semantically equivalent but create unnecessary diffs in Obsidian's git plugin, violate the user's formatting preferences, and break any key-order-dependent tooling.
    • ADR reference: ADR-0026 §4.3: "The frontmatter block is re-emitted byte-for-byte via frontmatter.dumps". The claim is false.
    • Test: T-CD-024b (test_frontmatter_key_order_preserved_xfail) — xfail(strict); T-CD-024 (test_frontmatter_integrity_after_rewrite) — PASS (semantic values correct).
    • Fix options (in preference order):
    • Switch to python-frontmatter with ruamel.yaml handler: frontmatter.loads(raw, handler=frontmatter.YAMLHandler()) using a ruamel.yaml CommentedMap preserves key order and list formatting.
    • Alternative: read the raw frontmatter block as a string, extract and replace only the sources: key value when pruning, leaving all other keys byte-identical (body-regex approach, limited to the specific key being mutated).
    • Last resort: after frontmatter.dumps(), compare with original and only write if body actually changed; accept the key-order change as a one-time cost per file touched by F13.
    • Affected paths: Every WikilinkRewrite file write (step 4) and every _prune_sources() call (step 5). Files that are NOT touched by F13 are unaffected.

    10. Re-verification (2026-06-29)¶

    Re-verifier: qa-test-engineer (claude-sonnet-4-6)

    Fix confirmation — DEFECT-F13-001 (I1: method (c) skipped on happy path)¶

    Location of fix: backend/app/ops/cascade_delete.py, _build_wikilink_rewrites, lines 526-528.

    Fix implemented: Added guard if not all_found_ids: before the _method_c_fulltext call. When method (a) and/or (b) finds at least one referencing page via the links index, all_found_ids is non-empty, the condition is False, and _method_c_fulltext is not called. When both (a) and (b) return nothing, all_found_ids is empty, the condition is True, and the full-text fallback fires as intended.

    Correctness verdict: The fix is production logic, not a test-only workaround. The guard is in the production function _build_wikilink_rewrites. Method (c) still fires on the no-links-hit path (empty links table) as required by ADR-0026 §3.1 last-resort semantics.

    Residual note — no-links-hit path test coverage: There is no dedicated test asserting that when (a) and (b) both return empty, method (c) does fire and its results are incorporated. The existing T-CD-026 proves the happy-path skip; the unhappy path is exercised indirectly by the session mock returning empty results on calls 4+ in several tests. This is a test-coverage gap (not a defect), filed as advisory: a future test test_method_c_fires_when_links_table_empty would complete coverage of ADR-0026 §3.1.

    Test T-CD-026 result: PASS (normal test, no xfail marker). method_c_called assertion confirms _method_c_fulltext was not called on the links-hit fixture.

    Fix confirmation — DEFECT-F13-002 (I5: frontmatter byte-identical after wikilink rewrite)¶

    Location of fix (two changes):

    1. backend/app/ops/cascade_delete.py, new helper _rewrite_body_preserving_frontmatter (lines 181-200): Splits the raw file on ---\n fences (max 2 splits), keeps the frontmatter block byte-for-byte as the middle part, runs _rewrite_body on the body only, and reassembles. No PyYAML round-trip whatsoever. Returns None if the body is unchanged (no write needed).

    2. backend/app/ops/cascade_delete.py, cascade_delete step 4 (line 689): Changed from frontmatter.loads/dumps round-trip to _rewrite_body_preserving_frontmatter(raw, plan.target_title or ""). The frontmatter block is now untouched for dead-wikilink rewrites.

    3. backend/app/ops/cascade_delete.py, _prune_sources (line 836): The sources[] mutation still uses frontmatter.dumps (a PyYAML round-trip is unavoidable when mutating a structured key), but now passes sort_keys=False to PyYAML so the existing key order is respected. Key order is preserved; only the sources value changes. This is the documented minimum-diff approach for intentional sources[] mutations.

    Correctness verdict: Both changes are production logic fixes. The wikilink rewrite path is now byte-identical in the frontmatter block. The _prune_sources path minimises diff to the sources[] value only (not key reordering). The ADR §4.3 claim "byte-for-byte via frontmatter.dumps" is now accurate for the rewrite path (via raw split, not PyYAML); _prune_sources intentionally mutates content so byte-identity is impossible but key order is now preserved with sort_keys=False.

    Test T-CD-024b result: PASS (normal test, no xfail marker). The raw frontmatter block extracted via split("---\n", maxsplit=2)[1] is byte-identical before and after the dead-wikilink rewrite.
