    US-F17 — InferenceProvider ABC and 3 backends¶

    Feature ID: F17 Story: As the Synapse ingest orchestrator, I want every inference call to go through a single InferenceProvider abstract base class with three concrete backends (Local/Ollama, API/Anthropic, CLI/claude-agent-sdk), selected at runtime from provider_config, so that no model name, API key, or endpoint URL ever appears outside the provider module files and the provider can be swapped per vault or per operation without touching orchestrator logic (invariant I6).

    Invariants touched: I6 (defining invariant — pluggable inference; no hardcoding anywhere), I7 (loops introduced by providers must be bounded — first real inference cost this sprint), I8 (D1 component diagram now must show the provider layer)

    AC-F17-1 — InferenceProvider ABC signature and mypy strict compliance¶

    Given backend/app/ingest/provider/base.py has been authored, When mypy --strict backend/app/ingest/provider/base.py is run, Then the command exits 0 with no errors, and the file defines an abstract class InferenceProvider that declares the following abstract methods with exact signatures:

    def analyze(self, source_text: str, vault_context: str) -> Analysis: ...
def generate(self, analysis: Analysis, retrieval_context: str) -> list[WikiPage]: ...
def chat(self, messages: list[Message], retrieval_context: str) -> AsyncIterator[str]: ...
def capabilities(self) -> ProviderCapabilities: ...

    where Analysis, WikiPage, Message, and ProviderCapabilities are Pydantic models defined in the same or an adjacent module, and all four methods are decorated with @abstractmethod.

    Testable assert:

    import inspect, abc
from app.ingest.provider.base import InferenceProvider
abstract_methods = {
    name for name, val in inspect.getmembers(InferenceProvider)
    if getattr(val, "__isabstractmethod__", False)
}
assert abstract_methods == {"analyze", "generate", "chat", "capabilities"}

    AC-F17-2 — All 3 providers implement the ABC; chat() raises NotImplementedError¶

    Given ollama.py, api.py, and cli.py each define a concrete class inheriting from InferenceProvider, When Python attempts to instantiate each class, Then no TypeError is raised (all abstract methods are implemented), and calling chat() on any of the three instances raises NotImplementedError immediately without making any external HTTP or subprocess call.

    The test must run against all three classes (parametrised or explicit); all three must pass.

    AC-F17-6 — Config resolution order: operation > vault > global¶

    Given the provider_config table contains three rows: - Row A: scope='global', vault_id=NULL, model_id='model-global' - Row B: scope='vault', vault_id=, model_id='model-vault' - Row C: scope='operation', vault_id=, operation='ingest', model_id='model-operation'

    When the config resolver is called with (vault_id=, operation='ingest'), Then it returns the config from Row C (operation scope wins).

    And when it is called with (vault_id=, operation=None), Then it returns the config from Row B (vault scope wins when no operation override).

    And when it is called with (vault_id=, operation=None), Then it returns the config from Row A (global fallback).

    Testable assert (unit test with 3 conflicting rows in a test DB):

    cfg = resolver.resolve(vault_id=test_vault_id, operation="ingest")
assert cfg.model_id == "model-operation"

cfg = resolver.resolve(vault_id=test_vault_id, operation=None)
assert cfg.model_id == "model-vault"

cfg = resolver.resolve(vault_id="unknown", operation=None)
assert cfg.model_id == "model-global"

    US-K2-INGEST — Orchestrated ingest loop with capability-aware routing¶

    Feature ID: K2 (ingest operation — full implementation) Story: As the Synapse ingest orchestrator, I want to resolve an InferenceProvider from provider_config, inspect its capabilities(), and route ingest to either the orchestrated analyze-generate-validate-retry loop (for Local/API providers) or a delegated agent loop (for CLI provider), so that wiki pages are produced by the most appropriate strategy for the selected provider without any conditional logic that inspects the provider's class name or type string (invariant I6).

    Invariants touched: I6 (routing reads capabilities(), never class name / isinstance), I7 (orchestrated loop is bounded: max_iter=3, token_budget=60000; cost logged per run), I1 (each page write is an incremental upsert, never a rescan), I5 (generated pages must pass Obsidian validity: YAML frontmatter present)

    US-F3 — Two-step CoT ingest, source traceability, language-aware¶

    Feature ID: F3 Story: As the Synapse ingest orchestrator, I want the orchestrated ingest loop to call analyze() first (producing a structured Analysis object identifying topics, entities, language, and suggested page types) and then generate() (producing wiki pages with wikilinks and source traceability in frontmatter), so that every generated wiki page can be traced back to its source document and is written in the detected language of that document.

    Invariants touched: I5 (generated pages must pass Obsidian YAML validity), I6 (analyze/generate called only through InferenceProvider ABC), I7 (validate-retry loop in orchestrator is bounded)

    Ambiguities and architect-escalation flags¶

    The following questions must be resolved by solution-architect before engineers begin implementation. Each is tagged with an AQ-v0.2 ID to distinguish from v0.1 AQs.

    AQ-v0.2-1 — Analysis re-run on retry: is analyze() called once or per iteration? (blocks AC-K2-5)¶

    The orchestrated loop is described as analyze → generate → validate → retry. It is ambiguous whether analyze() is called once (Analysis cached; only generate() is retried with augmented prompt) or called again on each retry iteration.

    Calling analyze once is cheaper (fewer tokens) but means the retry only augments the generation prompt. Calling analyze again is more expensive but allows the provider to re-examine the source with the validation error as context.

    Required action: architect to specify the retry strategy — single analyze or per-iteration analyze — and document in ADR-0007. AC-K2-5's assertion on analyze.call_count depends on this decision (either <= 1 or <= max_iter).

    AQ-v0.2-2 — Ingest run log: structured Python log or dedicated Postgres table? (blocks AC-K2-6)¶

    AC-K2-6 requires a log entry with provider_name, max_iter_used, total_tokens, total_cost_usd, converged. It is unspecified whether this is written as a structured JSON log line (Python structlog / logging.info(event="ingest_complete", ...)) or persisted to a Postgres ingest_runs table.

    A Postgres table enables dashboards and query-based cost analysis. A structured log is simpler but requires log aggregation to query. The test harness for AC-K2-6 differs significantly depending on the choice.

    Required action: architect to decide storage mechanism for ingest run records and document in ADR-0009 (or a new ADR). AC-K2-6 test implementation is blocked on this.

    AQ-v0.2-3 — Analysis and WikiPage Pydantic schemas: exact field definitions (blocks AC-F3-1, AC-F3-2)¶

    The Analysis and WikiPage Pydantic models are referenced in AC-F17-1, AC-F3-1, and AC-F3-2 but their exact field schemas are not defined in BACKLOG.md or the scope lock.

    Specifically unspecified: - Analysis.suggested_pages: list of what type? (SuggestedPage with title: str and type: PageType? Or a flat dict?) - WikiPage.frontmatter: typed dict or dict[str, Any]? - WikiPage.content: raw Markdown string (before or after frontmatter block stripped)? - Message type for chat(): simple {"role": str, "content": str} or full Anthropic Message type? - ProviderCapabilities: Pydantic model or TypedDict?

    Required action: architect or ai-agent-engineer to define the exact Pydantic schemas for Analysis, WikiPage, Message, ProviderCapabilities in a schema file (backend/app/ingest/schemas.py or backend/app/models.py) before provider implementations begin. These schemas are the contract between orchestrator and providers; any divergence breaks both.

    AQ-v0.2-4 — Token usage measurement per backend (blocks AC-K2-6, ADR-0009)¶

    total_cost_usd computation requires knowing input and output token counts. Each backend exposes this differently: - Anthropic API: response.usage.input_tokens + response.usage.output_tokens returned in the API response body. - Ollama: response["eval_count"] (output tokens) and response["prompt_eval_count"] (input tokens) in /api/chat response. - CLI (claude-agent-sdk): The SDK may return cost via metadata in its response object or not at all. If the SDK does not expose token usage, the service must set total_cost_usd = 0.00 and log a WARNING that cost could not be computed.

    Required action: ai-agent-engineer to confirm the exact SDK attributes for token usage in the claude-agent-sdk response, and document the cost computation logic in ADR-0009. The convention that "CLI cost = $0.00" (build-time credits, not runtime billing) must be explicitly stated and agreed by the architect.

    AQ-v0.2-5 — provider_config.scope column: how is "operation" scope parameterised? (blocks AC-F17-5, AC-F17-6)¶

    AC-F17-5 lists scope as values 'global', 'vault', 'operation'. For operation scope to be meaningful, there must be a way to identify the operation (ingest vs. chat vs. lint). The BACKLOG defines scope: global | vault | operation but does not define an operation column or enum. AC-F17-6 tests resolution with operation='ingest' but the schema as specified does not have an operation column — only scope and vault_id.

    Required action: architect to clarify the schema. Options: (a) Add an operation column (ingest, chat, lint, etc.) to provider_config. (b) Encode operation in the scope value itself (e.g., scope='operation:ingest'). (c) Use a separate operation text column nullable for non-operation scope rows. This decision directly affects the Alembic migration and the resolution query in AC-F17-6.

    AQ-v0.2-6 — MCP server transport: stdio-only, HTTP-only, or both simultaneously? (blocks AC-MCP-1, AC-MCP-8)¶

    The scope lock (v0.2-scope.md §2) states: "Server starts in stdio mode (for CliAgentProvider) and HTTP mode (for external clients)." Running both simultaneously requires either two process instances or a multiplexing server.

    FastMCP supports both transports but not always in the same process. The test client (AC-MCP-1 through AC-MCP-6) needs to know which transport to use for automated testing.

    Required action: architect to specify whether: (a) The MCP server starts in stdio mode only and HTTP is deferred to v0.4. (b) The MCP server supports HTTP in addition to stdio via FastMCP's HTTP transport option. (c) Two separate make targets: make mcp-stdio and make mcp-http. This decision affects make mcp, AC-MCP-1's startup assertion, and the CliAgentProvider subprocess setup.

    AQ-v0.2-7 — WikiPage validator: exact criteria for "invalid" to trigger retry (blocks AC-K2-5, AC-F3-7)¶

    The validator in the orchestrated loop decides whether generate() output is "valid" or "invalid". The BACKLOG states: "check each WikiPage has required frontmatter (type, title, sources[])." But the threshold for triggering a retry is not fully specified: - Does a page with an empty sources[] list (but the list key present) count as valid? - Does a page with a type value not in the allowed enum count as invalid? - What happens if only some pages in the returned list are invalid — is the whole batch retried or just the invalid pages?

    Required action: architect to define the validation schema precisely. Recommend: (a) Required non-null, non-empty: type (must be in enum), title (non-empty string), sources (non-empty list with at least the originating file path). (b) Partial batch failure: the whole batch is retried (simpler orchestrator logic). Document in the orchestrator's docstring and reference in ADR-0007. AC-K2-5 and AC-F3-7 test conditions depend on this exact definition.

    AQ-v0.2-8 — cost anomaly WARNING: at call site or in a separate monitoring hook? (blocks ADR-0009)¶

    The scope lock §6 defines a $1.00/run anomaly WARNING threshold. It is not specified where this check is performed: (a) In orchestrator.py at the end of every ingest run (inline check after computing total_cost_usd). (b) In a separate monitoring module called as a post-ingest hook. (c) In the provider itself before returning results.

    The test for this (asserting a WARNING is emitted when total_cost_usd > 1.00) depends on knowing where to capture the log output.

    Required action: architect to specify the check location. Recommend: orchestrator inline, after the cost log entry is written. AC-K2-6's test can then mock the cost computation to return 1.01 and assert a WARNING was emitted.
