    1. AQ decision table (the 7 questions)¶

    AQ P Blocks Question DECISION Where locked
    AQ-v0.5-1 P0 AC-F5-1 BM25 hybrid in Qdrant now, or dense-only first? Dense-only (bge-m3 cosine) in Phase 1. No sparse/BM25 index. The existing synapse_pages collection has a single dense vector config (qdrant_client.py); adding a named sparse vector is a collection-shape migration we do NOT take on the dependency-root feature.     3. Phase 1 — F5 retrieval pipeline (rag/retrieval.py)¶

    3.1 The data-model reality this design must respect¶

    Critical ground truth (drives the whole citation contract):

    • A pages row tracks a raw/sources/ source document, not a wiki/ page. pages.file_path is a raw/sources/... path; the Qdrant point (id == pages.id) embeds the raw source text (orchestrator.py text_for_embedding = raw_bytes).
    • The wiki/*.md files produced by write_wiki_page() are NOT separately rowed in pages. They carry sources: [...] frontmatter (F3 traceability) pointing back at the raw/sources/ paths.
    • Therefore the searchable/embeddable unit is the source page, and a [n] citation must resolve to something the user can act on. For Phase 1 the citation target is the pages row itself (the source-document page): id, a display title (frontmatter title or filename stem), and a url-friendly slug derived from it. This is honest — the answer is grounded in that indexed document — and it requires no new join table.

    Future enhancement (post-M5, reserved): index wiki/ pages as first-class searchable units so citations point at synthesized wiki pages rather than raw sources. That needs a page_kind discriminator on pages (or a parallel wiki_pages index) and is explicitly out of M5 scope. ADR-0022 §8 records this as the one accepted limitation.

    3.2 The four phases (in order, single pass — I7)¶

    retrieve(query, vault_id, context_window, k=8, expansion_depth=2) -> RetrievalContext

Phase 1 — TOKENIZED / VECTOR SEARCH  (I9: Qdrant + bge-m3 only)
  embedding = await get_embedding_client().embed(query)            # existing wrapper (I9)
  hits = qdrant.search(collection=synapse_pages, vector=embedding, # dense cosine top-k
                       limit=k, with_payload=True)
  seed_page_ids = [UUID(h.id) for h in hits]                       # point id == pages.id
  # AQ-v0.5-1: dense-only. No BM25/sparse. score = cosine similarity.

Phase 2 — GRAPH EXPANSION  (I2: read edges table; NEVER FA2; data_version unchanged)
  for depth in 1..expansion_depth:                                  # BFS, depth ≤ 2 (bounded, I7)
      neighbours = SELECT target/source FROM edges
                   WHERE (source_page_id IN frontier OR target_page_id IN frontier)
                   ORDER BY weight DESC                             # reuse 4-signal weight (F4)
      expanded += neighbours (dedup, keep highest edge weight as the expansion score)
  # also pull resolved links (links.target_page_id) for direct-link expansion.
  candidates = seed hits (vector score) ∪ expanded (edge-weight score)

Phase 3 — TOKEN-BUDGET ALLOCATION  (F14: 20% of context_window; AQ-v0.5-2 char/4)
  budget_tokens = int(context_window * 0.20)                       # the "retrieved" slice
  budget_chars  = budget_tokens * 4                                # char/4 safety cap
  rank candidates by combined score (vector seeds first, then edge-weight expansions)

Phase 4 — CONTEXT ASSEMBLY  (I3: server-side; build string + citation map ONCE)
  for each candidate in rank order, while budget remains:
      passage = load_passage(page)                                 # raw/sources file body, capped
      assign next 1-based citation index n
      append "[n] <title>\n<passage>\n" to context_text (whole-passage granularity)
      record Citation(n, PageRef(id, title, slug))
  drop lowest-ranked candidates that don't fit (AC-F5-4: never mid-sentence truncate-without-drop)
  return RetrievalContext(text, citations, query, ...)

    Bounding (I7). k (default 8), expansion_depth (default 2, hard max 2), and the char budget are all explicit caps. The expansion BFS is depth-bounded, not open-ended. No phase loops on a provider. There is no inference call in retrieve() at all — it is pure store reads + assembly.

    3.3 Passage source & granularity¶

    A "passage" in Phase 1 is the body of the source document for a candidate pages row, read from vault/raw/sources/<file_path> and capped per-passage (e.g. first budget_chars // k chars) so a single large source cannot consume the whole budget. Chunk-level passages (sub-document windows) are a post-M5 refinement; Phase 1 uses whole-document-capped passages to avoid a re-chunking/re-embedding project (I9 — minimize new machinery). Reading the file is an I1-safe targeted read (one file per candidate), never a vault walk.

    4. Phase 1 — data structures (the contract)¶

    All in backend/app/rag/retrieval.py, Pydantic v2 (parsed/serialized at the GET /search boundary and stored as JSONB in messages.citations):

    class PageRef(BaseModel):
    """A citable wiki/source page target. Resolves [n] → navigable page."""
    id: str            # str(uuid) of the pages row (== Qdrant point id, ADR-0002)
    title: str         # frontmatter title, else filename stem (never empty — falls back)
    slug: str          # url-friendly: slugify(title) (lower, hyphenated, ascii) — NOT NULL

class Citation(BaseModel):
    """One [n] marker → its source page + the matched passage's score."""
    n: int             # 1-based citation index, contiguous from 1
    ref: PageRef
    score: float       # vector cosine OR edge weight (provenance of the candidate)
    phase: Literal["vector", "expansion"]   # which phase surfaced it (audit/debug)

class RetrievalContext(BaseModel):
    """The assembled, budget-capped retrieval result. The ONE object F5 returns."""
    query: str
    text: str                      # assembled context string WITH inline [n] markers (≤ budget)
    citations: list[Citation]      # n → PageRef map; len == number of [n] markers
    token_budget: int              # the 20% slice used (for audit)
    approx_tokens: int             # char/4 estimate of `text` (≤ token_budget)
    data_version: int              # snapshot read before assembly (AC-F5-5 unchanged-after proof)

    4.1 The citation contract (load-bearing — F5 ↔ F6 ↔ frontend)¶

    1. What gets stored. On a successful chat turn whose RetrievalContext.citations is non-empty, chat/stream.py writes that list (serialized) into the assistant messages.citations JSONB column (already exists, reserved by ADR-0019). Each stored element is {n, ref:{id,title,slug}, score, phase}.

    2. What the chat stream emits. The done NDJSON event (ADR-0019 §2.2) gains an additive citations field: "citations": [{"n":1,"id":"...","title":"...","slug":"..."}, ...] (the compact projection — score/phase are stored but not streamed). Existing clients ignoring the field are unaffected (additive, non-breaking). The streamed assistant text contains the literal [n] markers the provider produced (the provider is instructed, via the retrieval_context preamble, to cite using the [n] markers present in the context — see §6.1).

    3. How the frontend resolves [n] → page. ChatMessage.tsx reads the message's citations array, builds an n → {title, slug} map, and post-processes the already-parsed-once settled markdown (I3 — parse stays single, ADR-0019 §2.6): a small regex replaces each [n] text token with a <sup role="link" title="{title}">[n]</sup> whose hover shows the page title and whose click navigates to the page (slug). This runs once on the settled message (memoized on raw+citations), never per streaming token (I3 preserved; G3 untouched).

    4. Consistency rule. len(citations) == count of distinct [n] in text. The assembler is the single authority: it emits [n] into text and the matching Citation together, so they cannot drift.

    4.3 slug is derived, not stored¶

    slug = slugify(title or file_stem) computed in rag/retrieval.py (a small pure helper). No new pages.slug column — adding one would require backfilling every row and a migration for a value we can derive deterministically. This keeps F5 migration-free (§2.1).

    5. Phase 1 — integration: chat & providers consume the context¶

    5.1 Chat (F6) wiring¶

    chat/stream.py today calls build_chat_context(...) → a plain string passed as retrieval_context. Phase 1 introduces a retrieval-aware context builder that:

    1. extracts the latest user message as the query,
    2. calls await retrieve(query, vault_id=..., context_window=window),
    3. prepends the light file context (purpose.md + overview.md, the existing build_chat_context output — kept as grounding header) to RetrievalContext.text, producing the final retrieval_context string handed to provider.chat(messages, retrieval_context),
    4. carries the RetrievalContext.citations out of band so the done writer can stamp messages.citations and the done event (§4.1.1–2).

    The provider chat() signature is unchanged ((messages, retrieval_context: str)) — I6 contract held. Citations travel beside the stream in the orchestration layer, not through the provider.

    I3 / G3 unchanged: retrieval runs once, before streaming starts (it is not per-token). The stream loop, the <think> scanner, and the parse-once-on-done rule are all untouched.

    5.2 The 60/20/5/15 budget (F14) — where retrieval sits¶

    Slice % Content
    Conversation history 60 prior turns (existing truncate-oldest, ADR-0019)
    Retrieved context 20 F5 RetrievalContext.text (this sprint fills it for real)
    System/grounding 5 preamble + purpose.md + overview.md header
    Answer headroom 15 reserved for the model's output

    F5 owns the 20% slice. retrieve() is handed context_window and computes its own 0.20 * window char/4 cap internally (AQ-v0.5-2). The 5% grounding header is the existing build_chat_context.

    5.3 GET /search endpoint (main.py)¶

    A thin handler: validate q, call retrieve(), return the §4.2 shape. Read-only; no data_version bump (AC-F5-5). Reuses the existing DB session + Qdrant + embedding singletons.

    6. Phases 2–4 interface stubs (coherence only — each has its own ADR)¶

    6.1 Provider citation instruction (shared, lands in Phase 1)¶

    The retrieval_context string is built with a one-line preamble instructing the model: "Cite sources using the [n] markers shown beside each passage; do not invent citation numbers." This is backend-neutral text (I6) — it works identically for Ollama/API/CLI. The model echoes [n] into its answer; the assembler's citations list is the authority for resolution.

    6.2 F9 review (Phase 3) — interface¶

    enqueue_review() is called fire-and-forget from run_ingest_pipeline after each write_wiki_page. generate_review_queries() makes one bounded InferenceProvider call (I6/I7); on timeout/error it returns None and the item is still enqueued (AC-F9-4). The deep-research action calls F10's POST /research/start and stores run_id on the row (AC-F10-5).

    6.3 F10 deep research (Phase 2) — bounds live on the run row¶

    run_deep_research() freezes {max_iter, token_budget, max_queries_per_iter=5, concurrency=3} at start (AQ-v0.5-4) into deep_research_runs. The loop: gen queries → SearXNG (semaphore=3) → fetch→md → assess (provider) → [sufficient → synthesize → ingest_file (AQ-v0.5-3) | else refine, while iter < max_iter]. On exhaustion: status max_iter_reached, no further provider calls. total_cost_usd logged (I7). SearXNG only via SEARXNG_URL (I9); static import guard bans tavily/ddg/google.

    6.4 F9 schema (Phase 3) — review_items (Alembic 0010 — F10 takes 0009, ADR-0024 §7)¶

    id UUID PK · vault_id String NOT NULL (AQ-v0.5-6) · page_id UUID FK→pages nullable · item_type enum(new_page|update_page|deep_research_candidate) · status enum(pending|approved|skipped|deep_researched) · pre_generated_query Text nullable · deep_research_run_id UUID nullable · created_at/reviewed_at/reviewed_by.

    6.5 F13 cascade (Phase 4) — single pass, Obsidian-safe, with mandatory dry-run¶

    ADR-0026 is the detailed design. Plan/apply are split at the function and REST boundary so the dry-run is mandatory (this deletes user data).

    plan_cascade_delete(page_id) (READ-ONLY, backs POST /pages/{id}/cascade-delete/preview): locate page → 3-method ref find (exact links title/target_page_id / slug-normalised / bounded full-text fallback, each logged) → partition sources[] into will_delete (empties) vs will_preserve_with_pruned_source (non-empties) → compute wikilinks_to_rewrite from the links back-reference index → collect shared_entity_warnings (source-overlap via edges, warn not block, AC-F13-3) → return CascadePlan without mutating anything.

    cascade_delete(page_id) (APPLY, single pass, backs DELETE /pages/{id}): compute the plan → soft-delete (deleted_at) the will_delete pages → delete_point() each → rewrite [[Target]]→Target ([[T|alias]]→alias) body-only via frontmatter.loads/dumps in only the referencing wiki/*.md files (targeted writes, I1) → re-persist_links those files + set residual links.dangling=True → prune X from the preserved pages' sources[] (frontmatter-safe) + pages.sources JSONB → delete touching edges rows → update_index() once (K3 entry auto-removed from the live read) → delete the originating raw/sources/ file (AQ-v0.5-5) → bump_version() EXACTLY ONCE + notify_bump() (debounced FA2, no forced run, I2). Wikilink replacement is body-only and never touches YAML frontmatter (I5); test_obsidian_check.py 15/15 is the gate. Migration-free (no D2 change). DELETE /pages/{id} 404s on a non-existent or already-soft-deleted id (idempotent double-delete, AC-F13-5c).

    7. Phase 1 — CliAgentProvider.chat() design (F17 carry-forward)¶

    Today cli.py chat() raises NotImplementedError (honest M4 stub). Phase 1 implements it as a delegated streaming chat turn via claude-agent-sdk, mirroring delegate_ingest():

    • Signature unchanged (I6): async def chat(self, messages, retrieval_context) -> AsyncIterator[str]. Returns an async generator of text deltas — the same shape run_chat_stream._consume() already consumes for Ollama/API (stream.py supports both coroutine-returning and async-gen providers).
    • Context injection: retrieval_context (the assembled F5 string incl. [n] markers + grounding header) is injected as the system prompt / leading context to the SDK session (read-only chat — the agent is NOT granted write_page; chat does not mutate the vault). Conversation messages map to SDK turns.
    • Bounding (I7, AQ-v0.5-7): three caps, no new DB column — (a) token_budget from the resolved chat config row (already flows through run_chat_stream), (b) timeout_seconds (F16 60s, enforced by run_chat_stream's asyncio.wait_for — unchanged), (c) CHAT_AGENT_MAX_TURNS env (default 8) caps SDK agent turns so a tool-looping agent halts. The loop terminates on first complete response or any cap (AC-F17-CHAT-1d).
    • Streaming: iterate client.receive_response(); yield assistant text blocks as {type:"token"} deltas (the existing <think> scanner in stream.py handles any <think> spans — no change there).
    • Cost (AC-F17-CHAT-3, NB-4): reuse _extract_sdk_cost/_merge_sdk_usage — SDK total_cost_usd when billed via API key, else Decimal("0.00") with a WARNING. Pushed via _record_usage so run_chat_stream logs/returns it like any other backend.
    • Lazy import + no-key path: the SDK is imported lazily (as in delegate_ingest); with no ANTHROPIC_API_KEY the method raises a clean configuration error before streaming (maps to the pre-stream error path) — never a fake stream. Dev default remains Ollama (ADR-0019); CLI chat is exercised via a mocked SDK in test_cli_chat.py (AC-F17-CHAT-1..3).

    9. Flagged tensions & risks¶

    1. Citation target = source page, not wiki page (accepted limitation). Because pages index raw/sources/ documents (§3.1), a [n] resolves to the source document, not the synthesized wiki/ page. This is correct grounding (the answer came from that indexed text) and honest, but a user clicking [n] lands on the source, not necessarily a polished wiki page. Indexing wiki/ pages as first-class searchable units is the clean fix — explicitly deferred post-M5 (needs a page_kind discriminator). Recorded in ADR-0022 §8. Not an invariant violation; a scope boundary.

    2. Dense-only retrieval quality (AQ-v0.5-1). Without BM25, exact-keyword/rare-token queries may under-retrieve vs. a hybrid. Mitigation: graph-expansion (Phase 2) widens recall via the edges table; bge-m3 is strong dense. Hybrid is reserved as a measured post-M5 enhancement, not a guess.

    3. char/4 budget under-fill (AQ-v0.5-2). A conservative char/4 cap may leave the 20% slice slightly under-filled for token-dense scripts (CJK). Accepted: under-fill is the safe direction (never blow the window); exactness is not required for a drop-lowest-ranked cap.

    4. I7 across F10 (Phase 2) is the headline risk. Deep research is the most loop-intensive feature in the roadmap. The bounds are frozen-at-start, per-run, with a hardcoded concurrency ceiling and a mandatory max_iter_reached test (AC-F10-2, EC-M5-5). Any PR weakening a bound is a P0 block — architect review (EC-M5-19) is mandatory before Phase 2 exit. Locked in ADR-0024.

    5. F13 vault integrity (Phase 4) is the headline I1/I5 risk. Targeted-edits-only (references found from the links back-reference index, not a vault walk) and frontmatter-safe wikilink cleanup (via the frontmatter.loads/dumps round-trip) are the gates; the data_version bump must be exactly +1 with no forced FA2 (AC-F13-4); the preserve-shared rule deletes a page iff its sources[] empties; a mandatory read-only dry-run (POST /pages/{id}/cascade-delete/preview) precedes the destructive DELETE /pages/{id}. Migration-free. Locked in ADR-0026. Placed last (vault stable) per the PM phase plan.
