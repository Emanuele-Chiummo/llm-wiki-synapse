    Sprint 2 — v0.2 — Scope Lock¶

    Milestone: M2 — "Agentic loop closed, 3 providers" Branch: sprint/v0.2 Locked by: product-manager on 2026-06-28 Status: LOCKED — no changes without PM + orchestrator approval Prerequisite: M1 CLOSED (v0.1-m1-closure.md — live demo passed, all EC items confirmed)

    1. Sprint objective¶

    Close the agentic inference loop end-to-end: an InferenceProvider is resolved from provider_config, capability-aware routing dispatches ingest to either the orchestrated loop (Local/API) or a delegated agent (CLI), and the wiki pages produced are schema-valid and written to vault/wiki/. The MCP standalone server becomes the programmatic interface used by the CLI provider during its own agent loop. Sequence diagrams (D3) and the first formally-required ADRs (D7) document every decision made this sprint.

    v0.2 makes the FIRST real LLM inference calls in Synapse. Cost tracking (I7) is therefore a first-class acceptance gate — not a nice-to-have.

    K3 / K5 scope decision¶

    Both K3 and K5 are FULLY in scope for v0.2, not partially.

    Rationale: K3 is a direct output requirement of F3 — the ingest loop must write to index.md. Deferring it would leave a core Karpathy feature (the LLM navigation entry-point) as a stub while the loop runs. K5 is required to persist the wikilinks the generate step produces, and deferring the links table to v0.3 would require an Alembic migration mid-graph sprint. Bringing both in now is the correct call for a solo evening dev — one schema change event, not two.

    4. Invariants in force — with emphasis on v0.2 firsts¶

    All 9 invariants apply. The two that are first-class acceptance gates this sprint:

    I6 — PLUGGABLE INFERENCE (the whole point of v0.2)¶

    Every inference call — analyze(), generate(), chat() (stubbed) — MUST go through InferenceProvider. No model name, no HTTP endpoint, no API key appears outside the provider module files (ollama.py, api.py, cli.py). The orchestrator knows only the ABC interface and capabilities() output.

    Enforcement: static analysis test asserting zero direct Ollama/Anthropic/SDK imports outside backend/app/ingest/provider/. Architect sign-off required. Any PR that hardcodes a provider or bypasses the ABC is rejected without discussion.

    I7 — BOUNDED LOOPS (first real loops exist in v0.2)¶

    Every loop introduced this sprint (orchestrated ingest refinement, provider fallback) MUST have: - A max_iter cap (see §6 for defaults) - A token_budget cap (see §6 for defaults) - A total_cost_usd log entry on completion (even if $0.00 for Local)

    The provider fallback loop is bound to exactly 1 retry (primary fail/timeout → fallback once → surface error). No recursive fallback chains.

    pytest must include a test that mocks a non-converging provider (always returns invalid pages) and asserts the loop stops at max_iter without overrun.

    Remaining invariants — status in v0.2¶

    Invariant Status in v0.2
    I1 — Incremental index only Inherited from v0.1; v0.2 must not break it. The orchestrator's write_page primitive writes to wiki/ only, never rescans.
    I2 — Graph layout server-side Not exercised yet (F4 is v0.3). No graph code introduced. The links table added in K5 is data-only.
    I3 — No per-token heavy work Not exercised (no chat UI in v0.2). The chat() stub returns a simple stream placeholder.
    I4 — CodeMirror 6 only Not exercised (no frontend in v0.2).
    I5 — Obsidian compatibility All wiki pages written by the loop must have valid YAML frontmatter (type, title, sources[]). Run the AC-K7-3 manual open check again after v0.2 ingest produces pages.
    I8 — Docs-as-DoD D3 + D7 are first-required this sprint. D1 + D2 + D4 must be updated. All required before M2 sign-off.
    I9 — Do not reinvent OllamaProvider reuses the already-running Ollama instance. No new inference containers. SearXNG NOT used in v0.2 (F10 deferred). bge-m3 embeddings unchanged.

    5. chat() method scope decision¶

    DECISION: chat() is IMPLEMENTED AS A STUB on all 3 backends in v0.2. It is NOT skipped from the ABC.

    Rationale: - The ABC signature chat(messages, retrieval_context) → stream is a structural part of F17 and must be locked in v0.2 so the interface is complete and type-checked. - Omitting chat() from the ABC now means a breaking change at v0.4 when it is needed. - A stub that raises NotImplementedError at call time is sufficient and keeps the ABC complete. No chat endpoint, no chat UI, no conversation logic in v0.2. - The stub is tested: a unit test asserts the method exists on all 3 concrete providers and raises NotImplementedError when called (confirming it is wired but intentionally incomplete, not accidentally missing).

    The full chat implementation (F6, streaming, cited-refs, conversation persistence) is v0.4.

    6. Cost tracking and bounded-loop defaults (I7 compliance)¶

    v0.1 cost: $0.00 (no LLM calls). v0.2 introduces real inference spend. I7 requires total_cost_usd logged per run and per bounded loop.

    Defaults (proposal — architect must ratify in ADR-0009)¶

    Parameter Default value Rationale
    max_iter (orchestrated ingest loop) 3 Analyze → generate → validate: 1 pass is the happy path. 2 retries covers malformed JSON or missing frontmatter fields. Beyond 3 is a provider quality problem, not a retry problem.
    token_budget (per-run, orchestrated loop) 60 000 tokens Covers a typical source doc analysis (5–10K) + generation of 3–5 wiki pages (20–30K) + 2 retries (×2 = headroom). Well within Anthropic API context windows. Tunable via provider_config.
    token_budget (CLI / delegated) 100 000 tokens CLI provider runs its own agent loop; higher headroom appropriate for agentic browsing + writing.
    provider_fallback_max 1 Primary fails → try fallback provider once → surface error. No chained fallbacks.
    total_cost_usd computation (input_tokens × input_price + output_tokens × output_price) per model from provider_config; Local (Ollama) = $0.00 always Price per model stored in provider_config; updated by user. Default seeded for current model IDs per CLAUDE.md §12.
    Cost anomaly flag threshold $1.00 per single ingest run Log WARNING if total_cost_usd > 1.00 for a single document ingest.

    These defaults are configurable via provider_config.max_iter and provider_config.token_budget (per-vault or per-operation override). The hardcoded fallback is that if provider_config is absent, the values above are used.

    Acceptance criterion for I7¶

    AC-I7-1: Every call to orchestrator.ingest_file() produces a log entry with keys: provider_name, max_iter_used, total_tokens, total_cost_usd, converged (bool).

    AC-I7-2: A pytest test mocks a provider whose generate() always returns pages with missing required frontmatter. Assert the loop stops after exactly max_iter=3 iterations with converged=False. Assert no further provider calls are made after the cap.

    AC-I7-3: For the CLI provider (delegated path), the token_budget is passed to the claude-agent-sdk invocation and the run is aborted if the budget is exceeded. Log the actual tokens used and total_cost_usd = 0.00 (CLI provider uses build-time agent credits, not runtime billing — document in ADR-0009).

    7. M2 Exit Criteria (concrete, checkable)¶

    All items must be MET before PM sign-off and before advancing to Sprint 3.

    EC-M2-1 (F17 — ABC + 3 backends): Provider layer complete¶

    • backend/app/ingest/provider/base.py defines InferenceProvider ABC with methods: analyze(), generate(), chat() (stub), capabilities(). All methods type-annotated (mypy strict). Abstract methods raise NotImplementedError in any non-implementing class.
    • ollama.py, api.py, cli.py each implement the full ABC (chat() stub included).
    • Static analysis test: zero direct Ollama/Anthropic/SDK imports outside provider/ directory.
    • Unit test: capabilities() returns correct {mode, supports_tools, supports_agentic_loop, max_context, name} dict for each provider; OllamaProvider.supports_agentic_loop == False; CliAgentProvider.supports_agentic_loop == True.

    EC-M2-2 (F17 — provider_config): Config table and resolution¶

    • Postgres provider_config table exists (Alembic migration); columns: id, scope (global/vault/operation), vault_id (nullable), provider_type (local/api/cli), model_id, base_url (nullable), max_iter, token_budget, created_at, updated_at.
    • GET /provider/config and POST /provider/config endpoints operational and documented in OpenAPI.
    • Config resolution order: operation-scope > vault-scope > global-scope.
    • Unit test: resolution order verified with 3 conflicting rows in the test DB.

    EC-M2-3 (F17 — capability routing): Correct branch taken¶

    • Integration test with OllamaProvider: supports_agentic_loop == False → orchestrator runs the analyze → generate → validate → retry loop.
    • Integration test with CliAgentProvider (mocked): supports_agentic_loop == True → orchestrator delegates; the CLI agent's ingest is invoked without the orchestrated loop body executing.
    • Static assertion: the routing branch reads capabilities() output — it does NOT check provider type string or class name.

    EC-M2-4 (I7 — bounded loop): Loop stops at max_iter¶

    • pytest: mock provider always returns pages with invalid frontmatter (missing type field).
    • Assert: loop runs exactly 3 times (= max_iter default) then exits with converged=False.
    • Assert: no provider call is made after iteration 3.
    • Assert: log entry for the run includes total_cost_usd (any value, including 0.00) and converged=False.

    EC-M2-5 (F3 — 3-provider smoke matrix): Each provider ingests a fixture¶

    Using a canonical fixture file (tests/fixtures/sample-source.md, a ~500-word Markdown document with valid frontmatter):

    Provider Ingest outcome Pass condition
    Local (OllamaProvider) analyze() + generate() run; pages written to wiki/ At least 1 page in Postgres with sources[] populated; valid YAML frontmatter; [[wikilinks]] parseable by K5 parser
    API (ApiProvider — Anthropic) Same Same
    CLI (CliAgentProvider) Delegated to claude-agent-sdk agent At least 1 page in wiki/; page passes schema validation; index.md updated

    All 3 providers must pass this smoke matrix on live infrastructure (TrueNAS / API key / claude-agent-sdk). This is a live-infra test, not a mocked test. Human checkpoint at M2 confirms the matrix was run.

    EC-M2-6 (F3 — content quality): Generated pages are schema-valid¶

    • All pages written to vault/wiki/ have YAML frontmatter with type, title, sources[] present and non-empty (I5 + K6).
    • At least one [[wikilink]] per generated page (except single-topic sources may have zero — acceptable; warn in log, do not error).
    • The K5 parser extracts all wikilinks from generated pages without exception.
    • vault/wiki/index.md is updated with the new page entry after ingest.
    • vault/wiki/overview.md is created (or updated) after first ingest into the vault.
    • vault/wiki/ opens in Obsidian without errors after v0.2 ingest (human checkpoint, re-runs AC-K7-3 with actual LLM-generated content).

    EC-M2-7 (MCP server): Tools exposed and callable¶

    • backend/app/mcp/server.py is a FastMCP standalone server.
    • Tools registered: search_wiki, write_page, get_page, list_pages.
    • MCP server starts as a separate process / via make mcp and exposes tools via stdio (for CliAgentProvider) and HTTP (for future clients).
    • Integration test: call each of the 4 MCP tools via the FastMCP test client; assert correct return types and Postgres/filesystem state changes.
    • MCP tool schemas documented in docs/api/mcp-tools.json.

    EC-M2-8 (K5 — wikilink parser): Links stored in Postgres¶

    • backend/app/ingest/parser/wikilinks.py parses [[Target Page|alias]] and [[Target Page]] syntax from generated wiki pages.
    • Postgres links table (Alembic migration): columns: id, source_page_id, target_title, alias (nullable), created_at.
    • After ingest, all wikilinks in generated pages have corresponding rows in the links table.
    • Dangling wikilinks (target page does not exist yet) are stored but flagged (dangling=True column); they do NOT cause an error.
    • Unit tests for parser: valid link, aliased link, multiple links, no links, nested brackets.

    EC-M2-9 (K3 — index.md): Catalogue auto-updated¶

    • After every successful page write, vault/wiki/index.md is updated with an entry for the new page (title + wikilink + type).
    • index.md retains valid YAML frontmatter (I5).
    • If index.md is missing, it is recreated (not an error).
    • Unit test: ingest fixture → assert index.md contains wikilink to new page.
