    Sprint 5 — v0.5 — PM Scope Lock¶

    Milestone: M5 — "Feature parity core" Author: product-manager (claude-sonnet-4-6) Date: 2026-06-29 Gate: Sprint start — formal scope confirmation Branch: sprint/v0.5 (cut from sprint/v0.4 after EC-M4-HCP cleared) Prerequisite: M4 CLOSED — docs/sprints/v0.4-m4-closure.md (EC-M4-HCP cleared 2026-06-29; v0.4.0 tagged and released; all 4 gate sign-offs on record) Inputs: CLAUDE.md §3/§4/§7/§8/§9, BACKLOG.md §Sprint 5, docs/sprints/v0.4-m4-closure.md

    1. Sprint Objective¶

    Deliver "feature parity core" — the set of features that transform Synapse from a usable prototype into a system that can handle real-world research workflows. The headline theme is depth over breadth: F5 retrieval closes the chat-quality loop; F10 Deep Research adds autonomous web-sourcing; F9 HITL review surfaces AI outputs for human curation; F12 multi-format ingest removes the .txt/.md constraint; F13 cascade deletion makes the vault self-healing.

    The headline invariant for M5 is I7 (bounded loops). F10 Deep Research is the most loop-intensive feature in the entire roadmap — its multi-query SearXNG loop has explicit max_iter and token_budget caps, an assess-sufficiency gate before each query, and a hardcoded concurrency ceiling of 3. Any PR that introduces an unbounded loop anywhere in M5 work is a P0 blocker.

    The headline reuse rule for M5 is I9: SearXNG is the only permitted web-search backend. Tavily must never be introduced. Qdrant + bge-m3 are the embedding stack. The existing OllamaProvider / ApiProvider / CliAgentProvider must not be bypassed — all new inference calls go through InferenceProvider.

    4. Acceptance Criteria Per Feature¶

    F5 — 4-phase RAG retrieval¶

    • AC-F5-1: backend/app/rag/retrieval.py implements all 4 phases in sequence: (1) tokenized keyword search over Qdrant (bge-m3 vectors + BM25 hybrid if available); (2) graph-expansion: for each top-k result, follow edges in the graph (links + shared-source) up to depth 1 or 2; (3) token-budget allocation per F14 20%-retrieved-context slot; (4) context assembly — passages ranked by relevance, truncated to budget, returned as RetrievalContext with page refs.
    • AC-F5-2: Each retrieved passage carries a citation index n (1-based). The assembled context string includes [n] markers inline. The RetrievalContext object maps each n to a PageRef (id, title, url-friendly slug).
    • AC-F5-3: The retrieval pipeline calls only Qdrant (bge-m3) and reads from the Postgres links and edges tables — no new embedding service introduced (I9). Static import check confirms no direct bge-m3 or Ollama embedding calls outside the existing Qdrant wrapper.
    • AC-F5-4: The token budget for retrieved context (20% of configured window per F14) is respected. If candidate passages exceed the budget, the lowest-ranked passages are dropped. Unit test: 10 fixture passages totaling 2× budget → assert assembled context ≤ budget.
    • AC-F5-5: Graph-expansion phase uses the existing edges table (Postgres) — does NOT trigger a graph recompute or FA2 run (I2 preserved). Verified: data_version is unchanged after a retrieval call.
    • AC-F5-6: GET /search?q=<query>&vault_id=<id> endpoint returns the top retrieval results with citation indices. Response documented in openapi.json (D4 zero-drift gate).
    • AC-F5-7: Unit tests cover: 0-result query, single-result, multi-page expansion, budget overflow truncation. Integration test: seed 5 pages with known wikilinks → retrieval returns linked pages in expansion phase.
    • AC-F5-8: After F5 ships, the chat endpoint passes the assembled RetrievalContext to the active provider's chat() call so that [n] markers appear in assistant messages (unblocks F6 AC-F6-3 and F17 CliAgentProvider.chat()).

    F6 (AC-F6-3 + AC-F6-5) — Citations and save-to-wiki¶

    • AC-F6-3: (Carry-forward unblocked by F5) Each assistant message in the chat UI displays [n] citation markers that link to source pages. Citation data is stored with the message in the messages table (JSONB citations field). On render, [n] is displayed as a clickable superscript that shows the page title on hover.
    • AC-F6-5: (Carry-forward) The "save to wiki" button on an assistant message is enabled and wired. Clicking it calls POST /ingest with the message content as source text. On 202, an inline status shows the resulting page title and wikilink. vitest test: mock POST /ingest → assert status rendered.

    F17 — CliAgentProvider.chat()¶

    • AC-F17-CHAT-1: CliAgentProvider.chat(messages, retrieval_context) is implemented (not NotImplementedError). The method delegates the chat turn to the claude-agent-sdk subprocess with the assembled RetrievalContext injected as a system tool or context message. The agent loop is bounded by max_iter and token_budget per I7.
    • AC-F17-CHAT-2: The streaming interface is consistent with OllamaProvider and ApiProvider: the method yields token chunks; the chat endpoint SSE stream is unchanged.
    • AC-F17-CHAT-3: Unit test: mock claude-agent-sdk output → verify stream chunks emitted, total_cost_usd logged correctly (SDK metadata or $0 convention per NB-4).

    F9 — Async HITL review queue¶

    • AC-F9-1: backend/app/ops/review.py implements an async review queue backed by a new Postgres table review_items. Columns: id (uuid), vault_id (uuid FK), page_id (uuid FK → pages, nullable), item_type (enum: new_page / update_page / deep_research_candidate), status (enum: pending / approved / skipped / deep_researched), pre_generated_query (text nullable), created_at, reviewed_at (nullable), reviewed_by (text nullable).
    • AC-F9-2: The ingest pipeline enqueues a review_item for each newly generated wiki page (item_type=new_page). This is non-blocking: the page is written to vault/wiki/ immediately (existing behaviour); the review queue entry is advisory.
    • AC-F9-3: REST endpoints:
    • GET /review/queue — list pending items (paginated, vault_id filter);
    • POST /review/queue/{id}/approve — sets status=approved, reviewed_at=now();
    • POST /review/queue/{id}/skip — sets status=skipped;
    • POST /review/queue/{id}/deep-research — sets status=deep_researched and triggers F10 deep research for the item's pre_generated_query (or the page topic if query absent). All endpoints documented in openapi.json (D4 zero-drift gate).
    • AC-F9-4: Pre-generated queries: when a review item is created, the ingest pipeline calls the active InferenceProvider to generate 1–3 research questions for the new page. These are stored in pre_generated_query (newline-separated). Bounded: max 1 generation call per item; if it fails or times out, query is left null (I7). The generation call goes through InferenceProvider (I6).
    • AC-F9-5: Review queue UI: a "Review" nav section (previously showing "coming in M5" placeholder) is activated. It renders the pending queue as a list with: page title / item type / pre-generated query (if present) / action buttons (Approve, Skip, Deep-Research). List is virtualized (TanStack Virtual) if >50 items (I4).
    • AC-F9-6: The review queue is NOT a blocker for normal operation. Pages are already written to vault/wiki/ when they appear in the queue. Approve = human confirmation only; does not re-trigger ingest. Skip = remove from queue. Deep-Research = trigger F10 and remove from queue.
    • AC-F9-7: Boundary from M4 Ingest View: the F1-INGEST-VIEW (ingest runs list + Run Ingest button) is not modified. F9 review queue is a separate section on the nav rail, not a modification of the Ingest section.
    • AC-F9-8: Integration test: seed 2 review_items; call approve on 1, skip on 1; assert both statuses updated; assert deep-research endpoint triggers F10 stub call.

    F10 — Deep Research loop¶

    • AC-F10-1: backend/app/ops/deep_research.py implements the full loop: (1) generate 2–5 SearXNG search queries from the research topic (via InferenceProvider); (2) execute queries against SearXNG (I9: never any other search backend); concurrency=3 (asyncio gather with semaphore); (3) fetch and parse result pages (html → markdown); (4) assess sufficiency: call InferenceProvider to judge whether collected content answers the research topic; if yes → proceed to synthesize; if no → generate follow-up queries (up to max_iter); (5) synthesize: call InferenceProvider to produce a synthesis document; (6) auto-ingest: feed synthesis into the ingest pipeline (not directly into vault/wiki/; must go through the orchestrated loop to satisfy I6).
    • AC-F10-2 (I7 — headline invariant): Every deep research run is strictly bounded: max_iter (configurable, default 3 refinement rounds), token_budget (configurable, default 100K tokens), max_queries_per_iter (default 5), concurrency (hardcoded 3, not configurable beyond 3 without architect approval). total_cost_usd logged per run in a new deep_research_runs table. Any PR that removes or weakens these bounds is a P0 blocker.
    • AC-F10-3 (I9): SearXNG is the only web search backend. The SearXNG base URL is read from SEARXNG_URL env var (already in the stack). No Tavily, DuckDuckGo, or Google API may be introduced. Static import test: no tavily, ddg, or google package imports anywhere in the codebase.
    • AC-F10-4: The deep research run is triggered via:
    • POST /research/start — body: {vault_id, topic, max_iter?, token_budget?}; returns HTTP 202 with {run_id: uuid};
    • GET /research/runs — list of past runs with status + cost;
    • GET /research/runs/{id} — details of a run including queries used, sources fetched, synthesis text. All endpoints in openapi.json (D4 zero-drift gate).
    • AC-F10-5: F9 integration: when F9's "Deep-Research" action fires, it calls POST /research/start with the pre_generated_query as topic. The run_id is stored on the review_item row for traceability.
    • AC-F10-6: New Postgres tables: deep_research_runs (id, vault_id, topic, status, queries used JSONB, sources_fetched int, total_cost_usd, started_at, completed_at, error_message) and deep_research_sources (id, run_id, url, title, fetched_content_md, relevance_score). Alembic migration. D2 regenerated via make er.
    • AC-F10-7: Unit tests: mock SearXNG returning 3 results → assert 3 fetch calls; mock InferenceProvider returning "sufficient" on first assess → assert loop stops after 1 round; mock InferenceProvider returning "insufficient" for max_iter rounds → assert loop stops at max_iter, status="max_iter_reached".
    • AC-F10-8: Playwright E2E: navigate to "Deep Search" nav section (previously placeholder); enter a topic; assert POST /research/start fires; assert run appears in runs list.

    F12 — Multi-format ingest¶

    • AC-F12-1: The backend ingest pipeline accepts the following additional file types at POST /ingest/upload (and via the watcher on vault/raw/sources/):
    • PDF (.pdf): text extracted via pypdf; images in PDF are skipped with a log warning.
    • DOCX (.docx): text + basic structure via python-docx.
    • PPTX (.pptx): slide text via python-pptx (one logical document per presentation).
    • XLSX (.xlsx): cell text (sheet name + row data as markdown table) via openpyxl.
    • Images (.png, .jpg, .jpeg, .gif, .webp): OCR via unstructured (if available) or "image file: no text extracted" placeholder if unstructured not installed.
    • Audio/Video (.mp3, .mp4, .wav, .m4a): "AV file: transcript not available in this release" placeholder (full transcription is out of scope). The extracted text is passed into the existing analyze()/generate() ingest loop unchanged — no new LLM calls are added for extraction.
    • AC-F12-2: The server-side file-type gate in POST /ingest/upload is extended: the HTTP 415 rejection now only applies to file types with no extractor at all. The accepted MIME type list is updated to include PDF, DOCX, PPTX, XLSX, and image MIME types. The F1-UPLOAD client-side accept attribute is updated to match.
    • AC-F12-3 (I1): Uploading the same multi-format file twice (same mtime/hash) does not create a duplicate Postgres row or Qdrant vector. The existing content-hash check covers this regardless of file type.
    • AC-F12-4 (I5): Extracted text for multi-format files is stored in vault/raw/sources/ as a companion .extracted.md file alongside the original binary. The original binary is preserved. vault/wiki/ is only written by the ingest pipeline. K1 3-layer rule intact.
    • AC-F12-5: New Python dependencies (pypdf, python-docx, python-pptx, openpyxl, unstructured) are added to backend/pyproject.toml. Docker image rebuilds cleanly. All new dependencies are pinned versions.
    • AC-F12-6: Unit tests: fixture PDF with 2 pages → assert extracted text contains content from both pages; fixture DOCX → assert paragraphs extracted; fixture XLSX → assert sheet rows present as markdown table; unsupported file type → HTTP 415.
    • AC-F12-7: The format extractor is in a new module backend/app/ingest/extract.py behind a dispatch function extract_text(file_path) -> str. Static guard: no format-specific imports (pypdf, docx, pptx, openpyxl) outside ingest/extract.py.

    F13 — Cascade deletion¶

    • AC-F13-1: backend/app/ops/cascade_delete.py implements cascade_delete(page_id): (1) identify the target page in Postgres; (2) find all wikilinks pointing TO this page (links table, target_title match); (3) remove the target page's Postgres row (soft delete: deleted_at = now()); (4) remove the Qdrant vector for the page; (5) remove the page's entry from vault/wiki/index.md; (6) scan all pages in vault/wiki/ that contain [[<target_title>]] wikilinks and replace them with <target_title> (dead wikilink becomes plain text); (7) update the dangling=True flag on affected links table rows.
    • AC-F13-2: 3-method matching for finding references: (a) exact wikilink title match in links table; (b) slug-normalised match (lower-case, hyphenated); (c) full-text scan of wiki/ .md files as fallback. Method used is logged per deletion.
    • AC-F13-3: Shared entities are preserved: if a page being deleted has content that is also referenced by other pages (via source overlap in the edges table), the deletion logs a warning listing the sharing pages and proceeds — it does NOT block. The human operator is responsible for reviewing the warning.
    • AC-F13-4 (I1): Cascade delete only touches records related to the deleted page. It NEVER triggers a full vault rescan or graph recompute (data_version IS bumped so the graph eventually updates via the debounce, but no immediate FA2 run is forced).
    • AC-F13-5: REST endpoint: DELETE /pages/{id} triggers cascade_delete(page_id). Response: HTTP 200 with {deleted_page_id, wikilinks_cleaned int, index_entry_removed bool, shared_entity_warnings list[str]}. Documented in openapi.json (D4 zero-drift gate).
    • AC-F13-6: A "Delete page" action is available in the wiki page view (frontend). It shows a confirmation modal listing any shared-entity warnings before executing. On confirm, calls DELETE /pages/{id}; on success, navigates back to the page tree.
    • AC-F13-7: Unit tests: delete a page with 3 incoming wikilinks → assert all 3 cleaned; delete a page with shared source overlap → assert warning logged, deletion completes; double-delete (page already deleted) → assert HTTP 404 gracefully.

    6. Invariants in Force — I7 and I9 Are First-Class This Sprint¶

    All 9 invariants apply. I7 and I9 are the headline invariants for M5.

    I7 — LOOPS ARE BOUNDED (headline for M5)¶

    F10 Deep Research is the most loop-intensive feature in the roadmap. Every component of I7 must be demonstrably satisfied in all M5 loops:

    1. deep_research.py: max_iter + token_budget + concurrency ≤ 3. No exceptions.
    2. review.py: pre_generated_query generation is bounded to 1 call per item with timeout.
    3. F9 Deep-Research action: delegates to deep_research.py — inherits its bounds.
    4. F17 CliAgentProvider.chat(): the delegated agent loop must have max_iter + token_budget.
    5. F12 extraction: no loop (stateless per-file extraction). Not in scope for I7.
    6. F13 cascade_delete: single pass (not a loop). data_version bump triggers the existing debounced recompute — already bounded by GraphCache (from v0.3).
    7. Every new run (deep_research_runs, any future loop log) must log total_cost_usd. Anomaly threshold per ADR-0009: flag if > $1.00 per run.

    Violation of I7 in any M5 feature is a P0 blocker. QA must include a specific I7 test for F10: mock InferenceProvider always returns "insufficient" → assert loop terminates at max_iter, status = "max_iter_reached", no further calls made.

    I9 — DO NOT REINVENT (headline for M5)¶

    SearXNG is already running on the developer's TrueNAS stack. F10 must reuse it.

    1. F10: SearXNG base URL from SEARXNG_URL env var. Never Tavily, DuckDuckGo, or Google.
    2. F5: Qdrant + bge-m3 already running. No new embedding service. No new vector DB.
    3. F12: pypdf / python-docx / python-pptx / openpyxl / unstructured are extractors, not inference providers. They must not make LLM calls directly — all inference goes through InferenceProvider (I6 also applies).
    4. F13: Uses existing Postgres links and pages tables. No new data structure.

    I1 — INCREMENTAL INDEX ONLY¶

    F13 (cascade delete) is the highest I1 risk this sprint: it modifies vault/wiki/ files. The implementation MUST: - Only touch files that reference the deleted page (not all files). - Not trigger a full watcher rescan. - Use targeted file updates (read → modify → write specific files only).

    I6 — PLUGGABLE INFERENCE¶

    F10 Deep Research uses InferenceProvider for (a) query generation, (b) sufficiency assessment, (c) synthesis. All three calls go through the active provider's methods. No direct Ollama/Anthropic/claude-agent-sdk calls in deep_research.py — these must route through the existing InferenceProvider abstraction. Static guard T-CQ-009 remains.

    I5 — OBSIDIAN COMPATIBILITY¶

    F12 and F13 both touch vault/wiki/ or vault/raw/. Verify: - F12: binary files go to vault/raw/sources/; extracted .md files also in vault/raw/; wiki/ is only written by the ingest pipeline. No broken frontmatter introduced. - F13: dead wikilinks cleaned in wiki/ files must not break YAML frontmatter. - test_obsidian_check.py (15 tests) must remain green after M5.

    I2 — GRAPH LAYOUT IS SERVER-SIDE AND CACHED¶

    F5 graph-expansion uses the Postgres edges table only — it does not trigger FA2. F13 data_version bump triggers the debounced recompute — already server-side (no change). No new client-side graph layout code may be introduced in any M5 PR.

    I3 — NO PER-TOKEN HEAVY WORK IN CHAT¶

    F5 retrieval results are assembled server-side and passed to the provider as a pre-built context string. No retrieval happens per-token during chat streaming. The Zustand store pattern from M4 is unchanged.
