    Sprint 6 — v0.6 → v1.0 — PM Scope Lock¶

    Milestone: M6 — "Shippable" (final milestone; tag target v1.0.0) Author: orchestrator (acting PM) — autonomous build Date: 2026-06-30 Gate: Sprint start — formal scope confirmation Branch: sprint/v0.6 (cut from sprint/v0.5; carries the nashsu/llm_wiki reference audit under docs/reference/llm_wiki-audit/) Prerequisite: M5 MET-PENDING-EC-M5-HCP (docs/sprints/v0.5-m5-exit-criteria.md). Baseline on cut: backend 926 pytest / frontend 621 vitest green. Inputs: CLAUDE.md §3/§4/§7/§8/§9, BACKLOG.md §"Sprint 6 — v0.6 — M6", docs/reference/llm_wiki-audit/* (parity reference).

    1. Sprint Objective¶

    Take Synapse from "feature parity core" (M5) to shippable v1.0: a self-hosted product a new user can install and use day-one, at functional and graphical parity with the reference app nashsu/llm_wiki. The headline theme is closing the loop: the third Karpathy operation (Lint) lands; the ingress surface (web clipper) is added securely; the app is packaged (PWA + Tauri v2) with a CI/CD that actually runs the test suite; and the full D1–D7 documentation set is completed and consistent.

    Headline invariants for M6: - I7 (bounded loops) — the new Lint-fix loop has max_iter + token_budget + cost log. - I6 (pluggable inference) — Lint's semantic pass routes through InferenceProvider; no hardcoded backend. - I8 (docs-as-DoD) — M6 is the final docs gate: every D-artifact complete and consistent, ER matches schema. - I9 (do not reinvent) — no Tavily/alt-search; SearXNG only; no local embedding reimplementation.

    Parity-reference rule: the llm_wiki audit is advisory input, not a spec to copy. Where the reference app made a security/architecture mistake, Synapse must NOT repeat it. The audit's standout finding — the reference's clip server on :19827 is unauthenticated, accepts drive-by cross-origin writes, and writes to an attacker-controlled base path — is an explicit anti-pattern: Synapse's F11 ingress MUST be authenticated, origin-checked, body-capped, and path-safe (see F11 ACs).

    3. Per-feature Acceptance Criteria¶

    K2 — Lint-fix loop¶

    • AC-K2-1 ops/lint.py runs a bounded loop (LINT_MAX_ITER, LINT_TOKEN_BUDGET, LINT_MAX_FINDINGS); total_cost_usd logged 4dp; terminal status always written (I7). ✅ (test_lint.py)
    • AC-K2-2 All LLM calls via resolve_provider_config — no hardcoded backend, no class-name branching (I6). ✅ static test.
    • AC-K2-3 Scan reads only pages/links (no vault walk); apply touches only referencing pages; data_version bumped ≤1 per fix (I1). ✅
    • AC-K2-4 Human gate: scan never applies a fix; apply only on explicit call. Flag-only categories (orphan/contradiction/stale) acknowledge without editing human-curated content (K8). ✅
    • AC-K2-5 Frontend Lint section: Run-Lint action, findings grouped by category with Apply/Dismiss, cost line, empty state; nav slot activated; i18n EN+IT parity; vitest green; I3/I4 respected. ⏳ (in progress)
    • AC-K2-6 ADR-0037 + docs/sequences/lint-fix.mmd present and valid; openapi.json + ER regenerated. ✅

    F11 — Web clipper + secure clip ingress¶

    • AC-F11-1 Chrome MV3 extension under extension/: Readability + Turndown (HTML→Markdown, tables), popup with project/vault picker, posts to the local ingress.
    • AC-F11-2 (SECURITY — non-negotiable) The local clip ingress requires a token (rejects unauthenticated → 401); validates Origin/Host against an allowlist (chrome-extension://<id> + loopback); caps the body size; and writes only inside the vault via a safe-join path check (no attacker-controlled base path). A regression test proves each: unauth→401, bad origin→reject, oversized→413, traversal→reject. (Directly answers llm_wiki audit S-1…S-6.)
    • AC-F11-3 A clipped page lands in vault/raw/sources/ and is auto-ingested through the existing pipeline (I1/I5/K1) — no new ingest path.
    • AC-F11-4 Clip content is sanitized before render; no unbounded polling race (use the existing watcher/queue, not a naive 3 s poll that double-ingests).
    • AC-F11-5 ADR for the ingress security model; D3 sequence web-clip.mmd; openapi.json updated if an HTTP endpoint is added.

    F15 (rest) — PWA + Tauri v2 + CI/CD¶

    • AC-F15-1 PWA: web manifest + service worker; the frontend is installable; offline shell loads. No violation of I2/I3/I4.
    • AC-F15-2 Tauri v2 shell wraps the frontend as a desktop app (macOS/Windows/Linux targets configured). Close-behavior is a user setting (not hardcoded).
    • AC-F15-3 (CI HARDENING) CI runs on sprint/** + main (fix NB-9 branch filter) and executes the test suites (pytest + vitest) and the linters with pinned versions (resolve the ruff/black drift on v0.5 files), plus the mmdc Mermaid render check (GAP-v0.5-3). Multi-OS build matrix produces installable artifacts.
    • AC-F15-4 Linter drift cleared: ruff check + black --check green tree-wide with pinned versions recorded in pyproject.toml/CI.

    F2 — purpose.md as provider context¶

    • AC-F2-1 Confirm (and complete if missing) that purpose.md is injected into the ingest analysis/generation prompts and the chat/query context; add a test asserting its presence in the assembled provider input.

    D1–D7 final gate¶

    • AC-D-1 All D-artifacts present and consistent; ER matches live schema (make er zero-drift); openapi.json current; D5 screenshots captured; ADR index complete; USER.md + DEPLOY.md cover all M6 additions (incl. LINT_*, clipper token/env, PWA/Tauri).

    4. Out of scope (M6) / never (invariants)¶

    • MinerU cloud PDF parsing, multimodal image captioning, Louvain communities + graph-insights, browser-extension for non-Chrome — not in M6 (post-v1.0 backlog; not part of the locked parity set).
    • The "never" list stands (BACKLOG §"Out of scope — never"): full-rescan (I1), main-thread force layout (I2), per-token parse (I3), WYSIWYG editor (I4), hardcoded provider (I6), unbounded loops (I7), skipping D-artifacts (I8), Tavily/alt-search or local embedding reimpl (I9).

    • EC-M6-1 K2 complete (backend + frontend), all ACs green.
    • EC-M6-2 F11 clipper + secure ingress, all security ACs proven by tests.
    • EC-M6-3 F15: PWA + Tauri shell build; CI runs tests + pinned linters + mmdc on the right branches; multi-OS artifacts.
    • EC-M6-4 F2 verified; deferred FUs disposed.
    • EC-M6-5 Full suite green (pytest + vitest + Playwright); linters clean tree-wide.
    • EC-M6-6 D1–D7 complete and consistent; ER zero-drift; ADR index complete.
    • EC-M6-7 wiki/ remains a valid Obsidian vault after all M6 ops (I5/K7).
    • EC-M6-HCP Human checkpoint (Emanuele) in a live browser/desktop build → then tag v1.0.0.

    7. Execution model¶

    Autonomous, one vertical at a time, each landed as a verified green commit before the next: K2-frontend → F11 → F15 → F2 + deferred FUs → docs gate → v1.0.0 tag. Subagent team per CLAUDE.md §13: ai-agent-engineer (provider-touching loops), backend-engineer (endpoints/data), frontend-engineer (UI), qa-test-engineer (gates), solution-architect (ADRs/invariants), devops-engineer (CI/CD/packaging), tech-writer (docs gate). No conflicting agents on the same files concurrently.
