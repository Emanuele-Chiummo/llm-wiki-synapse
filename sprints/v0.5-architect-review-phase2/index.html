    v0.5 — Architect Review — M5 Phase 2 (Deep Research, F10)¶

    • Reviewer: solution-architect
    • Date: 2026-06-29
    • Branch: sprint/v0.5
    • Scope: F10 Deep Research, judged against the frozen contract ADR-0024 and the 9 invariants.
    • Headline invariant: I7 (bounded loops).
    • Method: static read of the implementation against ADR-0024 §2/§3/§4/§5/§7/§8. Test suite NOT run (QA owns that, in parallel).

    Verdict: APPROVE-WITH-CONDITIONS¶

    The core of F10 — the bounded loop, the SearXNG-only web-search seam, the ingest-seam synthesis, and the data model — is implemented exactly to the frozen contract. All 11 Do-NOT items are HELD in the loop module. The boundedness posture (I7) is provable from the code: for iteration in range(1, max_iter_local + 1), a top-of-round budget gate, and a single hardcoded Semaphore(3).

    One non-P0 correctness defect blocks unconditional approval: the REST entry POST /research/start and the loop run_deep_research each generate a different run_id and each INSERT a row, so the loop writes to a row the client never polls. This does not violate any invariant (every loop path is still bounded; both rows are still terminal-written), but it breaks the fire-and-poll contract of ADR-0024 §8.1 and would make the UI poll a permanently-running orphan. It must be fixed before the phase gate. See Condition C1.

    I9 static guard¶

    The code cannot reach Tavily/Google/DDG/SerpAPI:

    • The only outbound search call is searxng.py:searxng_search → GET {SEARXNG_URL}/search?format=json. No second search function exists.
    • searxng.py imports only httpx, pydantic, app.config. No search SaaS SDK.
    • The static guard test_no_forbidden_search_imports (test_deep_research.py:913) scans all app/ops/**/*.py against tavily|duckduckgo|ddg|googlesearch|serpapi|google[-_]search[-_]results. Live grep of ops/ returns zero matches, so the guard passes and the ops docstrings themselves don't trip it.
    • On SearXNG failure: [] returned + logged (searxng.py:52, 62, 66). No fallback engine. POST /research/start returns 503 when SEARXNG_URL is unset (main.py:2240–2247) — no fake run, no fallback. HELD.

    Conditions¶

    C1 — run_id divergence (BLOCKING, non-P0, correctness)¶

    POST /research/start generates run_id = uuid.uuid4() (main.py:2251), pre-INSERTs a deep_research_runs row with that id (:2266–2285), returns it to the client (:2305), then schedules run_deep_research(vault_id=…, topic=…, max_iter=…, token_budget=…) without passing run_id (:2288–2295). Inside run_deep_research, a new run_id = uuid.uuid4() is created (deep_research.py:146) and _create_run_row INSERTs a second row (:147–153, :553–588).

    Consequences: - The client polls the endpoint's row, which the loop never updates → it stays status="running" forever. This defeats the fire-and-poll contract (ADR-0024 §8.1) and is a soft I7-observability regression (a row that never reaches terminal, even though the actual loop is bounded and its own row IS finalized). - Two rows per run pollute GET /research/runs and double-count sources_fetched=0 orphans.

    Required fix (pick one, architect-preferred = A): - A. run_deep_research accepts run_id: uuid.UUID and _create_run_row becomes an UPDATE-or-skip (the endpoint already INSERTed). The endpoint passes its run_id. The loop reads max_iter/token_budget from the existing row (preserving "frozen on the row" semantics). This keeps a single source of truth and the 202-then-poll guarantee. - B. The endpoint does NOT pre-INSERT; it lets run_deep_research own the INSERT and returns the loop's run_id. Requires awaiting the row creation before returning 202 (a short await, not the loop) so the poll target exists. More refactoring; A is cleaner.

    Either way: exactly one row per run, and the id returned to the client is the id the loop finalizes.

    C2 — confirm the QA loop-gate tests bind to the fixed entry (advisory)¶

    The mandatory max_iter_reached / budget_exhausted / error-never-running tests (test_deep_research.py:322, 468, 746) call run_deep_research directly, so they pass regardless of C1. Add/confirm one test_research_api.py assertion that POST /research/start then GET /research/runs/{id} eventually reaches a terminal status — this is the regression that would have caught C1. Hand to QA.

    C3 — D-artifact gate (I8, blocking for phase DoD, owned with tech-writer)¶

    Before the phase gate: make er (D2 — both tables + relationships in docs/er/schema.mmd), make openapi (D4 — the three /research/* endpoints), and docs/sequences/deep-research.mmd (D3, ADR-0024 §9) must be regenerated and committed with zero drift. Not verified in this review; flagged to tech-writer.

    Invariant tension the ADR did not anticipate¶

    • Mechanical (no-provider) path. _resolve_provider returning None (no provider_config row) is handled gracefully — query-gen falls back to [topic], assess treats any content as sufficient, synthesize assembles raw snippets (:303–306, 411–414, 465–472). ADR-0024 assumed a resolved provider. This is a reasonable, invariant-safe degradation (still bounded, still SearXNG-only, still ingest-seam), but it means a deep-research run can complete with zero LLM involvement and still write a raw/sources/ doc. Acceptable for Phase 2; note it in USER.md so users understand an unconfigured-provider run produces a low-quality stub. No invariant traded.
    • _create_run_row returns an expunged snapshot (:587) and the loop never re-attaches it — correct, and it's the mechanism that makes "frozen on the row" hold without lazy re-reads. Good. (This is also why C1's fix must keep reading bounds from the row, not from the function args, to preserve the audit semantics.)

    No invariant is traded for convenience. The two P0 headline invariants (I7, I9) are fully held in the implementation.

    • C2 → qa-test-engineer (add the start→poll→terminal API regression).
    • C3 → tech-writer (D2/D3/D4 regeneration + ADR-0024 README row).
