    v0.5 Architect Review — M5 Phase 1 (F5 retrieval + [n] citations)¶

    Reviewer: solution-architect · Date: 2026-06-29 · Branch: sprint/v0.5 Scope: pre-phase-gate sign-off of the Phase-1 implementation against ADR-0022 (frozen contract) and the 9 invariants. Test execution is owned by qa-test-engineer (parallel) and was deliberately NOT run here. Reviewed: backend/app/rag/retrieval.py, backend/app/chat/stream.py, backend/app/main.py (GET /search, POST /ingest/from-text), backend/app/ingest/provider/cli.py, frontend/src/components/chat/decorateCitations.ts, MarkdownView.tsx, backend/app/qdrant_client.py, app/models.py, app/migrations/versions/, docs/api/openapi.json.

    Verdict: APPROVE-WITH-CONDITIONS¶

    The implementation is a faithful, invariant-clean realisation of ADR-0022. All ten Do-NOT rejection triggers HOLD. The frozen data structures (PageRef / Citation / RetrievalContext) match §2.3 exactly. The one accepted limitation (§4) is honestly preserved, not silently worsened. No invariant is traded for convenience.

    The conditions are documentation-only (one ADR code-reference amendment + one inline provenance note), not code changes. They do not block the phase gate; they keep the docs-as-DoD ledger (I8) truthful. Detail in Conditions below.

    Do-NOT compliance table (ADR-0022 §5, items #1–#10)¶

    # Do-NOT rule Verdict Evidence
    1 No inference call inside retrieve() HELD retrieval.py imports only get_embedding_client (bge-m3 embed of the query string — not a generative call), get_qdrant_client, get_session. No provider import. The four phases are embed→vector-read→edge-read→file-read→assemble. retrieve() (L136–211) makes zero provider.* calls.
    2 No vault walk / rescan in retrieval or GET /search; GET /search leaves data_version unchanged HELD grep for rglob/glob/os.walk/scandir/iterdir/listdir in retrieval.py → NONE. The only filesystem touch is _load_passage (L487–503), a single targeted (vault_root / file_path).read_text() per surfaced candidate — one file, no directory walk. No UPDATE/INSERT/DELETE/commit in retrieval.py; _read_data_version (L509–524) is a pure SELECT, snapshotted BEFORE assembly (L168) and returned for the AC-F5-5 proof. GET /search (main.py L1229–1287) calls retrieve() and serialises — no mutation.
    3 Expansion reads edges, never GraphEngine/FA2 HELD _phase2_graph_expansion (L253–301) / _expand_frontier (L304–363) issue raw SELECT … FROM edges and SELECT … FROM links. No import of app.graph.engine, no GraphEngine, no igraph/FA2 reference anywhere in the module. Depth hard-capped at _MAX_EXPANSION_DEPTH = 2 (L62, clamped L165).
    4 No new embedding service / Qdrant collection / sparse index / tiktoken / sentence_transformers HELD pyproject.toml grep for tiktoken\|sentence_transformers\|bm25\|tavily → NONE. retrieval.py grep for create_collection\|recreate\|sparse\|VectorParams → NONE (only a docstring mention of "no new collection"). Phase 1 reuses get_embedding_client() (bge-m3) and the existing settings.qdrant_collection (synapse_pages, cosine, single dense VectorParams defined in qdrant_client.py L74–78). char/4 budget (_CHARS_PER_TOKEN = 4, L56) — no tokenizer.

No D2 (ER) change expected/made in Phase 1 — confirmed (no model/migration change).

    Result: 10 / 10 HELD. No VIOLATIONS.

    Frozen data-structure conformance (ADR-0022 §2.3)¶

    Structure §2.3 contract Code (retrieval.py) Match
    PageRef id: str, title: str, slug: str L70–80 — identical, slug derived in code ✓
    Citation n: int, ref: PageRef, score: float, phase: Literal["vector","expansion"] L83–95 — identical, including the Literal ✓
    RetrievalContext query, text, citations: list[Citation], token_budget: int, approx_tokens: int, data_version: int L98–116 — identical field set, types, and order ✓

    Authority invariant (len(citations) == distinct[n]count): structurally guaranteed — the assembler _phase4_assemble (L426–471) emits each [n] block and its Citation together in the same loop iteration; they cannot drift. The compact done//search projections (stream.py L342–345, main.py L1268–1278) are pure re-shapes of the same authoritative list.

    Public interface (retrieve(query, *, vault_id, context_window, k=8, expansion_depth=2)): matches §2.1. The added session: AsyncSession | None = None (L143) is a keyword-only, default-None test/transaction seam that does not alter the frozen positional contract or any caller — acceptable.

    Accepted-limitation honesty check (ADR-0022 §4)¶

    [n] resolves to the source-document pages row, not the synthesized wiki/ page. Preserved honestly and not silently worsened: - _load_page_meta reads pages.{id,title,file_path} (L544–558); file_path is the raw/sources/... path, and _load_passage reads that source body. The citation therefore points at the indexed source — exactly the §4 boundary, no new degradation. - title falls back to the filename stem so it is never empty (L555–557, §2.6). - The frontend renders the title on hover and navigates by slug (decorateCitations.ts L100, MarkdownView.tsx L89–103) — it surfaces whatever the backend cites; it does not paper over the source-vs-wiki distinction. The limitation remains visible and deferred post-M5, as designed.

    Invariant-tension scan (beyond the Do-NOT list)¶

    No new tension introduced. Two points worth recording for the gate:

    1. Retrieval failure is non-fatal to chat (stream.py L184–197). retrieve() errors are caught and chat proceeds with empty retrieval context (no citations). This is a sound resilience choice and does not violate any invariant — but it means a silent retrieval regression would degrade grounding without failing the turn. The WARNING log (L193) is the only signal. Acceptable for Phase 1; flag for QA to assert the log path and for a future health metric. Not a condition.

    2. _expand_frontier IN-list interpolation (retrieval.py L314–328, L348–351, L539–547). Bind placeholder NAMES (:f0,:f1,…) are f-string-interpolated, but every id value travels as a bound parameter — never as SQL text. # noqa: S608 is correctly scoped and the justification is documented inline. Not an injection vector. HELD, no action.

    Both are consistent with the ADR's "simple and correct over clever and fragile" stance.

    Conditions (documentation-only — do NOT block the phase gate)¶

    C1 — Amend ADR-0022 for query_points (REQUIRED, doc drift / I8). ADR-0022 §2.2 phase 1 and §3 (AQ-v0.5-1) literally reference qdrant.search(synapse_pages, vector, limit=k, with_payload=True). The implementation uses client.query_points(collection_name=…, query=vector, limit=k, with_payload=True) and reads response.points (retrieval.py L232–246). This is semantically equivalent and the correct call: qdrant-client removed search() (deprecated, dropped by 1.18); query_points is the current dense top-k API. Against the existing single-dense synapse_pages collection it performs the identical cosine top-k — same collection, same vector, same ScoredPoint(id, score, payload) results, same id == pages.id mapping (ADR-0002). No collection-shape change, no sparse vector. Acceptable and approved. ADR-0022's code reference should be updated from .search() to .query_points() (with a one-line note that .search() was removed in client 1.18) so the frozen contract matches the shipped code. Hand to tech-writer.

    C2 — Pin the qdrant-client floor (RECOMMENDED, reproducibility). backend/pyproject.toml declares qdrant-client>=1.9.0, but query_points returning a QueryResponse.points list is the modern shape; 1.9 is the lower bound that introduced it and older patch behaviour differs. Since the code now depends on query_points/.points, devops should confirm the lockfile resolves to a query_points-bearing release (≥1.10 is safe) and, ideally, bump the floor so a fresh environment cannot resolve a search()-only client. Non-blocking for this gate (CI installs a current client), but record it. Hand to devops-engineer.

    No code conditions. Nothing in the implementation needs to change to pass the gate.
