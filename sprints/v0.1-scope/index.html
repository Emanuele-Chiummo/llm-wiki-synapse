    Sprint 1 — v0.1 — Scope Lock¶

    Milestone: M1 — "Data flows end-to-end" Branch: sprint/v0.1 Locked by: product-manager on 2026-06-28 Status: LOCKED — no changes without PM + orchestrator approval

    1. Sprint objective¶

    Prove that the data pipeline is correctly wired: a Markdown file dropped into vault/raw/sources/ is detected by the watcher, indexed into Postgres (metadata) and Qdrant (bge-m3 vector), and retrievable via the REST API. No AI inference, no graph layout, no frontend UI.

    The walking skeleton establishes the foundation that every later sprint builds on. It is intentionally thin.

    4. Invariants in force for Sprint 1¶

    All 9 invariants apply from day 1. The ones most directly tested by v0.1:

    Invariant How it manifests in v0.1 Test / verification
    I1 — INCREMENTAL INDEX ONLY Watcher uses mtime + content hash; skip unchanged files; NEVER rescans directory on startup pytest: drop same file twice → single DB row
    I5 — OBSIDIAN COMPATIBILITY vault/wiki/.obsidian/app.json auto-created; all written .md files have valid YAML frontmatter Manual: open vault/wiki/ in Obsidian; no errors
    I7 — LOOPS ARE BOUNDED No loops in v0.1 (no LLM inference). The file-event handler is not a loop. N/A in v0.1; architecture decisions made now must not preclude bounds later
    I8 — DOCS-AS-DoD D1+D2+D4 required before M1 sign-off; make er and make openapi are automated CI: make er + make openapi must exit 0 with correct output
    I9 — DO NOT REINVENT bge-m3 via existing running service (env var URL); Qdrant via existing running instance; no new services introduced docker-compose.yml must NOT start a new Qdrant or embedding container

    Invariants I2, I3, I4, I6 are not directly exercised in v0.1 (no graph, no chat, no editor, no provider routing), but no design decision in v0.1 may make them harder to honour in v0.2+. Any such conflict must be escalated to solution-architect.

    5. M1 Exit Criteria (concrete, checkable)¶

    All items must be MET before PM sign-off and before advancing to Sprint 2.

    EC-1 (K1 + K4 + K6 + K7): Vault structure valid¶

    • vault/raw/sources/, vault/raw/assets/ exist.
    • vault/wiki/index.md, vault/wiki/log.md exist with valid YAML frontmatter.
    • vault/wiki/.obsidian/app.json is present and parseable as valid JSON.
    • vault/schema.md and vault/purpose.md (stub) exist.
    • Opening vault/wiki/ in Obsidian shows no errors (human checkpoint).

    EC-2 (Watcher + I1): Incremental ingest fires correctly¶

    • Drop a new .md file into vault/raw/sources/ → within 5 seconds: a row appears in Postgres pages table AND a point appears in Qdrant synapse_pages collection.
    • Drop the same unchanged file again → no new Postgres row, no new Qdrant point, no new log.md entry (I1).
    • Modify the file → Postgres row updated (updated_at changes), Qdrant vector replaced, log.md gets one new INDEXED entry.
    • Delete the file → Postgres row has deleted_at set, Qdrant point removed.
    • Service startup does NOT trigger any DB writes (no rescan).

    EC-3 (K4): log.md append-only¶

    • After 3 distinct ingest events, log.md has exactly 3 entries.
    • File is never shorter than after the previous run (append-only contract).

    EC-4 (K6 + Postgres): Frontmatter stored correctly¶

    • A .md file with frontmatter type: entity, title: "Test", sources: ["a.pdf"] produces a Postgres row with those exact values.
    • A .md file with missing frontmatter produces a row with NULL for missing fields (no exception).

    EC-5 (Qdrant + I9): Vectors from existing bge-m3 service¶

    • Qdrant synapse_pages collection exists after service startup.
    • After ingest, a point with correct payload is retrievable via Qdrant API.
    • EMBEDDING_URL env var is the only way the service reaches bge-m3; no local model loaded.

    EC-6 (REST API): All 4 endpoints operational¶

    • GET /status → HTTP 200, JSON with data_version field.
    • GET /pages → HTTP 200, paginated list; new page visible after ingest.
    • GET /pages/{id} → HTTP 200 for valid id; HTTP 404 for unknown id.
    • POST /ingest/trigger {"file_path": "..."} → HTTP 202; triggers ingest pipeline.

    EC-7 (F16 dataVersion): Version increments¶

    • data_version starts at 0.
    • After first ingest: data_version = 1 (GET /status confirms).
    • After second ingest: data_version = 2.
    • No ingest = no version change.

    EC-8 (Docker Compose + devtools):¶

    • docker compose up on a clean environment starts synapse-backend and postgres16 without error.
    • make test exits 0 with all pytest tests green.
    • make er generates docs/er/schema.mmd.
    • make openapi generates docs/api/openapi.json.
    • No secrets in committed files (.env is gitignored).
