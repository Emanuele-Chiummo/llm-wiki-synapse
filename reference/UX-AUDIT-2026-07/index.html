    UX Audit — Synapse v0.6 (July 2026)¶

    Conducted by: Frontend Engineer (senior UX audit role) Scope: sprint/v0.6 codebase + docs/screens screenshots Method: code review + screenshot analysis across 10 audit lenses

    Executive Summary¶

    Synapse has a coherent visual identity (GitHub-dark-inspired palette, --syn-* token system, Lucide icons) and the structural invariants (virtualisation, no main-thread layout, streaming parse guard) are well respected. The critical friction is concentrated in three areas: (1) the NavRail secondary group uses opaque Italian labels ("Revisione", "Profonda") that are not self-explanatory even in Italian, making the M5 features effectively invisible to new users; (2) the chat panel carries a severe density problem — conversations are all titled "Untitled" with no preview, the message list has no visual separation between turns, and the role labels (YOU / ASSISTANT in bold uppercase green/blue) dominate the content hierarchy; (3) inline-style proliferation across ReviewQueueView, ActivityBar, and Header means dark-mode regressions are one hardcoded hex away. Seven findings are P1 because they produce broken or confusing user flows today.

    Design-System Debt Note¶

    The current approach uses theme.css for token definitions and .syn-* utility classes, but component files (ReviewQueueView, ActivityBar, Header, MessageList, ConnectScreen) contain hundreds of inline style objects that duplicate token-derived values, bypass class-based theming, and make global changes require hunting through TSX files.

    Recommended strategy for v0.9:

    Do not introduce a CSS-in-JS library. Instead:

    1. Extend theme.css with a components.css layer containing scoped class rules for recurring patterns: .syn-button--ghost, .syn-meta-row, .syn-card-row, .syn-notice-banner, .syn-role-label. These already exist conceptually but are expressed as repeated inline objects.
    2. Define a strict rule: no inline style may reference a raw hex color or a raw pixel value that is not derived from a --syn-* token. Raw pixel values for layout (widths, flex gaps) are acceptable inline; color values never are.
    3. Audit priority order: ReviewQueueView.tsx (highest inline-style count, dark mode risk), ActivityBar.tsx (keyframes, status bar), Header.tsx (server chip), ConnectScreen.tsx (branding gradient).
    4. The --syn-bg-card token referenced in ActivityBar.tsx:544 and ConfirmDialog.tsx:118 does not exist in theme.css — it falls back to var(--syn-bg-soft) silently. Add it explicitly to theme.css or remove the undefined reference.

    Estimated remediation: M5-level effort to bring inline-style-to-class ratio from ~70/30 to ~20/80 across the affected components.
